# 写作规范技能（writing-style）

**适用场景**：数据分析报告、HTML 交互看板、多角色文档结构、可视化规范。

触发词：帮我写报告、写分析文档、做数据看板、写数据诊断、整理分析结果。

## 第一步：动笔前先回答这4个问题

在写任何文档之前，必须先明确以下内容，否则结构和内容都会跑偏：

### 1. 读者是谁？他们关注什么？

不同角色看同一份数据，关注的问题完全不同，需要用不同的语言和切入点：

| 读者角色 | 核心关注点 | 偏好的信息形态 | 不需要看的内容 |
|--------|---------|------------|------------|
| **管理层（总监/VP）** | 风险在哪、钱花对了没、下一步怎么做 | 1页总结、P0/P1优先级、HC数字、成本影响 | 详细口径定义、执行步骤细节 |
| **运营负责人** | 我的项目排在哪里、要改变什么、谁来做 | 分项目明细表、问题清单、对策建议 | 算法原理、数据计算方式 |
| **算法/技术** | 数据来源可靠吗、口径怎么定义的、模型跑的对吗 | 口径定义、数据质量说明、样本量、置信度 | 人员管理建议、成本分析 |
| **产品经理** | 用户/场景是什么、功能如何支撑策略 | 场景分类全景图、用户分层逻辑、优先级 | 人员结构细节、HRO/BPO区分 |
| **HR/人力** | 岗位是否匹配、层级调整方向、编制合理性 | 层级-场景交叉分析、错配明细、人员名单 | AI工具渗透率、SOP覆盖率 |

**实操规则**：写多角色文档时，在开头用一个"导读"说明哪章是给谁看的，让每个读者直接翻到对应章节，不需要读全文。

### 2. 他们为什么读这份文档？（目的）

- **决策类**：需要立场和优先级，结论要明确，不能两面都说
- **汇报类**：需要让上级知道发生了什么、做了什么、还缺什么
- **协同类**：需要建立共识，让不同团队看到同一件事的不同侧面
- **存档类**：需要完整性和可追溯性，口径和数据来源要清楚

### 3. 他们带着什么问题来？

好文档是"预先回答了读者脑子里的问题"。常见问题列表：

- "这件事严不严重？"→ 必须给数字和对比基准
- "跟我有没有关系？"→ 必须有分业务线/分项目的明细
- "要我做什么？"→ 必须有可执行的下一步，并标注谁负责
- "这个数字可信吗？"→ 必须有口径定义和数据来源说明
- "之前是什么情况？"→ 必须有基线值或历史对比

### 4. 为什么文档写得不好？（背景诊断）

写作质量差的根本原因通常是以下几种：

| 根本原因 | 表现症状 | 修正方向 |
|--------|--------|--------|
| **以写者视角而非读者视角** | "我知道这个数字在另一个Tab里" | 自包含原则：每章节需要的数字就写在那一章 |
| **数据和文字分离** | 文字说"存在问题"，数字在图表里 | 数字嵌入文字，文字引用图表，两者互相印证 |
| **结构服从写作顺序** | 按"我怎么分析的"写，而非"读者怎么理解" | 先问读者最关心什么，再决定章节顺序 |
| **没有明确受众** | 什么都说，对谁都不够清楚 | 确定主要读者，其他人作为次要读者 |
| **把过程当成结论** | "我们分析了三个维度……" | 直接说结论，分析过程放附录 |
| **没有优先级意识** | 所有内容平铺，读者自己判断轻重 | P0/P1/P2或一句话executive summary前置 |
| **数字孤立无对比** | 只写"39.5%"，没有参照系 | 同比/环比/行业基准/目标值，至少给一个 |

## 核心写作原则

### 原则一：总—分—总的视觉漏斗

**写一个章节的正确顺序**：

1. **一句话摘要**（让读者决定要不要继续读）
2. **可视化/数字卡片**（让数字从文字里脱离出来，可以被扫描）
3. **表格或分类全景图**（建立整体认知框架）
4. **逐条细化**（按优先级展开，越往后越细）
5. **本节小结或行动项**（可选，长文档用）

**反例（旧写法）**：

```
二、现状诊断深化：基于335人的全景透视

基于335人的全量数据，发现以下三大核心错位问题，且存在新的结构性风险：

1. 结构性失衡：高阶人力被低阶事务淹没
   数据洞察：在163名HRO中，70人（42.9%）存在重度错配...
```

问题：没有总起数字，没有可视化，直接进入叙述，读者需要读完整段才能建立全局感。

**正例（新写法）**：

```
二、现状诊断 → 三大核心问题

┌─────────────────────────────────────────────────────┐
│  113人（33.7%）涉及错配  │  3个项目成本倒挂  │  34人打杂指数  │
└─────────────────────────────────────────────────────┘

【七大场景全景地图】（先看整体分类框架）
S1 基础标产 → 标产  → 离岸化/自动化
S2 高阶质保 → 非标→标产 → 标准化后离岸
S3 数据分析 → 核心非标 → 强化与赋能
...（一张表，7行，看完就知道全局）

【场景×属性分布】
纯标产 113人（33.7%） | 标产为主 43人（12.8%）| 混合型 92人（27.5%）...

问题1：离岸化空间（正向机会，先讲）
问题2：结构性失衡（最严重的问题）
问题3：成本倒挂（最具体可操作）
```

### 原则二：数字内嵌文字，不要让读者去"找"

**核心问题**：文字描述问题时，数字藏在图表里或其他Tab里，读者需要反复切换才能理解。

**规则**：每一句有判断的话，旁边就要有支撑数字。文字和数字必须在同一段落甚至同一句话里出现。

❌ 旧写法（文字和数字分离）：
```
存在严重的人才错配问题，高层级人员做了大量低价值工作。
（读者：多少人？什么叫严重？要去哪里找数据？）
```

✅ 新写法（数字内嵌）：
```
在163名HRO中，71人（43.6%）存在重度错配——50%以上精力在做S1基础标产，
其中鹿晴线的多位Z5/Z6员工精力大量投入在"评测标注""圈店圈品"等基础执行工作。
```

**具体操作**：

- 每个结论句 = 数字 + 定义 + 典型案例，三件事写完才算一个完整的论点
- 数字格式：`X人（X%）` 而非 `X人` 或 `X%`——两个维度一起给
- 典型案例：至少一个具体的项目名、人员层级、工作内容

---

### 原则三：可视化是内容的一部分，不是装饰

**可视化的本质**：让不同信息密度的读者都能快速定位自己需要的层次。

**什么时候用什么图**：

| 表达目的 | 推荐形式 | 例子 |
|--------|--------|-----|
| 构成比例 | 数字卡片 + 百分比 | 335人 = HRO 163人(49%) + BPO 172人(51%) |
| 多维度对比 | 横向对比表格 | 6条业务线 × 错配/SOP/AI渗透率 |
| 分类框架 | 全景地图表格 | 七大场景 × 属性 × 优化方向 |
| 优先级分层 | 颜色标注表格 | P0红/P1橙/P2蓝 的行动清单 |
| 趋势变化 | 基线→目标的箭头描述 | SOP覆盖率 87.2% → 95%+ |
| 交叉分析 | 矩阵表格 | 层级 × 场景 × 错配情况 |
| 决策树 | OFS评分卡 | 3维度打分 → 立即离岸/混合/坚守 |

**反例（纯文字，强迫读者在脑中建模）**：
```
驻场人员共335人，其中HRO 163人，占比约49%，BPO 172人，占比约51%，
主要分布在6条业务线，鹿晴线人数最多为130人，占比38.8%，
暮声线126人，占比37.6%，两条线合计占76%……
```

**正例（卡片+表格，眼睛扫一下就懂）**：
```
┌──────────┐  ┌──────────────┐  ┌──────────────┐
│  335人    │  │  HRO 163人   │  │  BPO 172人   │
│ 驻场总人数  │  │   占比 ~49%  │  │   占比 ~51%  │
└──────────┘  └──────────────┘  └──────────────┘

业务线分布：
鹿晴线 130人（38.8%）▓▓▓▓▓▓▓▓▓▓  ← 问题最集中
暮声线 126人（37.6%）▓▓▓▓▓▓▓▓▓
龙芽线  29人（ 8.7%）▓▓▓
若蘅线  25人（ 7.5%）▓▓
蓝武线  21人（ 6.3%）▓▓
```

### 原则四：章节顺序服从读者需要，不服从写作顺序

**错误逻辑**：我先分析数据，然后发现问题，然后给建议——所以文档也按这个顺序写。

**正确逻辑**：读者最先需要知道什么？

- 管理层：结论 → 问题 → 建议（先看结论，有兴趣再看怎么来的）
- 协同方：共识框架 → 各自相关部分（先对齐分类，再看各自的）
- 执行层：行动清单 → 背景数据（先知道要做什么，再看为什么）

**章节顺序的黄金模板**（适用于数据诊断类文档）：

```
一、核心数字（30秒能读完的家底）
    → 可视化卡片，不是散文

二、全景框架（分类体系的完整地图）
    → 表格先行，让所有人对齐定义

三、三大发现（按重要性排序，不按发现顺序）
    → 每个发现：数字 + 现象 + 影响

四、分维度深化（按读者关心的维度分层）
    → 多角色文档在这里做分叉

五、优先级清单（P0/P1/P2 行动表）
    → 最后读，但通常是最多人翻到的

六、附录：口径定义（只有算法/技术会细读）
    → 必须在，但不妨碍其他人阅读
```

---

### 原则五：AI生成内容必须显式声明，且声明要前置

**为什么这是写作规范，不只是免责**：

AI分析的结论是基于当前数据的模式推断，不等于业务判断。读者需要知道"这个结论是怎么来的"才能决定是否接受它，以及接受到什么程度。不声明 = 让读者用错误的权重相信了AI的结论。

**声明的3种写法**（按位置）：

1. **章节前的块级声明**（影响范围最大时用）：
```
> *以下内容来自AI基于数据的自动分析，结论非定版。
> 涉及项目优先级、HC数、人员调整方向，均需结合各业务线
> 实际情况判断，以业务运营团队确认为准。*
```

2. **行内小字声明**（只影响具体数字时用）：
```
schema运营助理项目建议将13名重度错配人员转岗（AI判定，待业务确认）
```

3. **数字旁的置信度标注**（数据质量有疑问时用）：
```
AI工具渗透率：39.5%（基于S4描述关键词匹配，未人工核验）
```

**反模式**：
- 不声明 → 读者以为是人工核实的结论，会直接执行
- 只在文档最末尾放一条总声明 → 读者不会读到那里
- 声明写得太保守 → "所有数据仅供参考" → 等于没声明，读者忽略

---

### 原则六：跨角色文档需要导读索引

当一份文档面向多个角色时，在开头加导读表，让每个人知道自己该看哪里：

```
【读者导读】
- 管理层（10分钟）→ 第一章 + 第五章行动清单
- 业务线负责人    → 第一章 + 第三章对应业务线部分
- 算法/数据团队   → 第二章全景框架 + 第七章口径定义
- 执行团队        → 第五章P0/P1清单
```

---

### 原则七：语言习惯——不同场景的语气锚定

**分析报告**（客观描述+数字驱动）：
```
✅ "在163名HRO中，71人（43.6%）的S1占比≥50%，判定为重度错配。"
❌ "HRO的错配情况比较严重，需要引起重视。"
```

**管理汇报**（结论前置+影响清晰）：
```
✅ "当前有3个P0项目需立即关注，涉及71名错配人员，预计调整后可释放约30%的高阶人力成本。"
❌ "通过深入分析，我们发现了一些值得关注的问题，建议进一步研究和推进。"
```

**OKR撰写**（目标野心+结果可测）：
```
✅ O：通过岗位分类重构，将驻场高阶人力从标产事务中解放出来
   KR1：Q3末，重度错配从71人降至30人（减少57%）
   KR2：Q3末，10个标产项目完成离岸，释放102 HC
❌ O：推进驻场人力资源运营优化工作
   KR1：持续跟进错配问题，按需调整
```

**协同类文档**（共识框架+各方输入口）：
```
✅ "以下结论基于当前数据，各业务线如有不符合实际情况的地方，请在对应行备注。"
❌ "以上分析仅供参考，请各业务线自行判断。"
```

## 数据与文字配合的5种模式

### 模式1：数字开头，文字解释（最常用）
```
71人（43.6%）——这是163名HRO中，S1占比超过50%的人数，
他们做的是本应由BPO或离岸基地承接的基础标产工作。
```

### 模式2：现象描述，数字佐证
```
鹿晴线存在"伪专家"现象：Z5/Z6层级员工大量时间用于"评测标注""圈店圈品"等
基础执行工作——在该线130人中，38人（29.2%）被判定为重度错配。
```

### 模式3：对比揭示差距
```
AI工具渗透率差异显著：暮声线71.9% vs 鹿晴线19.2%，
同属一个团队体系，但渗透率相差近4倍。
```

### 模式4：基线→目标，量化改善空间
```
SOP覆盖率：当前全局92.3%，鹿晴线仅87.2%（低于90%基准），
目标：Q3末提升至95%+，重点补建鹿晴线缺口约37个SOP节点。
```

### 模式5：构成分解，占比直观
```
335人的工作属性构成：
  纯标产   113人（33.7%）  ← 离岸化主力来源
  标产为主  43人（12.8%）  ← 部分可离岸
  混合型    92人（27.5%）  ← 需逐人判断
  非标为主  41人（12.2%）  ← 暂不考虑离岸
  纯非标    42人（12.5%）  ← 核心专家，保留驻场
```

## 文档结构模板

### 通用模板（适用于大多数分析报告）

```
【导读】（多角色文档必须有）
  角色 → 建议阅读的章节

一、核心数字快览（≤1屏）
  3-6个数字卡片，覆盖最重要的维度
  一句话定性：最关键的分布/问题特征

二、全景框架（分类体系）
  场景/类型分类表格：名称 | 特征 | 判定标准 | 优化方向
  整体人员结构分布（数字卡片 or 简表）

三、核心发现（3条以内）
  按重要性排序（不按发现顺序）
  每条：一句话摘要 → 数字 → 典型案例 → 影响

四、分维度深化
  按业务线/角色/项目/时间轴等维度展开
  每个维度：关键指标横向对比表格 + 差异最大点的文字解释

五、优先级行动清单
  | 模块 | 具体行动（谁/做什么/目标值） | 优先级 |
  P0：立即处理
  P1：近期（本季度内）
  P2：持续跟进

六、附录：口径定义
  | 指标 | 定义 | 计算公式 | 数据来源 |
```

## HTML 形式的写作方式（交互式看板）

HTML 形式和文档形式是两种完全不同的信息传递机制，有自己独立的"写法"规范。

### HTML 形式的本质差异

| 维度 | 文档形式 | HTML 交互看板 |
|----|---------|-----------| 
| 读者行为 | 从头到尾线性阅读 | 带着问题来，直接点进感兴趣的入口 |
| 信息密度 | 高（需要上下文解释） | 低（每个卡片自解释） |
| 数字呈现 | 嵌入段落，有文字包裹 | 大字独立展示，颜色编码 |
| 深度探索 | 翻页、章节跳转 | 点击下钻，抽屉展开，弹出明细 |
| 内容来源 | 写作时固定 | 实时从数据渲染，数字永远最新 |
| 声明方式 | 段落前的文字块 | Tooltip 里写口径，不污染主界面 |

---

### HTML 写作原则一：每个数字是一个入口，不只是一个数字

HTML 形式里，数字 = 可点击的筛选器。每个 stat-card 点击后弹出人员明细列表，这是文档形式做不到的。

**写法规范**：
- stat-card 的 `label` 要精确描述口径，不能只写"错配"，要写"重度错配"
- `value` 只放最终数字，不放百分比（百分比放 Tooltip 或下一层）
- 有 `warn`/`good` 状态色的卡片必须配 `TooltipIcon`，解释判定标准
- 每个数字卡片的 `onClick` 必须跳到真实人员列表，不能是空操作

**对照（index.html 里的实际写法）**：

```jsx
// 正确：数字可下钻，Tooltip有口径，颜色有含义
<div className="stat-card warn"
  onClick={() => drill('重度错配', m => {
    const s1 = m.scenes['S1 基础标产执行'] || 0;
    return s1 >= 50 && (highLevels.includes(m.level) || m.mode.includes('HRO'));
  })}>
  <div className="value">{overview.mismatchSevere}</div>
  <div className="label">重度错配<TooltipIcon tip={TOOLTIPS.mismatchSevere}/></div>
</div>

// 错误：数字是孤立的，没有交互，没有口径
<div className="stat-card">
  <div className="value">71</div>
  <div className="label">重度错配人数</div>
</div>
```

### HTML 写作原则二：section-subtitle 是文档形式里的"一句话摘要"

每个 Tab 的 `section-subtitle` 相当于文档里的导读句，必须说清楚"这个 Tab 能看到什么、怎么用"。

**写法规范**：格式是"内容层次 → 内容层次"的路径描述，不是空话。

```jsx
// 正确：告诉读者信息的层次结构
<div className="section-subtitle">
  每条业务线的体检报告：概况→问题→项目优先级→人员明细
</div>

// 正确：指明数据口径和范围
<div className="section-subtitle">
  搜索1线驻场资源运营管理升级专项 — 基于{overview.total}人详细工作盘点数据的诊断分析
</div>

// 错误：空洞，没有信息
<div className="section-subtitle">
  以下是数据分析结果
</div>
```

---

### HTML 写作原则三：Tooltip 是 HTML 独有的口径承载位置

文档形式要在附录里写口径定义表；HTML 形式把口径定义放在 `TooltipIcon` 里，鼠标悬停时展示，不干扰主界面。

**Tooltip 的写作格式**（index.html 里的标准）：

```
口径名称
口径：[一句话定义]
判定条件（满足任一即算）：
① [条件一]
② [条件二]
例：[具体的人/项目例子，说明什么情况算、什么情况不算]
```

**实际例子**（`TOOLTIPS.mismatchSevere`）：

```
重度错配
口径：高层级或HRO花大量精力做基础标产
判定条件（满足任一即算）：
① Z4/Z5/Z6/W5/W5+/W6，且该人S1占比≥50%
② 驻场HRO（任意层级），且该人S1占比≥50%
例：Z5 HRO，S1=70%→重度错配；Z3 BPO，S1=80%→不算
```

**规则**：Tooltip 的例子必须有正例和反例，只有正例读者不知道边界在哪。

### HTML 写作原则四：热力图 = 用颜色写文字说不清楚的交叉关系

文档里写"鹿晴线在S1的参与占比较高"是一句话；HTML 里用热力矩阵，一眼就能看到所有业务线在所有场景的分布，还能对比哪条线哪个场景更集中。

**写法规范**（index.html 里的业务线×场景热力图）：

```jsx
// 颜色深浅 = 数值大小（intensity映射到透明度）
const intensity = Math.min(val / 50, 1);
background: val > 0 ? `${sceneColor}${Math.round(intensity * 40).toString(16)}` : ''

// 每个热力格子也是可点击的下钻入口
onClick={() => onDrill(`${bl.name} ${scene}参与者`,
  allMembers.filter(m => m.bizLine === bl.name && (m.scenes[scene] || 0) > 0))}
```

**热力图的文字配合**：热力图旁边必须有 Tooltip 解释"颜色越深代表什么"，不能让读者自己猜。

```
业务线 × 场景热力分布（Tooltip内容）：
单元格数值=该业务线在该场景的平均占比
口径：取该业务线中该场景占比>0%的人，计算其均值（参与者口径，非全员）
颜色越深=该场景在该业务线的参与者投入越集中
例：某格显示45%=该业务线参与该场景的人，平均45%精力在做这类工作
```

### HTML 写作原则五：diagnosis-box 是条件渲染的文字，不是固定文字

HTML 形式里，文字是从数据动态生成的，不是硬编码的结论。当数据变化时，诊断结论应该自动变化。

**写法规范**：

```jsx
// 正确：数字来自数据，条件渲染决定是否显示
{bl.mismatchSevere > 5 && (
  <div className="diagnosis-box danger">
    <h4>重度错配突出</h4>
    <ul>
      <li>{bl.name}有<strong>{bl.mismatchSevere}人</strong>重度错配，
          高层级/HRO人力做基础标产</li>
    </ul>
  </div>
)}

// 如果所有指标都正常，显示健康状态（不是空白）
{bl.mismatchSevere <= 5 && bl.costInversion === 0
  && bl.sopCoverage >= 80 && bl.chaosIndex <= 3 && (
  <div className="diagnosis-box success">
    <h4>结构相对健康</h4>
    <ul><li>错配少、SOP覆盖良好、职责清晰</li></ul>
  </div>
)}
```

**规则**：诊断 box 的阈值必须有业务依据（如"mismatchSevere > 5"是"超过5人才算突出问题"），阈值要和文档里的判定标准对齐。

### HTML 写作原则六：展开/收起的层级设计

HTML 独有的"渐进披露"能力：默认只展示摘要数字，展开后才看到明细。这需要设计好三层信息层次：

```
第一层（始终可见）：业务线名 + 总人数 + HRO/BPO/错配 概要数字
第二层（点击展开）：4个迷你指标卡（离岸候选/SOP覆盖/AI渗透/打杂指数）
                    核心问题诊断 box
                    项目优先级列表（每项目一行：名称+关键指标）
第三层（再次展开）：项目内的人员明细表格（姓名/层级/场景分布）
```

**写法规范**：
- 第一层数字要覆盖"是否有问题"（红色警告数字）
- 第二层指标要覆盖"问题严重程度"（和基准对比）
- 第三层是行动依据（具体是哪些人需要处理）
- 每一层展开按钮的文字要说明展开后能看到什么：`展开项目详情 ▼` 而不是 `查看更多`

---

### HTML 写作原则七：SceneStack 是人员场景分布的标准可视化

每个人的场景分布（S1 60% / S2 20% / S4 20%）在 HTML 里用 SceneStack 组件展示——一条彩色进度条，每段颜色代表一个场景，宽度代表占比。

**使用规范**：
- 在人员明细表格的最后一列放 SceneStack，不要用文字列举占比
- SceneStack 可点击打开人员详情弹窗
- 排序时把重度错配（S1高+高层级）的人放在最前面（视觉上红色背景行聚集在顶部）

```jsx
// 表格排序：错配程度作为排序依据
[...members].sort((a, b) => {
  const rank = m => {
    if (isSevere(m)) return 0;   // 重度错配排最前
    if (isMild(m)) return 1;     // 轻度错配次之
    return 2;                     // 正常排最后
  };
  return rank(a) - rank(b);
})
```

### 两种形式的分工关系

| 信息需求 | 用文档形式 | 用 HTML 形式 |
|--------|---------|------------|
| 让领导快速理解全局 | ✅ 一页总结，可打印 | 需要培训才能用 |
| 找出具体哪个人需要处理 | ❌ 不适合，要翻表格 | ✅ 筛选+下钻直达 |
| 说明判定逻辑和口径 | ✅ 附录章节 | ✅ Tooltip（更轻量） |
| 对比不同业务线差异 | ✅ 表格并排 | ✅ 热力图（更直观） |
| 存档和分享 | ✅ 可导出/复制 | 需要部署，分享困难 |
| 数据更新后自动刷新 | ❌ 需要重新写 | ✅ 绑定数据自动渲染 |
| 加入分析结论和判断 | ✅ 写作者的主观判断 | ❌ 只能做客观展示 |

**实践规则**：
- HTML 看板 = 数据层，展示什么、多少、在哪里
- 文档 = 判断层，为什么、怎么办、谁来做
- 两者互相引用：文档里说"详细人员明细见看板错配预警 Tab"；看板里的诊断 box 说"对策建议详见报告第六章"

## 质量自检清单（写完必查）

**内容层面**：
- [ ] 每个结论是否有内嵌数字（不是"去图表里看"）？
- [ ] 每个数字是否有对比基准（同比/目标值/跨维度）？
- [ ] 每条建议是否有目标值和时间节点？
- [ ] AI生成的内容是否有显式声明，且声明在内容前面而不是后面？

**结构层面**：
- [ ] 章节顺序是否服从读者需要，而不是写作顺序？
- [ ] 是否有导读（多角色文档）？
- [ ] 是否有全景框架图/表（让读者先建立整体认知）？
- [ ] 口径定义是否在附录里统一列出？

**语言层面**：
- [ ] 是否有"较多/一定程度上/有所提升"等模糊词语？（必须替换）

- [ ] 语气是否匹配读者角色（管理层/执行层/协同方）？

**可视化层面**：
- [ ] 核心数字是否从正文里脱离出来，用卡片/表格独立展示？
- [ ] 分类体系是否用表格（不是散文列举）？
- [ ] 优先级是否有颜色区分（P0红/P1橙/P2蓝）？

- [ ] section-subtitle 是否描述了层次结构（"A→B→C"格式）？
- [ ] 热力图是否有 Tooltip 解释颜色含义？
- [ ] diagnosis-box 是否用条件渲染（数据驱动），不是硬编码结论？
- [ ] 展开/收起按钮文字是否说明展开后看到什么？
- [ ] 人员表格是否按错配程度排序（问题最严重的排最前）？

## 常见反模式速查

| 反模式 | 真实原因 | 修正 |
|------|--------|-----|
| 数字在图表，文字只说"如图所示" | 写者以为读者会去看图 | 把关键数字写进文字，图作为补充 |
| 七大场景散落在正文各处 | 没有先建立全景框架 | 先放一张完整分类表，再展开细节 |
| "建议加强/持续推进" | 没有想清楚"谁做什么到什么程度" | 补充：谁负责、目标值、截止时间 |
| 全文一个字号、一种格式 | 没有区分信息优先级 | 最重要的数字用大字卡片，次要内容用正文 |
| AI分析结论和人工判断混在一起 | 省略了内容来源标注 | 每段AI分析前加斜体声明块 |
| 为不同角色写了同一份文档 | 没有做读者分析 | 加导读索引，或拆分为不同章节 |
| 文档结构跟着分析思路走 | 混淆了"我怎么想的"和"读者需要什么" | 先定读者最关心的问题，再组织结构 |
