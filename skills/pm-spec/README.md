# pm-spec — PRD 生成与批注系统

> 把需求讨论的结论，输出为一份**研发能直接开工**的结构化 PRD 网页，带在线批注、版本归档、公网预览。

## 它解决什么问题

需求讨论完了，结论散落在聊天记录里，研发拿到的 PRD 写着"支持查看详情"——然后猜着做，上线后返工。

pm-spec 强制按固定格式输出每个功能节点的**页面入口→展示内容→操作效果→异常处理→功能联动**，写不清楚不让过（5 条自检铁律）。研发看完知道"点哪里、出什么、改 A 会不会坏 B"。

## 核心能力

- **结构化 PRD 生成**：问题→目标→功能节点（按用户操作流程组织，不按功能分类）→联动→异常→里程碑→开放问题
- **在线批注**：生成的 PRD 网页支持选中文字添加批注，批注持久化到 `annotations.json`，支持修改、导出、历史回顾
- **版本归档**：每轮迭代自动归档旧版 PRD + 批注到 `planning/archive/`，新版本预览干净无残留批注
- **公网预览**：`prd-preview-server.js` 启动预览服务 + `port-mapping` 获取公网链接，发给协作者直接看

## 使用方式

```
/pm-spec          # 从已有需求文档生成 PRD
```

通常在 `demand-discovery`（需求聊透）之后触发，先把需求聊清楚，再用 pm-spec 输出正式 PRD。

## 自检铁律（写完每个功能节点必须过）

1. **交互可执行？** — "支持查看详情"不行，"点击详情按钮→弹出侧边栏，展示 XXX 字段"才行
2. **约束可验证？** — "支持配置人数"不行，"3≤M≤10，创建后不可修改"才行
3. **联动可追踪？** — "两个功能有关联"不行，"当配置A=X时，模块B 的行为变为……"才行
4. **开放问题是业务的？** — "用什么框架"不行，"超时未提交是否自动保存草稿"才行
5. **改了 A，B/C/D 怎么变？** — 每个联动写清触发条件、影响范围、不影响什么

## 在研发流水线中的位置

```
demand-discovery（聊需求）→ pm-spec（写 PRD）→ serial-agent-handoff（开发）
```

属于 autonomous-studio 六阶段流水线的第 ② 阶段（PRD），产出：`prd.md` + `prd.html` + `prd.json` + `test-cases.md`。

## 目录结构

```
pm-spec/
├── SKILL.md                # 技能定义（生成规则+批注规则+自检铁律）
├── README.md               # 本文件
├── prd-preview-server.js   # 预览服务器（含批注 API + 历史归档 API）
├── annotation-script.js    # 批注系统前端脚本（嵌入生成的 PRD）
├── annotation-bar.html     # 批注栏 HTML 片段（嵌入生成的 PRD）
└── example.html            # PRD 示例
```
