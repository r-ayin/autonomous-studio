# <功能名>开发任务交接记录

本文用于 Codex 主控与串行子 agent 交接。本文只记录执行顺序、边界、验收和交接状态，不替代 PRD / 技术合同 / 实施设计。

## 1. 总体工作方式

- 主控 Codex 负责拆分任务、启动子 agent、审核改动、集成、测试、提交和部署协调。
- 子 agent 默认串行执行，不并行修改同一阶段。
- 每个子 agent 一次只负责 1-2 个清晰功能块。
- 一个子 agent 完成并通过主控审核以后，下一位子 agent 才继续。
- 每阶段结束后，主控更新本文的状态、变更文件、验证结果和遗留问题。
- 如需突破本文约束，必须先回到主控确认。

**上下文预算约束（防 8.5x 膨胀，GLM-5.2 成本优化）**：

- 本 handoff 文件是跨子 agent / 跨会话的**唯一共享知识载体**。子 agent 不得要求主控复述完整 PRD 或重复贴大段代码——只读本文 + 按需 Read 指定文件路径。
- 主控 spawn 子 agent 时，只传递：本 handoff 文件路径 + 当前 task 的 prd.json 任务 id + harness 路径清单。**禁止在子 agent prompt 内联完整 prd.json**，改为指令"Read planning/prd.json 中 task {id} 的定义"。
- 子 agent 返回后，主控**立即摘要**其输出，不把完整子 agent 输出保留在后续上下文里。
- 执行日志（第 10 节）超过 20 条时，主控把旧条目压缩为一段摘要，只保留最近 5 条明细。
- 每个子 agent 的 prompt 上限目标：≤ 8000 tokens（不含系统指令）。超长说明任务未拆细，回主控再拆。
