### AE2-H-001 [HIGH] 信心阈值四处不一致 — 引擎行为不可预测

- **文件**: SKILL.md / decision-agent-prompt.md / README.md / ARCHITECTURE.md
- **行号**: SKILL.md L142-148; decision-agent-prompt.md L273-289; README.md L147-153; ARCHITECTURE.md L70-78
- **类型**: route-fix
- **finding**: 四个核心文档对信心阈值→行动级别的映射给出**四种不同数值**：

  | 行动级别 | SKILL.md (§⑤) | decision-agent-prompt.md (§⑤) | README.md | ARCHITECTURE.md |
  |---------|---------------|-------------------------------|-----------|-----------------|
  | OBSERVE | 0-30          | 0-30                          | <40       | 0-10            |
  | SUGGEST | 31-59         | 31-59                         | 40-59     | 11-30           |
  | PREPARE | (无)          | (无)                          | 60-70     | 31-50           |
  | ACT_NOTIFY | 60-70      | 60-70                         | 71-84     | 51-70           |
  | ACT_SILENT | 71-100     | 71-100                        | 85+       | 71-100          |

  后果：
  1. 子代理按 decision-agent-prompt.md 算出 confidence=65 → ACT_NOTIFY
  2. 但如果用户读 README.md 期望 confidence=65 是 PREPARE（不执行）→ 信任崩塌
  3. SKILL.md 的调度器 Step 4 处理返回时也可能用错阈值判断是否执行
  4. ARCHITECTURE.md 的旧版阈值（0-10/11-30/31-50/51-70/71-100）完全不同于 v2.2 的操作类型分档
  
  这是 v2.2 升级时「操作类型分档」引入后未同步到所有文档的典型版本漂移。
- **remediation**: 选定 decision-agent-prompt.md §⑤ 为权威源（它最详细且有操作类型分档），其他三处对齐。README.md 和 ARCHITECTURE.md 补充操作类型分档说明。SKILL.md Step 4 引用 prompt 而非自定阈值。
- **证据**: Read 四文件对应行号直接可见数值差异。

### AE2-H-002 [HIGH] save-checkpoint.py `find_task_files()` 使用 Path 但未在模块级导入 — 任何调用即 NameError

- **文件**: `.claude/hooks/save-checkpoint.py`
- **行号**: 137, 215
- **类型**: route-fix
- **finding**: `find_task_files()` 函数（L134-141）使用 `Path(PROJECT_DIR)` 遍历目录，但 `from pathlib import Path` 仅在 L215（`if __name__ == "__main__":` 块内）。这意味着：
  1. 当 save-checkpoint.py 被 incremental-save.py 通过 `subprocess.run([sys.executable, CHECKPOINT_SCRIPT])` 调用时，它在 `__main__` 上下文运行，Path 可用——**当前不会崩溃**。
  2. 但当 save-checkpoint.py 被 Claude Code hook 系统直接 import 或通过 PreCompact/Stop/SessionEnd 事件触发时（取决于 hook 注册方式），如果 hook runner 以 module import 方式加载，`__main__` 块不执行 → Path 未导入 → `find_task_files()` 调用即 NameError。
  3. 更重要的是：**build_checkpoint() 当前不调用 find_task_files()**，所以这个 bug 是潜伏的。但任何未来扩展（如增加 task 追踪）都会触发它。
  4. 即使作为 `__main__` 运行，import 放在 main 块内也是反模式——它使函数的依赖关系隐式且脆弱。
  
  注：此 finding 与 audit-2026-07-01-009 AE-H-002 部分重叠（该审计指出 Path import 位置问题），但该审计侧重 audit log 路径错误。本轮确认 Path 问题仍存在且未修复，并补充了 import-as-module 的触发场景分析。
- **remediation**: 将 `from pathlib import Path` 移到文件顶部 imports 区（L16 附近）。
- **证据**: Read 文件 L137 可见 `Path(PROJECT_DIR)`；L215 才 `from pathlib import Path`。

### AE2-M-002 [MEDIUM] decision-agent-prompt.md §0.2 冷启动 CS-ACT 要求 git stash + backup branch 双保险 — 过度复杂且冲突风险

- **文件**: `.claude/skills/autonomous-engine/decision-agent-prompt.md`
- **行号**: 59-83
- **类型**: route-fix
- **finding**: §0.2 Phase CS-ACT 三步安全检查要求：
  ```
  ① python .claude/hooks/save-checkpoint.py
  ② git stash 或 git branch backup-{timestamp}
  ③ 验证回滚路径
  ```
  然后 §⑥ EXECUTE (L308-316) 又说：
  ```
  1. save-checkpoint.py
  2. git stash && git stash apply 或 git branch backup-{timestamp}
  ```
  
  问题：
  1. "git stash **&&** git stash apply" 是无意义的往返操作（stash 后立即 apply 回来 = 没做任何保护）。应该是 "git stash" 单独，或者 "git branch backup-xxx" 单独。
  2. §0.2 和 §⑥ 对同一流程给出略微不同的步骤描述，子代理执行时可能混淆。
  3. save-checkpoint.py 本身已经记录了 git status，再额外做 git stash/branch 是冗余的安全层。对于可逆操作（confidence ≥60），checkpoint JSON + git commit 已足够回滚。
- **remediation**: 统一 §0.2 和 §⑥ 的检查点保护流程为单一权威描述。删除 "git stash && git stash apply" 的错误组合。简化为：checkpoint → 执行 → 失败则 git reset --hard。
- **证据**: Read decision-agent-prompt.md L59-83 和 L308-316。

### AE2-M-003 [MEDIUM] incremental-save.py 与 save-checkpoint.py 同在 Stop 事件触发 — 双重 checkpoint 写入竞争

- **文件**: `.claude/hooks/incremental-save.py`, `.claude/hooks/save-checkpoint.py`
- **行号**: incremental-save.py L47-56; save-checkpoint.py L213-222
- **类型**: route-fix
- **finding**: 
  - save-checkpoint.py 触发时机包括 "Stop"（文件头注释 L6）
  - incremental-save.py 由 "后台 bash 循环每 120 秒调用"（L3），但其内部调用 `subprocess.run([sys.executable, CHECKPOINT_SCRIPT])`（L50-56）也会触发 save-checkpoint.py
  
  当两者在同一时刻触发时（例如会话结束时的 Stop 事件恰好与 120s 定时器重合）：
  1. 两个进程同时写 `checkpoints/checkpoint_*.json` → 文件名基于时间戳，可能相同 → 写入竞争
  2. 两个进程同时 `shutil.copy2(filepath, LATEST_LINK)` → latest.json 可能被截断或写入不完整
  3. `prune_old_checkpoints()` 并发执行 → 可能误删对方刚写的文件
  
  实际影响：低频（120s 间隔与 Stop 精确重合概率低），但一旦发生会导致 resume-checkpoint 读到损坏的 latest.json → 恢复失败。
- **remediation**: incremental-save.py 加文件锁（fcntl.flock）或检测 save-checkpoint.py 是否已在运行（PID 文件）。或在 incremental-save.py 中不重复调用 save-checkpoint.py，改为只做 memory timestamp 更新。
- **证据**: Read incremental-save.py L47-56 可见 subprocess 调用 save-checkpoint.py。save-checkpoint.py L182-193 write_checkpoint 无锁保护。

### AE-S-003 [STRUCTURAL] Hook 触发时机重叠缺乏协调机制

- **涉及文件**: save-checkpoint.py / incremental-save.py / decision-observer.py
- **描述**: 多个 hook 在同一事件（Stop）上触发，且存在嵌套调用（incremental-save → save-checkpoint）。没有统一的 hook 编排层来保证执行顺序、防止重复、处理竞争。当前靠各 hook 自行 try/except 静默失败来避免崩溃，但这掩盖了数据完整性问题。
- **建议**: 引入 hook coordinator（如一个 master-hook 按序调度子 hook），或利用 Claude Code 的 hook 优先级机制（如果支持）。
