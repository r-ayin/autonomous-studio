## Findings

### H-001 [HIGH] termux-listener TCP message unsanitized → command injection via notification title/content
- **File**: termux-listener.py
- **Line**: 80-94
- **Kind**: route-fix
- **Category**: security
- **Evidence**: `data = conn.recv(1024).decode(...)` then split by `|` into `title`/`message`, passed directly to `subprocess.run(["termux-notification", "--title", title, "--content", message, ...])`. Although list-form avoids shell injection, termux-notification itself interprets escape sequences and newlines; a malicious sender can inject `--vibrate` or other flags by embedding `\n--vibrate\n...` in title (argv boundary is respected but termux-notification may parse content as multi-line control). More critically: recv(1024) with no length validation means an attacker can send >1024 bytes and the message is silently truncated mid-UTF8, producing garbage in notification.
- **Reproduction**: `echo -n "high|Test|--vibrate 999,999,999" | nc 127.0.0.1 9999` — if termux-notification parses embedded flags from content, this triggers unintended vibration pattern. Even without flag injection, oversized messages corrupt display.
- **Remediation**: (1) Validate priority against whitelist `{default, high, low, min, max}` before use. (2) Strip newlines/null bytes from title/message (`re.sub(r'[\x00-\x1f]', ' ', s)`). (3) Cap message at 500 chars with explicit truncation marker. (4) Add try/except around send_notification with logging.

### M-001 [MEDIUM] watchdog.sh log truncation race: tail+mv not atomic under concurrent cron overlap
- **File**: watchdog.sh
- **Line**: 72-74
- **Kind**: route-fix
- **Category**: integrity
- **Evidence**: `tail -100 "$LOG_FILE" > "$LOG_FILE.tmp" && mv "$LOG_FILE.tmp" "$LOG_FILE"` — between tail reading and mv replacing, another watchdog instance (cron tick during slow I/O) can append to LOG_FILE; those lines are lost when mv overwrites. flock protects main logic but NOT the truncation block (truncation runs outside lock scope conceptually since it's inside the locked region but tail reads while holding fd 200 — however a second cron-invoked instance will fail flock and exit, so actual race window is narrow). Real risk: tail itself reads incrementally; if log grows during tail, output may miss lines at boundary.
- **Remediation**: Use `cp "$LOG_FILE" "$LOG_FILE.bak" && tail -100 "$LOG_FILE.bak" > "$LOG_FILE.tmp" && mv "$LOG_FILE.tmp" "$LOG_FILE" && rm -f "$LOG_FILE.bak"` or simpler: hold flock through entire truncation sequence explicitly (already held, but document intent). Acceptable risk given single-instance guarantee, but worth hardening.

### M-002 [MEDIUM] watchdog.sh AGE variable used uninitialized when LATEST missing
- **File**: watchdog.sh
- **Line**: 66
- **Kind**: route-fix
- **Category**: correctness
- **Evidence**: Line 66 logs `checkpoint age ${AGE:-unknown}s` — AGE is only set inside `if [ -f "$LATEST" ]` block (line 44). When LATEST doesn't exist on first run, AGE is unset; `${AGE:-unknown}` handles expansion safely BUT `set -u` (from `set -euo pipefail` line 13) treats `${var:-default}` as valid, so no crash. However the HEALTH log becomes misleading ("unknown" persists across iterations even after LATEST appears because AGE isn't reset between loop iterations when LATEST disappears then reappears). Minor correctness issue.
- **Remediation**: Initialize `AGE=0` before the if-block, or reset at top of each loop iteration.

### M-003 [MEDIUM] watchdog-boot.ps1 hardcoded E:\x-tool path breaks on non-standard installs
- **File**: watchdog-boot.ps1
- **Line**: 17-19
- **Kind**: direction-shift
- **Category**: portability
- **Evidence**: `$LogFile = "E:\x-tool\.claude\.watchdog_boot.log"`, `$WatchdogScript = "/mnt/e/x-tool/.claude/watchdog.sh"`, `$CronPattern = "*/5 * * * * /mnt/e/x-tool/.claude/watchdog.sh"` — all hardcoded to E: drive. If user installs to C: or D:, boot script silently writes to wrong location, crontab points to nonexistent path, watchdog never runs. No env-var override, no auto-detection.
- **Remediation**: Read base path from registry/env var (`$env:CLAUDE_PROJECT_DIR` or detect via WSL mount table), fall back to E:\x-tool with warning. This is a direction-shift because it changes the deployment contract for Windows users.
