## Finding 详情

### PD-H001 — 时间门控时钟漂移风险（HIGH / route-fix）

**位置**: `scripts/resume-next-batch.js:51-55`

```js
const successTime = new Date(batchEvent.updated_at).getTime();
const elapsedMs = Date.now() - successTime;
const requiredMs = observeMinutes * 60 * 1000;
if (elapsedMs < requiredMs) { ... reject ... }
```

**问题**: `updated_at` 由 aone-agent-server 生成（服务器时钟），`Date.now()` 是执行脚本的本地机器时钟。两者不保证同步：
- 本地钟快 5 分钟 → 观察期实际 0 分钟就通过 → HIGH 异常窗口内推进部署
- 本地钟慢 5 分钟 → 观察期永远不满 → 部署卡死

**影响**: 生产部署的观察期是防劣化最后一道防线；时钟漂移可直接绕过或永久阻塞。

**修复方向**: 改用服务器提供的当前时间（API response header Date 或专门 endpoint），或在 batch SUCCESS 时记录本地 `local_success_time` 并后续用 `Date.now() - local_success_time` 做 delta（消除跨时钟比较）。最小改动：resume-next-batch 调 apiGet('/api/v1/sre/time') 拿 server_now，用 server_now - updated_at。

### PD-H002 — Terminal-state payload-only update 无审计（HIGH / route-fix）

**位置**: `scripts/lib/event-client.js:172-179`

```js
if (TERMINAL_STATUSES.has(existingEvent.status)) {
  if (!newPayloadFields || status) { return null; }
  // ← falls through to merge + PUT, but no audit-log call
}
```

**问题**: 终态事件允许追加 payload（设计意图：deploy-complete 后补 observation_result），但这条写入路径完全没有 audit-log。DO B 要求 deploy 敏感路径留痕；observation_result 直接影响后续批次是否推进，其修改不可追溯。

**修复方向**: updateEvent 内 payload-only 分支加 writeAuditLog({ auditAction: 'event-payload-append', ... })。需在 audit-log-helper ACTION_MAP 新增 'event-payload-append' mapping。

### PD-H003 — Poller 脚本零审计日志（HIGH / direction-shift）

**位置**: `scripts/poll-build.js` 全文, `scripts/poll-pre-check.js` 全文

**问题**: 两个 poller 是部署决策的关键输入（构建是否成功、准入是否通过），它们：
1. spawn `a1` CLI 获取外部状态
2. 解析结果并 updateEvent
3. 输出 JSON 给 agent 决定是否继续

整条链路无 audit-log。DO B 明确 "pipeline-trigger / deploy / compliance_check" 是敏感路径。poller 的输出直接驱动部署推进，却无任何事后可追溯的记录。

**修复方向**: 每个 poller main() 结尾根据 isTerminal/done 写一条 audit-log（action=pipeline_status_poll / precheck_poll）。这需要在 audit-log-helper ACTION_MAP 新增两个 action。因为涉及新增公共接口（ACTION_MAP 新 key），opt-worktree.sh 会判 direction-shift。

### PD-M001 — 双凭证共存无互斥（MED / route-fix）

**位置**: `scripts/lib/http-base.js:28-36`

```js
if (process.env.CODE_PRIVATE_TOKEN) {
  headers['X-Agent-Authorization'] = `Code ${process.env.CODE_PRIVATE_TOKEN}`;
} else if (process.env.DEVOUT_TOKEN) {
  headers['token'] = process.env.DEVOUT_TOKEN;
}
```

**问题**: 两个 env 同时存在时 CODE_PRIVATE_TOKEN 胜出，但无任何 warning。operator 可能以为 DEVOUT_TOKEN 生效而排查错方向。

**修复方向**: 两者共存时 stderr WARN 一次（启动时或首次 getHeaders 调用时）。

### PD-M002 — safeJsonParseCli audit action 不在白名单（MED / route-fix）

**位置**: `scripts/report-event.js:30-40` + `scripts/lib/audit-log-helper.js:40-46`

report-event.js 在 JSON parse 失败时调用 `writeAuditLog({ auditAction: 'cli-json-parse-failed', ... })`，但 ACTION_MAP 不含此 key → audit-log-helper L63-66 warn+return → 恶意/错误的 CLI JSON 注入尝试完全不留痕。

**修复方向**: ACTION_MAP 新增 `'cli-json-parse-failed': { action: 'input_validation', sensitiveLevel: 'medium' }`。

### PD-M003 — Observation gate 静默跳过（MED / route-fix）

**位置**: `scripts/resume-next-batch.js:67-96`

当 `observation_result` 不存在时（report-observation.js 从未成功运行），gate 完全跳过，只靠 M-004 fix 的 stderr WARN。但 agent 可能不读 stderr，直接拿 approved:true 推进。

**修复方向**: 增加 `--strict-observation` flag 或环境变量 RESUME_REQUIRE_OBSERVATION=1；启用时 observation_result 缺失 → approved:false + 明确 error message。默认保持向后兼容。

### PD-M004 — Sunfire 错误同质化（MED / route-fix）

**位置**: `scripts/lib/sunfire-client.js:105-114`

```js
} catch (err) {
  return { skipped: true, reason: `Sunfire request failed: ${err.message}` };
}
```

JSON.parse(raw) 抛 SyntaxError（API 返回 HTML/空响应）与 https.request 网络错误被同化为 skipped。前者可能意味着 API 变更/中间人/数据损坏，应告警而非静默降级。

**修复方向**: 区分 err instanceof SyntaxError（API 响应异常，返回 `{ skipped: true, reason, severity: 'warning' }`）与网络错误（severity: 'info'）。resume-next-batch 据此决定是否需要人工介入。

### PD-M005 — complete-task 非原子写（MED / route-fix）

**位置**: `scripts/complete-task.js:43-51`

```js
await createEvent(taskId, 'finish', { summary }, null, finishStatus);
await updateTask(taskId, { status: finishStatus, ... });
```

createEvent 成功后 updateTask 失败 → 事件系统认为已完成，task 表仍 RUNNING。Phase 0 断点恢复查 SUCCESS 事件看到 finish，但 getTask 显示 RUNNING → 状态矛盾。

**修复方向**: 先 updateTask 再 createEvent；或在 updateTask 失败时回滚事件（deleteEvent API 若存在）；或 Phase 0 增加交叉校验（finish SUCCESS 事件 + task.status 必须一致）。最小改动：调换顺序 + updateTask 失败时 stderr ERROR + exit(1) 让 agent 重试。

### PD-L001 — audit-log result 默认 success（LOW / route-fix）

**位置**: `scripts/lib/audit-log-helper.js:84`

```js
result: result || 'success',
```

caller 传 `result: ''` 或 `result: 0` 会被改写为 'success'。虽然当前所有 caller 都传显式字符串，但这是隐患。

**修复方向**: 改为 `result: result ?? 'success'`（nullish coalescing）。

### PD-L002 — parseInt NaN 透传（LOW / route-fix）

**位置**: `scripts/report-event.js:227-229`

```js
const batchIndex = parseInt(values['batch-index'], 10);
```

CLI 传 `--batch-index abc` → batchIndex=NaN → payload.batch_index=NaN → JSON.stringify 变 null → findBatchEvent 永远匹配不到 → 每次调用创建新事件。

**修复方向**: isNaN(batchIndex) → console.error + process.exit(1)。

### PD-L003 — extractIdFromLine 误匹配（LOW / route-fix）

**位置**: `scripts/poll-pre-check.js:65-68`

```js
const match = line.match(/(\d{4,})/);
```

a1 输出含时间戳 `2026-07-03T23:45:00` 或版本 `v1.2.3456` 时会误抽为 ID。

**修复方向**: 收紧正则为 `/id[=:]\s*(\d+)/i` 或仅在已知列格式下行抽取。fallback 保留但加 stderr WARN。

### PD-L004 — findBatchEvent O(N) 客户端过滤（LOW / route-fix）

**位置**: `scripts/lib/event-client.js:90-102`

服务端 query API 不支持 batch_index 过滤，全量拉取再 client-side find。10 批次部署尚可，100 批次会成为瓶颈。

**修复方向**: 短期可缓存最近一次 query 结果；长期推动服务端支持 batch_index filter。本轮不修，记 structural-debt。

## 审计结论

prod-deploy 整体架构合理：lib 层抽象干净、event-client 有 safeJsonParse 容错、audit-log-helper fail-safe 设计正确。但**时间门控的跨时钟比较**（H001）和**关键 poller 零审计**（H003）是两个真正的结构性风险——前者可让观察期形同虚设，后者让部署决策不可追溯。其余 M/L 级问题多为防御性编程缺失，可在后续轮次逐步收敛。
