## Summary

personal-assistant/web 是 React 单文件 bundle + Node dev-server 的前端入口层。本模块与已知后端安全问题（H-001 .env 明文 key、H-002 CORS * 零认证、路径遍历）**深度耦合放大风险**：前端无认证假设 + CDN 无 SRI + 无 CSP + api key 在前端流转 = 完整攻击链。React 自动转义提供了基本 XSS 防护，但无法弥补传输层和配置层的系统性缺陷。

## High Findings

### W-001 · fetch 全链路零认证（与后端 H-002 耦合）
- **file**: web/api.js
- **line**: 9-27
- **severity**: high
- **kind**: structural
- **finding**: 所有 fetch 请求（get/post）均不携带 credentials（无 `credentials:'include'` 或 `'same-origin'`），也无 Authorization header。前端假设后端无认证即可操作——与已知后端 H-002（CORS `*` 零认证）耦合，任何同网络攻击者可直接调用所有破坏性端点（/ingest, /chat, /verify, /settings/llm, /inbox/upload）。
- **remediation**: 1) 后端加认证（至少 session/cookie 或 bearer token）；2) api.js 的 get/post 添加 `credentials:'same-origin'` 和 Authorization header；3) CORS 收窄为特定 origin。三步缺一不可。
- **data_flow_context**: 用户操作 → api.js get/post → fetch(无 credentials) → 后端（零认证 CORS *）→ 数据读写。任何同子网第三方页面可跨域调用全部端点。

### W-003 · CDN 资源无 SRI + development 版本
- **file**: web/index.html
- **line**: 8-12
- **severity**: high
- **kind**: structural
- **finding**: 从 unpkg.com/cdnjs.cloudflare.com 加载 React 18.3.1（development 版本）、Babel Standalone 7.26.0、Tailwind CDN、Font Awesome 6.4.0，均无 Subresource Integrity（SRI）hash。CDN 被劫持或投毒时可注入任意 JS。且使用 react.development.js 而非 production.min.js，暴露额外调试信息。
- **remediation**: 1) 给所有 CDN `<script>`/`<link>` 加 `integrity='sha384-...'` 属性；2) 切换到 react.production.min.js；3) 考虑 self-host 这些依赖以消除外部依赖风险。
- **data_flow_context**: 浏览器 → 请求 unpkg.com/cdnjs → 若 CDN 返回被篡改的 JS → 在页面上下文完全执行 → 可读取所有 DOM 数据和 api key。

### W-004 · 无 Content-Security-Policy
- **file**: web/index.html
- **line**: 1-190
- **severity**: high
- **kind**: structural
- **finding**: 无 Content-Security-Policy（CSP）meta 标签或 HTTP header。内联脚本（line 106-108, 132-188）与外部 CDN 脚本并存，无 nonce/hash 约束。配合 CORS `*` 后端，XSS payload 一旦注入将无任何策略阻挡。
- **remediation**: 添加 `<meta http-equiv='Content-Security-Policy' content="default-src 'self'; script-src 'self' https://unpkg.com https://cdn.tailwindcss.com 'nonce-{random}'; style-src 'self' 'unsafe-inline' https://cdnjs.cloudflare.com; connect-src 'self'">` 并给内联 script 加 nonce。
- **data_flow_context**: 无 CSP → 任何注入的 `<script>` 可执行 → 窃取 api key、调用后端破坏性端点、读取 DOM 中的 PII。

### W-006 · InboxPage 文件上传路径遍历（前后端联动漏洞）
- **file**: web/InboxPage.jsx
- **line**: 18-19
- **severity**: high
- **kind**: route-fix
- **finding**: 文件上传端点 `/inbox/upload` 在前端使用 `f.text()` 直接读取文件内容作为 request body，filename 通过 query param 传递。后端 api.py:279 直接 `inbox / filename` 写入磁盘，无路径遍历过滤。攻击者可构造 `filename='../../etc/cron.d/evil'` 实现任意文件写入。前端虽用 encodeURIComponent，但后端 Python 解码后 `../..` 仍生效。
- **remediation**: 后端必须对 filename 做 `Path(filename).name` 取纯文件名，拒绝含 `/` 或 `..` 的输入。前端额外校验文件名不含路径分隔符。
- **data_flow_context**: 用户选择文件 → f.name → encodeURIComponent → query param → 后端 Query(filename) → urllib 解码 → inbox / filename → Path.write_bytes → 路径遍历可写任意位置。

### W-008 · API key 明文 HTTP POST 传输
- **file**: web/SettingsPage.jsx
- **line**: 99
- **severity**: high
- **kind**: route-fix
- **finding**: API key 明文通过 HTTP POST body 发送到 `/settings/llm` 端点（body.api_key = apiKey）。后端零认证+CORS `*`，任何同网络攻击者可嗅探或主动 POST 修改 api key。虽然 line 105 清除了 React state 中的明文，但传输过程无 HTTPS 强制保证（取决于部署）。
- **remediation**: 1) 强制 HTTPS（HSTS header）；2) 后端加认证保护 `/settings/llm` 端点；3) 考虑将 key 设置移出 web 界面，仅在服务端 env/文件配置。
- **data_flow_context**: 用户输入 api key → React state → JSON body → fetch POST /settings/llm（无认证无加密保证）→ 后端写入 config → 明文存储（已知 H-001）。

## Medium Findings

### W-002 · fetch 错误静默吞掉
- **file**: web/api.js
- **line**: 9-27
- **severity**: medium
- **kind**: route-fix
- **finding**: fetch 错误被静默吞掉（catch(e) { return null }），HTTP 错误状态码（401/403/500）只返回 null，无日志无用户提示。安全事件（如认证失败、权限拒绝）完全不可观测。
- **remediation**: 至少对非 2xx 响应记录 status code 到结构化日志或 UI 错误提示；对 401/403 特别处理（提示重新登录或权限不足）。
- **data_flow_context**: fetch → r.ok 为 false → return null → 调用方收到 null 静默降级为 mock 数据，用户无法区分真实数据和 mock 数据。

### W-005 · Babel Standalone 运行时编译
- **file**: web/index.html
- **line**: 132-188
- **severity**: medium
- **kind**: route-fix
- **finding**: App 组件通过内联 `<script type='text/babel'>` 定义（Babel Standalone 运行时编译），而非预编译 bundle。生产环境中 Babel Standalone 体积 ~3MB，增加攻击面和加载时间，且 Babel 本身可能被利用执行任意代码转换。
- **remediation**: 改为构建时预编译 JSX（Vite/esbuild），消除运行时 Babel 依赖。package.json 中已有 build 脚本但未使用 JSX 编译。
- **data_flow_context**: 浏览器加载 babel.min.js → 运行时编译所有 text/babel 脚本 → 若 babel 被篡改可改写任何组件输出。

### W-007 · InboxPage upload 绕过 api.js 包装
- **file**: web/InboxPage.jsx
- **line**: 15-22
- **severity**: medium
- **kind**: route-fix
- **finding**: doUpload 中的 fetch 调用绕过 api.js 的 PA.post 包装，直接使用原始 fetch，不带 Content-Type 和 Accept header，也不检查响应状态。上传失败（如 500）被静默忽略，doIngest 仍会继续执行。
- **remediation**: 统一使用 PA.post 或至少检查 response.ok；添加错误提示 UI。
- **data_flow_context**: 文件上传 → fetch(直接) → 响应被忽略 → 无论成功失败都调用 doIngest → 用户无反馈。

### W-009 · API key 掩码回显对短 key 无效
- **file**: web/SettingsPage.jsx
- **line**: 82
- **severity**: medium
- **kind**: route-fix
- **finding**: displayKey 变量在用户输入 apiKey 时暴露前4位和后4位（apiKey.slice(0,4)+'...'+apiKey.slice(-4)）。对于短 key（如 8 字符以下）几乎等于全部明文暴露在 DOM 中。
- **remediation**: 仅在 key 长度 >= 12 时显示首尾各 4 位；短 key 仅显示 '****已设置'。
- **data_flow_context**: 用户输入 api key → apiKey state → slice(0,4)+slice(-4) → 渲染到 DOM Row 组件 → 任何能看到屏幕或读取 DOM 的攻击者可获取 key 片段。

### W-011 · window.PA_BASE 可被外部脚本篡改（direction-shift）
- **file**: web/api.js
- **line**: 6
- **severity**: medium
- **kind**: direction-shift
- **finding**: window.PA_BASE 可被任意脚本设置（如浏览器扩展或注入的 XSS payload），若被篡改为恶意 URL，所有 API 请求将被重定向到攻击者控制的服务器（SSRF/数据外泄）。api.js 在 IIFE 内读取一次 BASE 后不再验证。
- **remediation**: 在 IIFE 内固化 BASE 后 freeze 或至少校验其 URL scheme/hostname 在白名单内。配合 CSP connect-src 限制出站请求目标。
- **data_flow_context**: window.PA_BASE（可被外部脚本篡改）→ BASE 变量 → 所有 fetch(BASE + path) → 请求发送到攻击者服务器 → api key/用户数据泄露。

### W-010 · base_url 硬编码暴露 LLM provider
- **file**: web/SettingsPage.jsx
- **line**: 15
- **severity**: low
- **kind**: route-fix
- **finding**: base_url 默认硬编码为 'https://open.bigmodel.cn/api/paas/v4'，暴露了使用的 LLM provider 端点。虽为公开信息，但结合 api key 泄露可直接被利用。

### W-012 · bootstrap() 页面加载触发写操作
- **file**: web/api.js
- **line**: 98
- **severity**: low
- **kind**: route-fix
- **finding**: bootstrap() 在 line 98 调用 post('/verify', {}) 做初始化验证，这是一个可能触发数据删除的写操作（后端 run_all 会 delete 不落地数据）。页面加载即触发写操作不符合最小权限原则。
- **remediation**: bootstrap 应仅执行读操作（GET /verify/report）；写操作（POST /verify）应仅在用户点击按钮时触发。

### W-013 · MockData 含半真实 PII
- **file**: web/MockData.jsx
- **line**: 3-11
- **severity**: low

## Info Findings

### W-014 · 无 X-Frame-Options（clickjacking）
- **file**: web/index.html
- **line**: 103
- **severity**: info
- **kind**: none
- **finding**: index.html 无 X-Frame-Options 或 frame-ancestors CSP 指令，页面可被嵌入任意 iframe 实现 clickjacking 攻击。配合零认证后端，攻击者可诱导用户在 iframe 中点击按钮触发破坏性操作。
- **remediation**: 后端 StaticFiles 中间件添加 X-Frame-Options: DENY 响应头；或在 CSP 中加 frame-ancestors 'none'。

### W-015 · package.json 无依赖声明
- **file**: web/package.json
- **line**: 1-9
- **severity**: info
- **kind**: none
- **finding**: package.json 无任何 dependencies/devDependencies 声明，所有依赖通过 CDN `<script>` 标签加载。无法通过 npm audit 检测已知漏洞，版本锁定完全依赖 URL 中的版本号。
- **remediation**: 将依赖纳入 package.json 并通过 npm/yarn 管理，启用 dependabot 或 npm audit CI。

### W-016 · ChatPage 安全承诺与实际行为不一致
- **file**: web/ChatPage.jsx
- **line**: 87
- **severity**: info
- **kind**: none
- **finding**: UI 中声称 'key 永不出现在前端；原始数据不持久化'，但 SettingsPage.jsx 实际上允许用户在前端输入 api key 并通过 POST 发送。此声明具有误导性。
- **remediation**: 要么移除前端 key 输入功能使声明为真，要么修改声明准确描述实际行为。

## Positive Observations

1. 所有 JSX 组件使用 React 的 `{}` 插值渲染用户内容，无 `dangerouslySetInnerHTML` 或 `innerHTML` 使用，XSS 通过 React 自动转义得到基本防护。
2. SettingsPage 对 api key 做了掩码回显（displayKey），且在保存成功后清除 React state 中的明文（line 105 setApiKey('')），避免 key 在内存中长期驻留。
3. InboxPage 文件上传使用 encodeURIComponent 编码文件名，虽然无法防御后端路径遍历，但至少避免了 URL 注入。
4. sessionStorage 的使用范围仅限于 segment/memory 高亮跳转（hl_seg/hl_mem），不存储敏感数据。
5. MockData 中的数据虽包含半真实 PII，但不含真实 API key、密码或 token 等凭证。
6. ChatPage 用户消息通过 JSON body 发送（PA.post），非 URL 参数，降低了消息内容被记录到服务器访问日志的风险。
