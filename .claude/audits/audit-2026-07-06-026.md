## Project Summary

Personal-assistant control panel UI. Single-file React bundle (30 JSX components inlined in index.html via browser Babel). Node.js HTTP dev-server serves static files + proxies `/api/health` to personal-assistant backend. esbuild-free build script validates bundle integrity (component count + reference resolution). Smoke test suite with 30+ tests covering path traversal, method guards, MIME dispatch, graceful shutdown, port priority, etc.

Attack surface is narrow (single HTTP server, no auth, no write paths). Defense-in-depth is unusually strong for a dev server — nosniff headers, method guards, protocol-aware proxy, query stripping, path normalization, graceful shutdown. Most findings are robustness/DoS-class rather than RCE/data-leak.

## Findings

### H-001: Upstream proxy response unbounded accumulation (OOM DoS)

- **file**: dev-server.js
- **line**: 105-107
- **severity**: high
- **kind**: route-fix
- **finding**: `/api/health` reverse proxy accumulates upstream response body in `let b = ''; up.on('data', (c) => { b += c; })` with no size cap. `upstreamReq.setTimeout(2000, ...)` bounds time but not bytes — a malicious or misbehaving upstream can stream >100MB within 2s and exhaust the dev-server process heap before timeout fires. Default ONEDAY_API_BASE points to local personal-assistant (systemd 127.0.0.1:8000); if that backend is replaced, compromised, or returns unexpectedly large payloads (e.g., debug dump endpoint accidentally exposed at /healthz), the dev-server crashes with OOM, taking down the UI + all three health probes.
- **remediation**: Add `const MAX_UPSTREAM_BYTES = 64 * 1024;` before handler. In `up.on('data')`, check `if (b.length > MAX_UPSTREAM_BYTES) { upstreamReq.destroy(new Error('upstream response too large')); return; }`. Fail handler already exists (`fail(reason)`) and responds 502 + JSON diagnostic — reuse it. Add regression test asserting /api/health returns 502 when upstream streams >MAX bytes.

### M-001: ONEDAY_DIR arbitrary directory read (trusted-env assumption)

- **file**: dev-server.js
- **line**: 20
- **severity**: medium
- **kind**: route-fix
- **finding**: `const dir = process.env.ONEDAY_DIR || __dirname;` trusts ONEDAY_DIR env verbatim. Path traversal defense (normalize+join) prevents escaping `dir`, but if ONEDAY_DIR itself points outside the project (e.g., `/etc`, `/home/admin/.ssh`), every file under that directory becomes readable via GET. Test harness uses this intentionally (negative tests inject temp dirs), but production PaaS could inherit shell env with stale/malicious ONEDAY_DIR from prior job. No validation that ONEDAY_DIR is absolute, inside an allowlist, or even exists as a directory.
- **remediation**: At startup, if ONEDAY_DIR is set: (a) resolve to absolute via `path.resolve`, (b) verify it's a directory (`fs.statSync(dir).isDirectory()`), (c) optionally constrain to known roots (project dir parent, /tmp for tests). Log warning if unconstrained. This is defense-in-depth — not a vulnerability in trusted deployment, but a hardening gap.

### M-002: `/api/*` namespace collides with static file routing

- **file**: dev-server.js
- **line**: 86, 121
- **severity**: medium
- **kind**: route-fix
- **finding**: Only `/api/health` is handled by the proxy branch. Any other `/api/xxx` path falls through to static file serving at line 121: `path.join(dir, safePath)`. If a file named `api/foo.json` exists in the served directory, GET /api/foo.json returns it as static content with 200 — indistinguishable from a real API response. Future developers adding `/api/memories` or `/api/config` endpoints may forget to add a proxy branch and silently serve stale static files instead of proxying to backend. The semantic intent ("API routes go to backend") is implicit, not enforced.
- **remediation**: Add explicit catch-all for `/api/*` prefix: if `safePath.startsWith('/api/') && safePath !== '/api/health'`, return 404 with JSON `{status:'not found', reason:'no proxy route registered'}` (or 501 Not Implemented). This makes the API namespace explicit and fails loudly when a new API route is added without a corresponding proxy handler. Add regression test.

### L-001: MIME table missing common web asset types

- **file**: dev-server.js
- **line**: 27-36
- **severity**: low
- **kind**: route-fix
- **finding**: MIME map covers .html/.js/.jsx/.css/.json/.png/.svg/.ico (8 types). Missing: .woff/.woff2 (web fonts), .map (source maps), .txt, .xml, .mp4/.webm (video), .wasm. Unknown extensions fall back to application/octet-stream (safe, tested by smoke ⑳), but browsers may refuse to load fonts/videos/wasm served as octet-stream due to strict MIME checking, causing silent functional regressions when these assets are added.
- **remediation**: Add entries for .woff/.woff2/.map/.txt/.xml/.wasm to MIME table. Low urgency — current bundle is self-contained HTML with no external font/video assets.

### I-001: Build script component-count regex fragile to syntax drift

- **file**: build.js
- **line**: 45
- **severity**: info
- **kind**: none (informational)
- **finding**: Component counting regex `(?:function\s+[A-Z]...|const\s+[A-Z]...\s*=\s*(?:function|\(|React\.memo)|class\s+[A-Z]...)` covers current patterns but will miss future idioms: `const X = forwardRef(...)`, `const X = lazy(...)`, `export default function X()`. Reference-resolution regex (defRe line 60) is broader and would still find these definitions, so build wouldn't exit 1 on unresolved refs — but reported count would silently undercount. Smoke test ⑧ catches count drift against docs, providing secondary defense.
- **remediation**: Informational only. When new component patterns are introduced, update both regexes together. Consider switching to AST-based counting (acorn/babel parser) for long-term robustness, but current regex + smoke ⑧ + ref-res check provide adequate triple-redundant coverage for MVP.
