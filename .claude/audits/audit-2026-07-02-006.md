## 摘要

prod-deploy 是自治引擎触达生产环境的**最关键外部副作用通道**：变更单创建、流水线触发、分批部署、发布准入均经此 skill 执行。本审计覆盖 17 个文件（11 个 JS 脚本 + 6 个 phase 文档 + SKILL.md），聚焦正确性、安全性（B 节敏感路径埋点）、健壮性、可观测性、与引擎契约一致性五个维度。

**整体风险评级：HIGH**

- 功能正确性与安全隔离（execFile 防注入、URL encodeURIComponent）做得不错
- **三个 high 级问题需紧急修复**：HTTP 客户端全无超时、零 audit-log 埋点、Sunfire observation 退化静默放行
- 断点恢复逻辑（Phase 0）有 design-level 缺陷，属 direction-shift

## Findings

### High Severity

#### H-001 · HTTP 客户端无超时控制（task-client / event-client）
- **file**: skills/prod-deploy/scripts/lib/task-client.js
- **line**: 30
- **severity**: high
- **kind**: route-fix
- **finding**: task-client.js 和 event-client.js 的 fetch() 调用均无超时控制（无 AbortController/signal），HTTP 请求可无限挂起导致部署流程永久阻塞
- **remediation**: 为 apiGet/apiPost/apiPut 统一加 AbortController + setTimeout(30000)，fetch 选项中传 signal。event-client.js 同理。两个文件的三个函数模式相同，可抽公共 helper。
- **evidence**: task-client.js L30-43 apiGet()、L46-57 apiPost()、L60-71 apiPut() 均为裸 fetch() 无 signal/timeout；event-client.js L32-44 apiPost()、L46-57 apiPut() 同样无 timeout

#### H-002 · sunfire-client httpPost 无请求/socket 超时
- **file**: skills/prod-deploy/scripts/lib/sunfire-client.js
- **line**: 32
- **severity**: high
- **kind**: route-fix
- **finding**: sunfire-client.js 的 httpPost() 使用原生 https.request 但无请求超时和 socket 超时，若 Sunfire API 无响应则 observation 轮询永久挂起，阻塞整个部署批次推进
- **remediation**: 在 https.request 后加 req.setTimeout(30000, () => { req.destroy(); reject(new Error('Sunfire request timeout')); })；或给 socket 加 'timeout' 事件处理。
- **evidence**: sunfire-client.js L32-54 httpPost() 函数：只有 req.on('error', reject) 和 req.end()，无 setTimeout/socket timeout 设置

#### H-003 · 整个 prod-deploy 模块零 audit-log 埋点
- **file**: skills/prod-deploy/scripts/**
- **line**: 0
- **severity**: high
- **kind**: route-fix
- **finding**: 整个 prod-deploy 脚本集无任何 audit-log 埋点。部署/变更单/流水线触发属于 autonomous-constraints.md B 节强制埋点的敏感路径（deploy、release、CR、流水线触发），但所有脚本均未写 audit JSONL
- **remediation**: 在 complete-task.js（任务完成）、report-event.js 的 build/deploy_batch handler（流水线触发、批次部署）中，添加 audit-log 写入（按 audit-log.schema.json 格式追加 .audit/audit-YYYY-MM-DD.jsonl）。最低限度：complete-task.js 写一条 action=deploy-complete，report-event.js 的 build 首次创建时写 action=pipeline-triggered，deploy_batch 创建时写 action=batch-deploy-started。
- **evidence**: grep 'audit|log\.append|jsonl|\.audit' skills/prod-deploy/ 返回 0 结果；autonomous-constraints.md B 节明确列出 '部署/变更单（deploy、release、CR、流水线触发）' 为强制埋点路径

### Medium Severity

#### M-001 · deserializeEvent JSON.parse 未 try/catch
- **file**: skills/prod-deploy/scripts/lib/event-client.js
- **line**: 63
- **severity**: medium
- **kind**: route-fix
- **finding**: deserializeEvent() 对 payload 执行 JSON.parse(raw.payload) 未做 try/catch，若服务端返回畸形 payload 字符串将抛未捕获异常导致整个脚本崩溃
- **remediation**: 在 deserializeEvent L68 加 try/catch：payload 解析失败时返回空对象 {} 并 console.error 记录，不中断流程。
- **evidence**: event-client.js L63-69: function deserializeEvent(raw) { return { ...raw, payload: raw.payload ? JSON.parse(raw.payload) : {} }; } — 无 try/catch 包裹 JSON.parse

#### M-002 · drillDownToPreCheckTask 串行遍历 + 缓存持久化时机晚
- **file**: skills/prod-deploy/scripts/poll-pre-check.js
- **line**: 129
- **severity**: medium
- **kind**: route-fix
- **finding**: drillDownToPreCheckTask() 串行遍历所有 jobId 调用 a1 CLI，若 job 数量多（>5）则 drill-down 耗时可能超过 Claude 调用间隔（5s），且每次 poll-pre-check 调用都可能重新 drill-down（仅当 event payload 中缓存了 task_id 才跳过）。首次 drill-down 成功后，如果 updateEvent 写入 payload.task_id 失败（网络抖动），下次仍会重新 drill-down
- **remediation**: 在 drill-down 成功后立即单独调 updateEvent 写入 task_id 到 payload，确保缓存持久化，不依赖后续的整体 updateEvent 调用。同时可并行请求 job 的 task list 而非串行。
- **evidence**: poll-pre-check.js L129-156: for (const jobId of allJobIds) { ... await runA1(...) ... } — 串行循环；L166-169: cachedTaskId 仅在函数返回后由 L213 的 updateEvent 间接写入 payload

#### M-003 · Phase 0 断点恢复只查 RUNNING 状态，漏掉 SUCCESS 导致重复执行 ⚠️ direction-shift
- **file**: skills/prod-deploy/phases/00-checkpoint-recovery.md
- **line**: 9
- **severity**: medium
- **kind**: direction-shift
- **finding**: 断点恢复（Phase 0）只查 RUNNING 状态事件来决定恢复点，但 deploy_plan 事件无 RUNNING 状态（直接创建时可能传 null initialStatus，默认 RUNNING）。若中断发生在 Phase 6（提交发布单）后、Phase 7（分批部署）前，deploy_plan 已 SUCCESS 但 deploy_batch 尚未创建，此时无 RUNNING 事件，恢复逻辑会从 Phase 1 重新开始，导致重复创建 CR / 重复触发流水线
- **remediation**: Phase 0 恢复逻辑应同时查 SUCCESS 状态事件，建立完整的已完成阶段列表，从最后已完成阶段的下一步恢复。需要修改 00-checkpoint-recovery.md 中的恢复策略，以及可能需要在 list-events.js 中支持查全量事件。**此为 direction-shift：触及 phase 协议契约**。
- **evidence**: 00-checkpoint-recovery.md L6-8: '如果有 status=RUNNING 的事件...根据最后事件的 event_type 决定从哪一步继续'；但若所有事件均已 SUCCESS 且后续阶段未开始，无 RUNNING 事件可匹配，回退到 '没有 RUNNING 事件，从 Phase 1 开始'

#### M-004 · Sunfire observation skipped 时 conclusion 标 'passed' 静默退化
- **file**: skills/prod-deploy/scripts/report-observation.js
- **line**: 64
- **severity**: medium
- **kind**: route-fix
- **finding**: 当 Sunfire 凭证未配置或 API 出错时，report-observation.js 将 conclusion 设为 'passed'（L76），结合 resume-next-batch.js 的逻辑（观测数据 skipped 时不阻断），意味着无 Sunfire 配置时观察期门控完全退化为纯时间门控，HIGH 级别生产异常将被静默放过
- **remediation**: 至少在 report-observation.js 输出中将 skipped=true 的 conclusion 标记为 'skipped' 而非 'passed'，并在 SKILL.md 中明确文档化这一退化行为。考虑在 resume-next-batch.js 中对 skipped 观测加告警日志而非静默放行。
- **evidence**: report-observation.js L65-78: if (result.skipped) { ... console.log(JSON.stringify({ conclusion: 'passed', skipped: true })); return; }；resume-next-batch.js L67: if (observationResult && !observationResult.skipped) — skipped 时跳过观测门控检查

- **severity**: low

#### L-003 · report-event.js 多处 JSON.parse 未 try/catch
- **file**: skills/prod-deploy/scripts/report-event.js
- **line**: 114
- **severity**: low
