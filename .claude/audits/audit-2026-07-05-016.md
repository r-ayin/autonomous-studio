## 摘要

`x-tool/wanxia` 的数据层（参数化 SQL、WAL 持久化、.env/.db/.posts 已正确 gitignore）卫生状况**良好**——这是相对 x-tool 其他子项目（pdd监测数据泄露、api监测明文 key 入库）的显著进步。但 **HTTP 边界**与**部署路径**存在两类真问题：

1. **HTTP server 全量未鉴权**：所有路由（含 `POST /api/social/fetch`、`POST /api/social/calibrate` 这类驱动**凭证-bearing 外部自动化**的写端点，以及 `GET /api/social/status` 这种**披露 cookie 是否配置**的端点）均无任何鉴权；`app.listen(PORT)` 默认 bind `0.0.0.0`。任一可达网络的访客可触发受害者 Weibo/小红书 cookie 的大规模自动抓取（→ 账号被风控/封禁），并可探测哪些账号凭证已配置。
2. **health-check.py 硬编码 WSL 路径** `/mnt/e/x-tool/wanxia`：在 Linux 原生部署下 `ensure_deployed()` → `shutil.copytree` 必抛 FileNotFoundError → restart 路径永久失效。与 audit-007/008 的 firmware 硬编码 `E:/x-tool/...` 同类病。
3. **外部气象 fetch 无超时** + **缓存 loading 锁的去重缺漏** → open-meteo 挂起时请求堆积 + stampede。

数据层、预测算法（`prediction-engine.js` 的 null-guard / weightsSnapshot race 保护 / clamp）、客户端渲染（用 textContent、city 名来自静态 cities.js）均无注入/XSS 风险。

## Findings

### WX-AUTH-01 [HIGH] 全量路由未鉴权 + 默认 bind 0.0.0.0 → 凭证-bearing 写端点开放

- **file**: `src/routes.js`（全文件）+ `server.js:95`
- **evidence**: `server.js:95` `app.listen(PORT, () => ...)` 未传 host → Node http 默认 bind `0.0.0.0`（所有网卡）。`routes.js` 中 `router.post('/social/fetch', ...)`(L369)、`router.post('/social/calibrate', ...)`(L399)、`router.get('/social/status', ...)`(L353) 均无任何 auth 中间件。`/social/fetch` 调 `fetchAndStoreSocialData`（`social-scraper.js:315`），后者用 `process.env.WEIBO_COOKIE` / `XIAOHONGSHU_COOKIE` 对 `s.weibo.com` / `m.weibo.cn` / `xiaohongshu.com` 发起大规模自动抓取（34 城 × 每城 Playwright 浏览器导航 + 1.8-3.5s 间隔）。`/social/status` 返回 `{ weibo: bool, xiaohongshu: bool, dual: bool }`（`social-scraper.js:345-349`）——直接披露哪些平台凭证已配置。
- **影响**: 任何能访问 8080 端口的网络访客（同网段 / 公网若暴露）可：(a) 反复触发 `/social/fetch` 使受害者的 Weibo/小红书账号被风控甚至封禁（cookie 仍属受害者）；(b) 通过 `/social/status` 探测凭证配置情况做定向窃取的前置侦察；(c) `/social/calibrate` 触发昂贵 DB 重算。这是把"凭证在我这、动作由我发"的信任边界完全打开。
- **remediation**: 最小 fix 二选一——
  - (a) **bind 127.0.0.1**（最小、零依赖）：`server.js:95` 改 `app.listen(PORT, '127.0.0.1', () => ...)`。若仅本机/反代消费，这一行即闭合。
  - (b) **共享密钥中间件**（若需远程访问）：在 `routes.js` 顶部加 `function auth(req,res,next){ if(req.query.key!==process.env.WANXIA_API_KEY && req.header('x-api-key')!==process.env.WANXIA_API_KEY) return res.status(401).json({error:'unauthorized'}); next() }`，挂在写端点 `router.post('/social/fetch', auth, ...)` / `router.post('/social/calibrate', auth, ...)` / `router.get('/social/status', auth, ...)` 上；`/api/health` 与只读预测端点可放行。
  - 推荐 (a) + 可选 (b)。`process.env.WANXIA_API_KEY` 未设时 (b) 应 fail-closed（拒绝）而非放行。
- **kind**: route-fix（改 `server.js` 1 行或 `routes.js` 加中间件，不触及 `.claude/public-interfaces.txt` 公共接口）

### WX-HEALTH-01 [HIGH] health-check.py 硬编码 WSL 路径 → Linux 部署下 restart 永久失效

- **file**: `health-check.py:6`
- **evidence**: `SRC_DIR = "/mnt/e/x-tool/wanxia"`。`ensure_deployed()`(L16-25) 在 `WORK_DIR=/tmp/wanxia` 不存在时 `shutil.copytree(SRC_DIR, WORK_DIR, ...)`。本仓库实际 Linux 原生部署在 `/home/admin/workspace/x-tool/wanxia`，`/mnt/e/x-tool/wanxia` 不存在 → `shutil.copytree` 抛 `FileNotFoundError` → `restart()`(L27) 异常未捕获 → health-check 进程崩溃，restart 路径死掉，服务挂了拉不起来。
- **影响**: 与 audit-2026-07-02-007（autonomous-engine latest.json 含 `/mnt/e/x-tool`）、audit-2026-07-05-008（`build_engine_firmware` 硬编码 `E:/x-tool/...`）同类"WSL 路径污染"——健康检查/自愈机制本身失效，比一般 bug 更隐蔽（平时 check() 返回 200 时不走 restart 分支，问题潜伏到服务真挂时）。
- **remediation**: `health-check.py:6` 改 `SRC_DIR = os.environ.get('WANXIA_SRC_DIR', os.path.dirname(os.path.abspath(__file__)))`——即默认指向脚本所在目录（Linux 原生即正确），并留 env 覆盖口子。1 行。
- **kind**: route-fix（单文件 1 行，非公共接口）

### WX-ENV-01 [MEDIUM] .env 解析器：boot 崩溃 + 引号值破坏 cookie

- **file**: `server.js:5-13`
- **evidence**: `const envLines = readFileSync(new URL('.env', import.meta.url), 'utf8')`(L5) 无 try/catch——本克隆验证 `.env` 不存在（`ls .env` → no such file），`readFileSync` 抛 ENOENT，server 启动即死，无任何兜底/默认。解析器 L7-12 按首个 `=` 切分后 `trim()` 但**不剥离引号**：`.env` 中写 `WEIBO_COOKIE="SUB=_xxx"`（dotenv 惯例）时，值变成字面量 `"SUB=_xxx"`（含双引号），作为 `Cookie` 头发出 → Weibo 视为无效 cookie → `fetchDesktop` 抛 "Weibo cookie expired"（L151）→ 抓取永远失败，且日志误导成"cookie 过期"而非"引号没剥"。
- **影响**: (a) 缺 .env 时服务无法启动（当前克隆即此状态）；(b) 按 dotenv 惯例写引号值的用户会得到看似有效实则损坏的 cookie，调试误导。
- **remediation**: 最小——L5 包 `try { ... } catch { /* .env optional */ }`；L10 value 剥引号：`v.replace(/^["']|["']$/g, '')`。或直接 `import 'dotenv/config'`（已有依赖生态）。推荐前者零新依赖。
- **kind**: route-fix（单文件 server.js）

### WX-TIMEOUT-01 [MEDIUM] weather/grid fetchJSON 无超时 → 挂起 + 与 WX-CACHE-01 叠加 stampede

- **file**: `src/weather-fetcher.js:6-10`、`src/grid-fetcher.js:59-72`
- **evidence**: `weather-fetcher.js` `async function fetchJSON(url){ const res = await fetch(url); ... }`——**无 AbortController、无 timeout**。`grid-fetcher.js:59` 同样 `await fetch(url)` 裸调（仅 429 重试，无挂起保护）。对照 `social-scraper.js:81-91` 的 `timedFetch` 用 `AbortController` + 9s timeout——同项目内已存在正确范式，weather 层却漏了。`/api/predictions`(routes.js:108) 走 `getPredictions()`→`fetchAllCityData()`→`fetchJSON`；若 open-meteo 挂起（TCP 连上但不返响应），请求永不返回。
- **影响**: 单个挂起请求本身只是慢，但叠加 WX-CACHE-01（loading 锁不释放 + 去重缺漏）会 stampede：多个并发请求都进入 `fetchAllCityData()`，放大成对 open-meteo 的并发压力，且客户端全挂。
- **remediation**: 给两处 `fetchJSON` 加 `AbortController` + 10-15s timeout，复用 `social-scraper.js:81-91` 的 `timedFetch` 模式。2 文件。
- **kind**: route-fix

### WX-CACHE-01 [MEDIUM] 缓存 loading 锁无去重 → wait-then-fallthrough stampede

- **file**: `src/cache.js`（全文件）+ `src/routes.js:47-53`、`141-167`、`169-228`
- **evidence**: `cache.js` 的 `loading` 是 `Set`，`setLoading(key)` 对已在集合中的 key 是 no-op（不记录等待者）。`routes.js:47-53` 的 wait 路径：若 `cache.isLoading(cacheKey)` 且无 cached 数据，先 `await sleep(2000)`，再 `cache.get(cacheKey)`→null，**随后 fall through 到 `cache.setLoading(cacheKey)`（no-op）+ `await fetchAllCityData()`**——即等待者超时后自己也发起一个并发 fetch，而不是复用 in-flight 那个。N 个并发请求 → N 个并发 fetch。成功路径 `cache.set`(L67) 调 `cache.js:18` `this.loading.delete(key)` 是对的；错误路径 `clearLoading`(L75) 也对；问题只在"无 cached + loading 中"的 fallthrough。
- **影响**: 冷启动或缓存过期时若多个客户端同时打 `/api/predictions`（前端 5min 轮询 + contour stream 也会调 `getPredictions()`），会并发打 open-meteo，叠 WX-TIMEOUT-01 加剧挂起。
- **remediation**: `cache.js` 改用 `Map<key, Promise>` 记录 in-flight promise：`getOrLoad(key, loader)` 返回已存在的 promise 或调 loader 新建。`routes.js` 把 `setLoading/isLoading/clearLoading` 三段式重构为 `cache.getOrLoad(cacheKey, async () => {...})`。属于模式性小重构但聚焦在 2 文件内，不算跨模块 structural。
- **kind**: route-fix（cache.js + routes.js；虽是模式调整但单位小、边界清晰，不进 structural-debt）

### WX-SILENT-01 [LOW] 空 catch 静默吞错 → 故障不可观测

- **file**: `src/routes.js:194`、`src/routes.js:275`、`src/social-scraper.js:180`、`src/social-scraper.js:242`
- **evidence**: `routes.js:194` `try { const predData = await getPredictions(); augmentGridWithCities(...) } catch (e) {}`——`augmentGridWithCities` 失败时 contour 用未增强的 grid 分数，无任何日志。`routes.js:275` contour/stream 中同模式。`social-scraper.js:180` `} catch { /* skip */ }`（mid→engagement 采样失败静默）、`social-scraper.js:242` `} catch {}`（DOM 抽取失败静默）。
- **影响**: 抓取/增强静默失效时，social_score 偏低或 contour 城市分数不增强，运维无信号。非安全问题，是可观测性债务。
- **remediation**: 改 `catch (e) { console.warn('[augmentGrid]', e.message) }` 等，保留语义但留诊断痕迹。
- **kind**: route-fix（routes.js + social-scraper.js，可与 WX-AUTH-01 同 worktree 或独立）

### WX-XHS-REUSE-01 [LOW] 浏览器单例 + closeBrowser 竞态 + --no-sandbox

- **file**: `src/social-scraper.js:24-45`、`social-scraper.js:42-45`
- **evidence**: `_browser`/`_browserContext` 模块级单例。cron 22:00 `finally { await closeBrowser() }`(server.js:52) 关闭 context；若此时 HTTP `/api/social/fetch` 正在用同一 context（`getBrowserContext()` 返回单例），context 被中途关闭 → in-flight `page.goto` 抛"Target closed"。`chromium.launch` 带 `--no-sandbox`(L30)（容器中常见但若以 root 跑则降级 Chromium 沙箱保护）。
- **影响**: cron 与 HTTP 触发并发时偶发抓取失败（低概率，因 cron 22:00 与人工触发窗口通常不重叠）；`--no-sandbox` 在非 root 容器下可接受，root 下需评估。
- **remediation**: (a) `closeBrowser` 加引用计数或在用时不关；或 (b) HTTP `/social/fetch` 用独立 context（不复用单例）。`--no-sandbox` 若确认非 root 运行可保留并加注释。低优先级。
- **kind**: route-fix（social-scraper.js，模式小调整）
