## Detailed Findings

### H-001 — `_rep()` 合成 rep 绕过 scan_project 集成路径

**File**: tests/test_scout_scan.py:26-35
**Severity**: high
**Kind**: route-fix

`_rep()` 构造的 dict 仅含 `name/progress_stale_days/gates_exists/has_planning/markers/file_count_indexed` 6 个键。`scan_project()` 实际返回的 rep 还包含 `progress_pending_wts/gates_pending_wts/triage_pending_wts/planning_pending_wts/pending_worktrees/markers_deferred/marker_files/git/symbols` 等字段。当前 `health_priority()` 用 `.get(..., [])` 兜底所以测试能跑通，但这意味着：

1. 测试只验证 `health_priority` 单元行为，**不验证 `scan_project → health_priority` 集成路径上的字段映射正确性**（例如 `count_markers` 三元组返回值是否正确解包到 rep、`pending_triage_in_worktrees` 的 mfiles 参数传递是否正确）。
2. 未来若 `health_priority` 改成必访问某新字段（不再用 `.get`），测试 KeyError 但报错信息指向断言行而非字段缺失，误导诊断。
3. 合成 rep 无法触发 `scan_project` 内部的副作用（写 .codebase-index/ JSON），该副作用的正确性完全无测试守护。

**Remediation**: 增加至少一个集成测试：在 tmpdir 里造最小项目结构（含 PROGRESS.md/GATES.md/planning/ + 几个代码文件带 TODO），跑 `scan_project({"name":"x","path":tmpdir,"has_git":False,"has_studio":False})`，断言返回 rep 含所有预期键且 `health_priority(rep)` 分数合理。保留 `_rep()` 单元测试作为补充而非替代。

### M-001 — 硬编码项目名假设 workspace 拓扑

**File**: tests/test_scout_scan.py:60-71
**Severity**: medium
**Kind**: route-fix

`WORKSPACE = os.path.abspath(os.path.join(HERE, "..", ".."))` 假定 tests/ 永远位于 `<workspace>/tests/` 且 workspace 下必有名为 `autonomous-studio` 的子目录。但 loop 常态以 `/home/admin/workspace/autonomous-studio-aone` 自身为 workspace（见 autonomous-constraints.md step 1），此时 `discover_projects()` fallback 返回根项目名 `autonomous-studio-aone`。实测确认：

```
$ python3 scripts/scout-scan.py --workspace /home/admin/workspace/autonomous-studio-aone --json
projects: ['autonomous-studio-aone']
recs: ['autonomous-studio-aone']
```

测试断言 `"autonomous-studio" in names` 在此拓扑下必 fail。**测试只在 `/home/admin/workspace` 父级目录通过**（那里才有 autonomous-studio 子项目），而 loop 的实际运行拓扑恰是它不通过的拓扑。

**Remediation**: 改为断言 `len(recs) >= 1`（推荐列表非空）+ `all("name" in r and "score" in r for r in recs)`（结构完整）。若要保留"autonomous-studio 不被排除"语义，应改为"当 workspace 含 autonomous-studio 子项目时它必须出现"，前置条件检查而非硬断言。

### M-002 — pytest 未安装，tests/ 从未被执行

**File**: (environment)
**Severity**: medium
**Kind**: structural

`python3 -m pytest` 报 `No module named pytest`，`pip show pytest` 报 not found。tests/ 目录存在但环境无 pytest，测试套件**从未被执行过**。这是 M-001 环境耦合 bug 一直没被发现的根因。

审计视角：tests/ 是"看起来有测试保护但实际零运行"的假安全网。引擎/loop 没有任何机制确保测试定期跑或在提交前跑。opt-worktree.sh commit gate 也不跑 pytest。

**Remediation**: 属结构性债务——需要决定测试策略（装 pytest + CI gate？还是删 tests/ 承认无测试？）。写入 structural-debt.md，不直接派 fix。

### M-003 — test_score_independent_of_commit_recency 断言过弱

**File**: tests/test_scout_scan.py:48-57
**Severity**: medium
**Kind**: route-fix

断言仅验证 `isinstance(hp["score"], float)` 和 `hp["work_unit"]` 非空。若 `health_priority` 回归为恒返回 `{"score": 0.0, "reasons": [], "work_unit": "placeholder"}`（极端退化），测试仍通过。不变量"score 不依赖 last_commit"未被数值验证。

**Remediation**: 增加数值断言：构造两个 rep 仅 `progress_stale_days` 不同（如 0.1 vs 100），断言 score 差异 > 0（stale 贡献确实生效）；或构造已知 score 的 rep 断言精确值（锁定公式）。

### L-001 — test_text_output_has_recommendation_section 仅验字符串存在

**File**: tests/test_scout_scan.py:74-83
**Severity**: low
**Kind**: route-fix

断言 `"推荐工作单位" in r.stdout` 和 `"#1" in r.stdout`。不验证 #1 行的格式（是否含 score、work_unit、reasons）、不验证排序正确性（#1 score >= #2 score）、不验证空 recommendations 时的降级行为。

**Remediation**: 解析 stdout 提取 #N 行，验证格式正则 `\s+#\d+\s+\S+\s+\(score=[\d.]+\)\s+—\s+.+`；或用 --json 模式测结构化输出（已有 test_autonomous_studio_not_excluded_from_ranking 走 json，可复用）。

### L-002 — 缺失 5 类关键边界用例

**File**: tests/test_scout_scan.py (missing tests)
**Severity**: low
**Kind**: structural

scout-scan.py 注释明确记录了多个历史修复点（case-049/327/219/373、marker 自指虚高 7 类剥离），但测试套件无任何回归守护：

2. **pending_triage 精确抵消** (line 547-549)：case-049 修复核心
3. **case_only worktree 降级推荐** (line 582-588)：case-373 链修复核心

这些是 scout-scan.py 最复杂的业务逻辑，也是历史上反复回归的点。无测试 = 下次重构/改动时高概率再坏。

**Remediation**: 属结构性债务——补全这 5 类测试需要设计合成 fixture（tmpdir + git init + 造 worktree），工作量超单轮最小 fix。写入 structural-debt.md。

### I-001 — subprocess 测试无 mock，慢且环境耦合

**File**: tests/test_scout_scan.py:62-83
**Severity**: info
**Kind**: route-fix

两个测试各 spawn 完整 scout-scan.py 进程（timeout=180s）。慢、依赖 filesystem/git 状态、CI 不友好。scout-scan.py 已模块化（health_priority/scan_project/discover_projects 都是纯函数或可隔离函数），可用 importlib 加载 + 直接调用 + io.StringIO 捕获 stdout 替代 subprocess。

**Remediation**: 重构为 importlib 调用；保留 1 个 subprocess smoke test 验证 polyglot shim 即可。
