## 1. 项目画像

pdd监测 是 Windows 桌面端拼多多商家后台数据采集工具。两条采集路径：

1. **Selenium 连接现有 Chrome 调试端口**（文档主路径，`pdd_selenium_connect.py` / `pdd_collector/pdd_selenium_connect.py`，bat 入口 `pdd_automation.bat` 调 `pdd_selenium_connect.py`）。前置：`chrome_debug_setup.py` + `start_chrome_debug.bat` 用 `--remote-debugging-port=9222 --user-data-dir=C:\temp\chrome_debug` 启动一个独立 profile 的 Chrome，用户在该 Chrome 里登录 PDD 商家后台，采集脚本通过 `debuggerAddress=127.0.0.1:9222` 复用已登录会话。

### PDD-AUTH-01 — `pdd_web_automation.py` 实现账密登录，违反项目"不处理登录/切勿硬编码账密"规则 [MEDIUM]

- **file**: `x-tool/pdd监测/pdd_web_automation.py`
- **line**: 35 (`login_pdd_web(driver, username, password)`)、145 (`main_web_automation(username=None, password=None)`)、156–159 (`if not username or not password: ... password = input("密码: ")`)
- **severity**: medium
- **kind**: route-fix
- **finding**: `CLAUDE.md` 规则 2 明确"依赖已登录会话：不处理登录/验证码，直接复用浏览器现有 cookie，**切勿在代码中硬编码账号密码**"。但 `pdd_web_automation.py` 专门实现账密登录路径：`login_pdd_web` 把 `username`/`password` 用 `send_keys` 直接敲进页面字段，`main_web_automation` 用 `input("密码: ")` 明文回显读取密码（Python `input` 不掩码）。这条路径与项目设计的"复用已登录会话"原则冲突，且明文密码流经内存/终端历史。虽非硬编码，但建立了"代码可处理账密"的先例，易被后续改成硬编码。
- **evidence**: `pdd_web_automation.py:35 def login_pdd_web(driver, username, password)`；`:53 password_field.send_keys(password)`；`:159 password = input("密码: ")`。
- **remediation**: 删除 `login_pdd_web` 与 `main_web_automation` 的账密路径，或加显式 `raise RuntimeError("本项目不处理登录，请用 chrome_debug_setup 启动调试 Chrome 后复用会话；账密登录路径已禁用")` 守卫；保留 `collect_web_data` 等纯采集函数。

### PDD-SANDBOX-01 — `pdd_advanced_collector.py` 新启浏览器用 `--no-sandbox` + Default profile，与文档隔离 profile 设计冲突 [MEDIUM]

- **file**: `x-tool/pdd监测/pdd_advanced_collector.py`
- **line**: 37–51 (`launch_new_browser`)
- **severity**: medium
- **kind**: route-fix
- **finding**: 文档主路径要求用 `--user-data-dir=C:\temp\chrome_debug` 独立 profile 启 Chrome 后复用（`chrome_debug_setup.py`、bat、CLAUDE.md 一致）。但 `pdd_advanced_collector.py:launch_new_browser` 在检测不到 9222 调试端口时 fallback `pyppeteer.launch({'args': ['--start-maximized', '--no-sandbox', '--disable-dev-shm-usage', '--disable-blink-features=AutomationControlled', '--disable-extensions', '--profile-directory=Default']})`。问题：(1) `--no-sandbox` 关闭 Chrome 沙箱，安全降级；(2) `--profile-directory=Default` 会附加到**用户真实 Default Chrome profile**，若用户已有 Chrome 在跑会冲突或直接驱动用户真实会话（包括其他网站 cookie），与"独立 debug profile"设计背道驰驰；(3) 该 fallback 静默发生（`connect_to_existing_browser_via_cdp` 的 except 链直接走 `launch_new_browser`），用户以为复用隔离 profile 实则新启 Default profile。
- **evidence**: `pdd_advanced_collector.py:22 http://localhost:9222/json/version` 失败 → `:28 return await launch_new_browser()`；`:44 '--no-sandbox'`；`:48 '--profile-directory=Default'`。
- **remediation**: fallback 也应使用 `--user-data-dir=C:\temp\chrome_debug`（或项目内独立路径）+ 移除 `--no-sandbox`（仅当以 root 跑容器时才需 no-sandbox，Windows 桌面不需要）；若不复用则明确报错让用户先跑 `chrome_debug_setup`。

### PDD-DEBUG-01 — Chrome 9222 调试端口长期暴露已登录 PDD 会话，本地任意进程可 CDP 接管 [MEDIUM, structural]

- **file**: `x-tool/pdd监测/chrome_debug_setup.py`, `start_chrome_debug.bat`, `pdd_automation.bat`
- **line**: `chrome_debug_setup.py:61` (Arguments 写入快捷方式)、`:97` (bat 启动行)、`pdd_advanced_collector.py:22,34` (任意进程可 `http://localhost:9222/json/version` + `ws://localhost:9222/devtools/browser/` 接管)
- **severity**: medium
- **kind**: structural
- **finding**: 设计上 Chrome `--remote-debugging-port=9222` 默认绑 127.0.0.1，且该 Chrome 长期运行、已登录 PDD 商家后台（cookie + session 在 profile 里）。本机任意进程都能 `GET http://localhost:9222/json/version` 拿到 DevTools WebSocket 端点，再 `pyppeteer.connect(browserWSEndpoint='ws://localhost:9222/devtools/browser/')` 完全接管该浏览器：读所有 cookie、读 localStorage、以登录态发请求、改 LLM 后端配置等。这是 Chrome 远程调试的固有性质，但本项目无任何缓解（无 `--remote-debugging-port=0` 随机端口、无 `--remote-allow-origins=*` 收紧、无端口防火墙、无提示用户采集完即关 Chrome）。属设计性风险，单点 fix 拆不成最小单位（要么改启动协议+采集协议+bat 三处，要么接受风险并文档化）。
- **evidence**: `chrome_debug_setup.py:61 $Shortcut.Arguments = '--remote-debugging-port=9222 --user-data-dir="C:\\temp\\chrome_debug"'`；`pdd_advanced_collector.py:22,34` 任意本地进程可连；无端口随机化/收尾关闭。
- **remediation**: 需 direction-shift 级变更：① 启动改 `--remote-debugging-port=0`（随机端口，输出到 stdout 让采集脚本解析）+ `--remote-allow-origins=localhost`；② 采集脚本从随机端口连接而非硬编 9222；③ bat 采集完提示关闭 Chrome。拆不成最小 fix → 入 structural-debt。

### PDD-SWALLOW-01 — 全项目裸 `except:` 静默吞异常（含 KeyboardInterrupt），采集失败不可观测 [LOW, structural]

- **file**: `pdd_selenium_connect.py`, `pdd_collector/pdd_selenium_connect.py`, `pdd_monitor.py`, 几乎所有 `pdd_monitor_*.py` 变体
- **line**: `pdd_selenium_connect.py:66,83,98,113,129,271,277,297,303,322,335,352`（每个 `try` 后裸 `except: continue`）；`pdd_monitor.py:31,185,271,310,344,377,686,720,755,783,859,877,959,967` 等
- **severity**: low
- **kind**: structural
- **finding**: 采集循环里 `try: el.text ... except: continue` 模式遍布所有采集器。裸 `except:` 捕获 `BaseException` 含 `KeyboardInterrupt`/`SystemExit`，用户 Ctrl+C 无法中断死循环采集；同时 selenium `StaleElementReferenceException` 等真错误被静默跳过，采集结果可能大面积缺字段而用户只看到"成功保存"。这与 [[autonomous-engine-operations]] 静默吞错反模式一致。涉及 10+ 文件上百处，单点 fix 无法覆盖。
- **evidence**: grep `except:` 命中遍布全项目；`pdd_selenium_connect.py:66 except: continue`（紧跟 `el.text` 读取）。
- **remediation**: 抽公共 `_safe_get(fn, default=None)` helper（捕获 `Exception` 非 `BaseException`，可选 log 计数），批量替换；或至少改 `except Exception:`。需 direction-shift（引入 helper 模块 + 跨文件替换）→ 入 structural-debt。

### PDD-PSH-01 — `chrome_debug_setup.py` 把路径 f-string 拼进 PowerShell `-Command`，模式不安全 [LOW]

- **file**: `x-tool/pdd监测/chrome_debug_setup.py`（与 `chrome_setup/chrome_debug_setup.py` 重复副本同问题）
- **line**: 57–67 (`powershell_script = f'''... CreateShortcut("{shortcut_path}") ... TargetPath = "{chrome_path}" ...'''` + `subprocess.run(["powershell","-Command",powershell_script], check=True)`)
- **severity**: low
- **kind**: route-fix
- **finding**: `chrome_path` 来自 `get_chrome_path()`：先试 3 个硬编路径（`os.path.exists` 验证），再 fallback `winreg.QueryValue(HKLM, "SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\App Paths\\chrome.exe")`——**注册表值未 `os.path.exists` 验证**。该值被直接 f-string 拼进 PowerShell 脚本的 `TargetPath = "{chrome_path}"`。正常情况安全；但若 App Paths 注册表项被篡改（含 `";Add-Type ...;#` 之类），即 PowerShell 注入。输入非用户直接控制，但属"把数据当代码拼进 shell"反模式，且注册表 fallback 路径未做存在性验证。
- **evidence**: `chrome_debug_setup.py:36-37 winreg.QueryValue` 返回值直接进 `chrome_path`；`:60 $Shortcut.TargetPath = "{chrome_path}"`；`:67 subprocess.run(["powershell","-Command",powershell_script])`。
- **remediation**: 对 `winreg` fallback 返回值加 `if not os.path.exists(returned): return None`；或改用 PowerShell 参数化（`-File` + 脚本文件传参）避免 f-string 拼接；或改 `subprocess` 不走 `-Command` 而走 `[-Command, ...]` 列表已部分缓解（仍是字符串拼）。最小 fix = 加 `os.path.exists` 校验。

## 4. 跨文件 sink 扫描小结

- **无 `eval` / `exec` / `pickle` / `os.system` / `shell=True`** 命中——本项目无 Python 代码注入 sink（`driver.execute_script` 的 f-string 参数 `simulate_mouse_move`/`simulate_page_scroll` 用浮点数插值，值由脚本内部计算，非用户输入，不构成注入）。
- **无 `requests`/`urllib`/`urlopen`**——所有 HTTP 都走 selenium driver.get / pyppeteer page.goto，URL 均硬编 PDD 域名，无 SSRF（无用户可控 URL）。
- **无 `api_key`/`token`/`secret` 硬编码**——`password` 仅出现在 `pdd_web_automation.py`（见 PDD-AUTH-01）。
- **`winreg`** 仅 `chrome_debug_setup.py` 只读 `QueryValue`（App Paths 查 Chrome 路径），无 `SetValue`/`RegCreateKey` 写注册表（bat/快捷方式方案未动注册表，README 提到"修改注册表"但代码未实现，文档与实现不符——记为信息项，非 finding）。
