# Deep Audit: shizi Browser Extension
## audit-2026-07-07-047

**Date**: 2026-07-07
**Project**: shizi (/home/admin/workspace/shizi)
**Scope**: Full project — background, sidepanel, popup, shared, build config
**Depth**: deep (full audit cycle)
**Method**: 4 parallel sonnet sub-agents + adversarial verification (workflow crashed during verify phase; findings below are raw from module auditors, unverified)

## Executive Summary

The shizi browser extension is a functional OCR+vision-model text counter for e-commerce images, but it carries significant security and reliability debt. The most critical issues are: (1) the background message handler accepts unvalidated input from chrome.runtime messages, enabling credential exfiltration via crafted settings injection; (2) multiple O(n²) base64 encoding routines in the service worker can cause OOM on large images; (3) host_permissions grants `<all_urls>` despite no content script or tab access being needed, vastly expanding the attack surface; (4) test coverage includes a false-positive pattern where filter tests duplicate source code instead of importing it, providing zero regression protection. The sidepanel has real state management bugs (drag always moves region[0], draw replaces all regions) that affect multi-region workflows. Build configuration has dual-manifest drift between source and output. Overall code quality is reasonable for a v1.0 extension, but security hardening and test infrastructure fixes should precede any production deployment.

## High Severity Findings

### SZ-BG-H-001: Message handler trusts unvalidated fields from chrome.runtime messages
- **File**: src/background/index.ts
- **Line**: 42, 53-58, 64, 73
- **Kind**: route-fix
- **Description**: handleMessage destructures message.id (deleteHistory), message.ocrResults/message.regions/message.imageSize (refilter), message.imageData/message.region (ocrRegion), and message.settings (saveSettings) directly without any type-checking or validation. A compromised content script or externally-injected message can pass arbitrary shapes. For saveSettings, an attacker could inject a crafted VisionConfig with a malicious baseUrl/apiKey that gets persisted and used for all subsequent API calls (credential exfiltration via controlled endpoint).
- **Evidence**: No runtime typeof checks, no schema validation, no allowlisting of settings keys. Direct destructuring of all incoming message fields.
- **Remediation**: Add runtime validation at the top of handleMessage: check typeof for each expected field, validate imageData is a string starting with 'data:image/', validate settings against DEFAULT_SETTINGS shape (allowlisted keys only), validate region coordinates are numbers in [0,1], validate id is a non-empty string. Reject with error response on mismatch.

### SZ-BG-H-002: blobToBase64 uses O(n²) string concatenation for large binary data
- **File**: src/background/ocr-internal.ts
- **Line**: 161-166
- **Kind**: route-fix
- **Description**: `let str = ''; for (let i = 0; i < bytes.length; i++) str += String.fromCharCode(bytes[i]);` — For a 5MB image blob, this creates ~5M intermediate strings due to JS string immutability. Each `+=` copies the entire accumulated string. This runs on every tile upload (4 tiles × upscale + enhance = ~12 calls per pipeline run). Service workers have limited memory; this can cause OOM or extreme GC pressure on large images.
- **Evidence**: Quadratic string concatenation pattern in hot path called 12+ times per pipeline execution.
- **Remediation**: Replace with chunked approach: accumulate in arrays of ~8192 chars, then join. Or use FileReader.readAsDataURL or a dedicated base64 encoder.

### SZ-BG-H-003: Tile ownership quadrant check drops valid results near tile boundaries
- **File**: src/background/ocr-internal.ts
- **Line**: 253-271
- **Kind**: route-fix
- **Description**: Lines 268-271 compare result center against scaled midpoint to determine tile ownership. Tiles overlap by 30%. Text blocks whose center falls in the overlap zone get assigned to exactly one quadrant, but text spanning across midX/midY with center just past the boundary gets dropped from the owning tile AND incorrectly claimed by adjacent tile. Causes silent data loss for texts near the image center.
- **Evidence**: Center-point comparison against global midpoint rather than tile actual bounds. Overlap zone (30%-70% range) produces ambiguous ownership.
- **Remediation**: Use the tile's actual bounds for ownership instead of global midpoint. Assign each result to the tile whose non-overlapping core region contains the result center, or keep results from all tiles and rely solely on NMS dedup.

### SZ-SP-H-001: containerRef assigned to two different DOM elements causing erratic layout
- **File**: src/sidepanel/SidePanelApp.tsx
- **Line**: 272, 281-282
- **Kind**: route-fix
- **Description**: containerRef is assigned to TWO different DOM elements: line 272 (the scrollable tab-content div) and line 281 (the flex layout div inside the analyze tab). React refs can only point to one element — the second assignment overwrites the first. The ResizeObserver observes whichever element containerRef points to at mount time, but after re-renders when imageData becomes non-null, the ref target silently changes. This means containerWidth is driven by the WRONG element depending on render phase.
- **Evidence**: Same ref attached to two JSX elements. ResizeObserver target changes based on render phase.
- **Remediation**: Use two separate refs: scrollContainerRef and layoutContainerRef. Attach the ResizeObserver to the layout container only.

### SZ-SP-H-002: Drag always moves region[0] regardless of which region was clicked
- **File**: src/sidepanel/SidePanelApp.tsx
- **Line**: 664-667, 683-685
- **Kind**: route-fix
- **Description**: In handleMouseMove for drag type 'move', the code maps over regions and replaces using reference equality `reg === regions[0]`. This always moves the FIRST region regardless of which region the user clicked. When multiple product regions exist, dragging any region visually moves region[0]. Same bug for 'resize' mode. handleHandleMouseDown captures the correct region in origRegion but never uses it to identify which array element to update.
- **Evidence**: Reference equality check against `regions[0]` instead of the clicked region index.
- **Remediation**: Store the index of the dragged region in DragState (e.g. `regionIndex`) and use it to update the correct element: `regions?.map((reg, i) => i === dragState.regionIndex ? { ...reg, x: newX, y: newY } : reg)`.

### SZ-PP-H-001: No file size limit on image upload — memory bomb via large file base64 encoding
- **File**: src/popup/ImageUploader.tsx
- **Line**: 11-24
- **Kind**: route-fix
- **Description**: processFile() accepts any File, checks only MIME prefix, then reads entire file as base64 DataURL. A 50MB PNG would produce ~67MB base64 string held in memory. Chrome extension popup has limited heap (~256MB); a crafted or accidental large image can OOM-crash the popup. Base64 inflates data by ~33%, compounding the problem when sent via chrome.runtime.sendMessage.
- **Evidence**: No file.size check before reader.readAsDataURL(file). Only MIME prefix validation.
- **Remediation**: Add size guard: if (file.size > MAX_SIZE) { alert('图片过大'); return; }. Recommend MAX_SIZE = 10MB. Optionally resize/compress via OffscreenCanvas before encoding.

### SZ-SH-H-001: host_permissions grants <all_urls> but extension never uses tab/scripting APIs
- **File**: manifest.json
- **Line**: 7
- **Kind**: route-fix
- **Description**: manifest.json declares `host_permissions: ['<all_urls>']`. Grep of entire src/ finds zero uses of chrome.tabs.*, chrome.scripting.*, or any content script injection. The extension only uses fetch() to specific API endpoints. activeTab permission is also declared but chrome.tabs.query/get is never called. Both are over-granted. MV3 host_permissions grants ability to read/modify any website the user visits.
- **Evidence**: Zero references to chrome.tabs, chrome.scripting, or content scripts in codebase. Only fetch() to hardcoded URLs.
- **Remediation**: Remove host_permissions entirely (fetch from service workers doesn't need it for specific origins unless CORS-blocked). If needed, scope to exact API domains. Remove 'activeTab' if unused. Reduces attack surface and speeds up Chrome Web Store review.

### SZ-SH-H-002: filterByRegion test duplicates source code — tests verify copy, not actual implementation

### SZ-BG-M-001: sendResponse after async may silently fail if sender disconnects
- **File**: src/background/index.ts

- **Kind**: route-fix
