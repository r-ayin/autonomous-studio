## Findings

### 🔴 HIGH

#### OTA-H-001 — HTTP API 全无认证 + 可执行真实改价 + auto_confirm 绕过风险确认
- **file**: `interfaces/api.py` (全文件) + `interfaces/cli.py:113-115` + `orchestrator.py:116-126`
- **kind**: route-fix
- **evidence**:
  - `interfaces/api.py` 所有 endpoint（`/v1/tasks/execute`、`/v1/tasks/{id}/confirm`、`/v1/tasks/preview` …）均无任何 auth dependency（无 token / api-key / 签名校验）。`/v1/tasks/execute` 接收 `auto_confirm` body 字段（`:80`）直接调用 `service.execute_text(..., auto_confirm)`，触发真实改价。
  - `interfaces/cli.py:113` `host = args.host or api_config.get("host", "127.0.0.1")` — 服务端 bind 完全听命于 `config/app.json`，operator 可设 `0.0.0.0` 把改价 API 暴露到全网。注意：`OtaApiClient.from_settings`（hermes_discord_bot.py:308）的 `0.0.0.0→127.0.0.1` 重绑只作用于**客户端**，不影响服务端 bind。
  - `orchestrator.py:124` `if auto_confirm or not record.risk.requires_confirmation: return self.confirm_task(...)` — `auto_confirm=True` 即使 `risk.level==HIGH`（价格越界）也直接执行，绕过人工确认。
- **impact**: 默认 127.0.0.1 下，本机任意进程（或被攻陷的本地服务 SSRF / 恶意依赖）可 POST `{"text":"豪华大床房改价到99999999","auto_confirm":true}` 改任意房型到任意价格；若 operator 把 host 设为非 loopback 则全网可达。改价 API 无任何认证是价格自动化系统的核心安全缺口。
- **remediation**: 三件套拆 3 个 route-fix：(1) `api.py` 加 token auth dependency（`X-OTA-Token` header，token 来自 env `OTA_API_TOKEN`，未设则 401）；(2) `cli.py` bind host allowlist（拒绝 `0.0.0.0`/`::`/非 loopback，强制 127.0.0.1 除非显式 `--allow-public`）；(3) `orchestrator.py` `execute_text/execute_structured` 当 `risk.level==HIGH` 时无视 `auto_confirm`，强制走确认。
- **derived_fixes**: 拆 3 条（OTA-H-001a auth / OTA-H-001b bind allowlist / OTA-H-001c HIGH-risk auto_confirm guard）— 优先级 1/2/3

### 🟡 MEDIUM

#### OTA-M-001 — M11 修复不完整：hermes bridge bot 仍从 config/app.json 读 Discord token
- **file**: `interfaces/hermes_discord_bot.py:663-664`
- **kind**: route-fix
- **evidence**: `run_bot`: `token = os.getenv(token_env_var) or discord_config.get("token")`。对照 `interfaces/discord_bot.py:146`（M11 已改为 `token = os.getenv(token_env_var)` env-only，无 config fallback）。同一安全理由（token 不应落 config/app.json 以免随仓库泄漏）适用于 hermes bridge，但 M11 只补了一个文件。
- **remediation**: 删 `or discord_config.get("token")` fallback，对齐 discord_bot.py；token 缺失时同款 RuntimeError + 释放 lock。
- **priority**: 4

#### OTA-M-002 — 监控恢复告警一次性：第二次及之后的 down→up 转换不再通知
- **file**: `monitor.py:510-513` + `_should_send_event:633-642`
- **kind**: route-fix
- **evidence**: `_record_service_health` 恢复分支 `_send_alert(f"service:{name}:recovered", ..., repeat=False)`。`_should_send_event(key, repeat=False)`：event 存在即返 False。首次恢复 mark `recovered` event 已发送 → **永不清理** → 第二次 down→up 时 `_should_send_event("service:X:recovered", repeat=False)` 仍命中已存在 event → 返 False → 恢复告警静默。operator 仅被通知第一次恢复。
- **remediation**: 服务 down 转换时清理 `recovered` event（`self.state["events"].pop(f"service:{name}:recovered", None)`），使下次恢复能重新触发；或恢复告警改 `repeat=True` 配合 `repeat_alert_minutes` 节流。
- **priority**: 5

#### OTA-M-003 — monitor state 非原子写：崩溃即丢全部告警去重状态
- **file**: `monitor_state.py:39-40` (`_save_json`)
- **kind**: route-fix
- **evidence**: `_save_json` 直接 `path.write_text(json.dumps(...))`。`save_state`/`save_routes`/`write_review`/`write_last_run` 全走它。进程在 `save_state` 中途被 kill（OOM / SIGKILL / 磁盘满）→ `state.json` 截断 → `load_state` 的 `_load_json` 捕 JSONDecodeError 返默认空 dict → **所有 events 去重记忆丢失** → 重复告警风暴或漏告警。同类：autonomous-engine AE-NEW-001/004（audit-005）。
- **remediation**: temp file + `os.replace` 原子替换（`tmp = path.with_suffix(".tmp"); tmp.write_text(...); os.replace(tmp, path)`）。
- **priority**: 6

#### OTA-M-004 — cancel_task 对 RUNNING 任务竞态丢失：用户见"已取消"但改价照常完成
- **file**: `orchestrator.py:143-150`（cancel）+ `152-170`（confirm）
- **kind**: route-fix
- **evidence**: `cancel_task` **不获取 `execution_lock`**，直接 load→set CANCELED→save。若 cancel 发生在 confirm 持锁执行 workflow 期间（confirm 已于 :162 存 RUNNING），cancel 把状态写 CANCELED，但 confirm 的最终 `:169 save` 覆盖回 SUCCEEDED/FAILED。用户在 Discord 点"取消任务"收到"已取消"，价格却已改。`confirm_task` 末尾也未 reload 检查是否被 cancel。
- **remediation**: `cancel_task` 若 `record.status == RUNNING` 则不覆盖（返回当前 record 或置 `cancel_requested` flag）；`confirm_task` 在最终 save 前 reload 检查是否被 cancel，若是则不标 SUCCEEDED（并注明"改价可能已生效，需人工核对"）。最小实现：cancel_task 加 `if record.status == TaskStatus.RUNNING: return record`。
- **priority**: 7

#### OTA-M-005 — 改价后 verify_result.ok 未校验：失败的改价可被报为成功
- **file**: `platforms/meituan.py:70-99`
- **kind**: route-fix
- **evidence**: apply 阶段 `:58 if not apply_result.get("ok"): raise`（已校验），但 verify 阶段 `:70 verify_result = self._run_change_price_script(page, task, mode="read")` 后**只读 afterPrice/beforePrice（:90-92），不校验 `verify_result.get("ok")`**。若 post-save 读脚本失败（返回 `{ok:false,...}` 或无 afterPrice），`success=True`（:96）照常返回，`after_price` 兜底为 beforePrice 或 None。一次失败的改价被记为成功。
- **remediation**: verify 后加 `if not verify_result.get("ok"): raise RuntimeError(verify_result.get("error") or "改价后校验失败")`（落入 :105 except 标 FAILED）；或在 success 判定中纳入 verify ok。
- **priority**: 8

#### OTA-M-006 — auto_confirm 绕过 HIGH 风险确认（OTA-H-001 的 orchestrator 侧）
- **file**: `orchestrator.py:116-138`
- **kind**: route-fix
- **evidence**: `execute_text`/`execute_structured` `:124/:136 if auto_confirm or not record.risk.requires_confirmation: return self.confirm_task(...)`。`risk.evaluate`（risk.py:19-24）对越界价格置 `level=HIGH` 但 `requires_confirmation` 仍随默认 True（除非 blocked_room）。`auto_confirm=True` 直接跳过确认 → HIGH 风险（如 99999999 元）价格被自动执行。即便 API 加了 auth，获授权的调用方仍可借 auto_confirm 绕过价格护栏。
- **remediation**: `execute_text/execute_structured` 开头 `if auto_confirm and record.risk.level == RiskLevel.HIGH: raise ValueError("HIGH 风险任务不允许 auto_confirm，必须人工确认")`。
- **priority**: 9
- **note**: 与 OTA-H-001c 同文件同函数，可 stack 同一 worktree。

### 🟢 LOW

- **evidence**: `DEFAULT_HERMES_ENV_PATH = Path(r"\\wsl$\Ubuntu\root\.hermes\.env")`，`load_hermes_api_key` 在 `os.name=="nt"` 时从此文件读 `API_SERVER_KEY`。WSL 发行版非 Ubuntu 或非 root 用户时 `path.exists()` False → 静默返 None → "Hermes API key not found"。M12 修了 room-cleanup 的硬编码 Windows 路径，本处同类遗留。

#### OTA-L-002 — DiscordNotifier 部分发送后失败：已发 chunk 重发
- **file**: `monitor.py:131-164`（`send`/`_post_message`）
- **kind**: route-fix
- **evidence**: `send` 循环调 `_post_message`，第 N chunk `raise_for_status` 抛 HTTPError 时异常传播到 `_send_alert:665` 被吞、`sent=False`、event 不 mark → 下轮 `_should_send_event` 仍 True → 重发，**已成功发送的第 1 chunk 重复**。
- **remediation**: `send` 改为追踪已发 chunk 数，失败时返回部分成功信号；或 `_send_alert` 对部分发送也 mark event（防重发）。
- **priority**: 11

#### OTA-L-003 — TaskActionView 按钮只捕 RequestException：其他异常裸逃
- **file**: `interfaces/hermes_discord_bot.py:388,404,418`
- **kind**: route-fix
- **evidence**: confirm/cancel/status 按钮 `except requests.RequestException as exc:`。但 `get_task`→`response.json()` 可抛 `ValueError`（非 RequestException），`_send_followup_with_retry` 重试耗尽抛 `RuntimeError`（:357），二者均逃出 except → discord.py 顶层兜底，用户只看到通用错误无干净提示。
- **remediation**: except 加 `Exception` 兜底分支，followup 发友好错误；或把 `RuntimeError`/`ValueError` 纳入。
- **priority**: 12

#### OTA-L-004 — task 记录非原子写：confirm 最终 save 崩溃即丢任务状态
