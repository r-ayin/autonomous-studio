## Finding OD-SEC-01（HIGH）— 静态分发未过滤 dotfile/点目录，泄漏 .git/config 内嵌凭证 + .audit 审计日志

**file**: `dev-server.js`
**line**: 121（`const filePath = path.join(dir, safePath === '/' ? 'index.html' : safePath);`）+ 127（`fs.readFile(filePath, ...)`）

**finding**: 静态文件分发只做 `path.normalize`（折叠 `..`）+ `path.join`（保证落点在 `dir` 内），**无 dotfile/点目录段过滤**。`dir = __dirname`（项目根）下存在 `.git/`、`.audit/`、`.claude/`、`.gitignore`、`.opt-lock` 等敏感文件/目录，全部可被 `GET /.<seg>` 直接读出。这与既有回归锁 ⑯（`test/smoke.test.js:655` 路径穿越）互补但**不同维度**：⑯ 只证「穿越形 URL 不泄露 served 目录**外**文件」（`/../../etc/passwd` → 404），**未覆盖 served 目录内 dotfile**——而 `.git/config` 本就在 served 目录**内**，不经穿越即可直达。

**evidence（实证，非散文）**：起 `PORT=3091 node dev-server.js 3091`，curl 实测：
```
GET /.git/config     → HTTP 200 558B  → [remote "origin"]\n\turl = https://syp02536326:HStZLJue84222jXR1oWW@code.alibaba-inc.com/OneDayAI-RPUKYRO/1BfrYW9R.git
GET /.git/HEAD       → HTTP 200 30B   → ref: refs/heads/onedaybot-dev
GET /.gitignore      → HTTP 200 366B  → （含 .env / *.pem / *.key 模式，反推本仓确有密钥文件命名约定）
GET /.opt-lock       → HTTP 200 0B
GET /.audit/audit-2026-07-01.jsonl → HTTP 200 448B → {"id":"audit-20260701-155420-ccbbf4","timestamp":"...","userId":"autonomous-engine","userRole":"engine","action":"delete","resource":...}
GET /../../etc/passwd → HTTP 404 （穿越仍被拦，baseline 回归确认）
GET /.claude/settings.json → HTTP 404 （该路径不存在，仅 .claude/decisions/ 子目录）
GET /.design/ → HTTP 404 （目录 EISDIR → 404，无目录列举）
```
即任意访客可拉取 `.git/config`（含 code.alibaba-inc.com push 凭证）、`.git/HEAD`（分支名）、`.audit/*.jsonl`（引擎内部 userId/action/resource 审计流）。

**impact**: dev-server 亦为 `npm start` 生产入口（CLAUDE.md ⑬⑭ + package.json `start`），`server.listen(port)`（dev-server.js:147）未传 host → 绑 0.0.0.0 全网卡。PaaS 部署或本机暴露于 LAN 时，`.git/config` 内嵌凭证（见 OD-SEC-02）可被任意同网段主机 curl 拉取，进而推 code 仓。`npm run dev` 本地虽仅 loopback 风险低，但 `npm start` 通道放大为网络可达。`.audit/*.jsonl` 泄漏引擎内部操作流水（userId/action/resource），属内部信息不应公开。

**remediation_sketch**: 在静态分发前（line 121 之前）加 dotfile 段守卫：
```js
// 阻断 dotfile/点目录泄露：safePath 任一段以 '.' 开头（.git/.audit/.claude/.gitignore/.opt-lock/.env...）→ 404。
// 与 ⑯ 穿越锁互补：⑯ 证不泄露 served 目录外文件，本守卫证不泄露 served 目录内敏感 dotfile。
const segs = safePath.split('/').filter(Boolean);
if (segs.some(s => s.startsWith('.'))) {
  res.writeHead(404, { 'Content-Type': 'text/plain; charset=utf-8' });
  res.end('Not found');
  return;
}
```
补回归锁 `test/smoke.test.js`：`GET /.git/config`、`GET /.git/HEAD`、`GET /.audit/<file>`、`GET /.env` 均断言 404 + body 不含机密标记。注意守卫须落在 `/healthz`/`/readyz`/`/api/health` 探针路由**之后**（探针无 dotfile 段不受影响）且静态分发**之前**。

**kind**: route-fix（单文件 dev-server.js + 测试，1-3 文件可解，复用 opt-worktree）
**priority**: 1（HIGH，凭证泄漏）

## Finding OD-SEC-02（HIGH, structural）— git remote URL 内嵌 Basic Auth 凭证

**file**: `.git/config`（非源码，ops 层；不进 opt-worktree 提交）
**line**: `[remote "origin"]` url 字段

**finding**: `git remote get-url origin` 返回 `https://syp02536326:HStZLJue84222jXR1oWW@code.alibaba-inc.com/OneDayAI-RPUKYRO/1BfrYW9R.git`——用户名 `syp02536326` + 密码/token `HStZLJue84222jXR1oWW` 明文嵌在 remote URL。这是 git 凭证反模式：凭证落 `.git/config`（本机）、可能进 git 历史/构建产物/备份；OD-SEC-01 使其经 dev-server 网络可达。即便 OD-SEC-01 修复后不再被 curl，凭证仍以明文形态存于 `.git/config`。

**evidence**: 见 OD-SEC-01 的 `GET /.git/config` 响应体（远程 URL 含 `user:pass@host`）。`HStZLJue84222jXR1oWW` 形态（22 字符字母数字混合）符合阿里 code push token 而非人脑密码。

**impact**: 凭证可推 code.alibaba-inc.com 仓 `OneDayAI-RPUKYRO/1BfrYW9R.git`。已通过 dev-server 暴露（OD-SEC-01实证）；即便堵住 dev-server，凭证仍可能在：git reflog / 构建产物 / 时间机器备份 / 共享开发机。须按"已泄漏"处置。

**remediation_sketch**: (a) 立即轮换：登录 code.alibaba-inc.com 吊销 `HStZLJue84222jXR1oWW` 并签发新 token；(b) 剥离内嵌：`git remote set-url origin https://code.alibaba-inc.com/OneDayAI-RPUKYRO/1BfrYW9R.git`（移除 `user:pass@`）；(c) 改用凭证助手 `git config --global credential.helper store`（0600 `~/.git-credentials`）或 `.netrc`，secret 不进 `.git/config`；(d) 扫描历史 `git log -p -- .gitconfig` 确认未提交。属 ops 动作，非源码提交，故 kind=structural 入 structural-debt 队列 SD-OD-01，引擎不自动修。

**kind**: structural（ops 卫生，无源文件可提交）
**priority**: 2（HIGH，但需人工执行轮换）

## Finding OD-SEC-03（MEDIUM）— 全响应零 Content-Security-Policy 头

**file**: `dev-server.js`
**line**: 46（仅 `res.setHeader('X-Content-Type-Options', 'nosniff')`，无 CSP）

**finding**: dev-server 已发 `X-Content-Type-Options: nosniff`（line 46，已实证 `/` 响应头），但**无 `Content-Security-Policy` 头**。`index.html` 从 CDN 加载 React/Tailwind/FontAwesome（`cdnjs.cloudflare.com` 等），无 CSP 时：CDN 遭供应链入侵、或任一组件引入 XSS、或网络中间人注入（HTTP 站点）→ 攻击者脚本可在浏览器上下文执行并读 `localStorage.getItem('1bfr_settings_v1')`（含 OD-SEC-04 的 apiKey），`fetch` 外传。`nosniff` 只防 MIME 嗅探，不防脚本执行。

**evidence**: `curl -sD - http://127.0.0.1:3091/ | head -15` 响应头仅 `X-Content-Type-Options: nosniff` + `Content-Type: text/html`，无 `Content-Security-Policy`。`test/smoke.test.js` grep `Content-Security|CSP|x-frame` 零命中（无回归锁）。

**impact**: 与 OD-SEC-04 叠加——apiKey 明文存 localStorage，任一 XSS 即可窃取。当前 UI 无 innerHTML/dangerouslySetInnerHTML/eval（grep 零命中，洁净），故为防御纵深缺口而非现存漏洞。但 React CDN 直连+SRI 缺失使供应链面非零。

**remediation_sketch**: 在 line 46 后补 CSP 头（与 nosniff 同机制，全局 setHeader 一次，所有路径自动带）：
```js
res.setHeader('Content-Security-Policy',
  "default-src 'self'; " +
  "script-src 'self' 'unsafe-inline' https://cdnjs.cloudflare.com; " +   // React Babel CDN + 内联 babel 块需 unsafe-inline
  "style-src 'self' 'unsafe-inline' https://cdnjs.cloudflare.com; " +     // Tailwind CDN + 内联 style
  "font-src 'self' https://cdnjs.cloudflare.com; " +
  "img-src 'self' data:; " +
  "connect-src 'self' http://127.0.0.1:8000; " +                          // /api/health 反代上游（ONEDAY_API_BASE 默认）
  "object-src 'none'; base-uri 'self'; frame-ancestors 'none';"
);
```
注意：`'unsafe-inline'` 因单文件 `<script type="text/babel">` 内联不可避免；SRI 对内联块不适用。若后续拆 CDN→自托管 React/Tailwind 可收紧去 `unsafe-inline` 与 CDN 域。补回归锁断言响应头含 `Content-Security-Policy`。connect-src 须含 `ONEDAY_API_BASE` 实际部署值（部署期 PaaS 注入时同步更新 CSP）。

**kind**: route-fix（单文件 dev-server.js + 测试，与 OD-SEC-01 同文件可栈堆）
**priority**: 3（MEDIUM）

## Finding OD-SEC-04（MEDIUM, structural）— LLM apiKey 明文存 localStorage

**file**: `index.html`
**line**: 1042（`const _lsKey="1bfr_settings_v1";`）/ 1043（`JSON.parse(localStorage.getItem(_lsKey)...)`）/ 1133（`localStorage.setItem(_lsKey, JSON.stringify({backend,model,baseUrl,apiKey,ctxWin,maxTok,effort}))`）

**finding**: SettingsPage 将 LLM 配置含 `apiKey` 明文写 `localStorage`。localStorage 性质：(a) 任何同源 JS 可读（XSS 即取，见 OD-SEC-03）；(b) 跨会话持久（浏览器关闭后仍在）；(c) 所有同源 tab 共享；(d) 无 HttpOnly 保护（HttpOnly 仅 cookie 可用）。CLAUDE.md 明示"与 `cli llm` 同源（localStorage）"——即浏览器 localStorage 是前端 UI 与 personal-assistant CLI 的**共享真相源**，CLI 也读这同一份。故 apiKey 必须落 localStorage 是**架构决定**，非偶然。UI 侧虽做掩码回显（`keyMask=apiKey.slice(0,4)+"…"+apiKey.slice(-4)`，line 1077），但存储值仍明文。

**evidence**: `grep localStorage index.html` 命中 line 1043/1080/1133；line 1133 `setItem` 载荷含 `apiKey` 字段明文（无加密、无引用句柄）。无 innerHTML/dangerouslySetInnerHTML/eval 命中（XSS sink 洁净，故当前无即时利用路径，但 OD-SEC-03 缺 CSP 使未来 XSS 不被阻断）。

**impact**: 单用户本地优先工具的理论威胁低，但：(1) OD-SEC-03 无 CSP → 未来引入任一第三方脚本/XSS 即可窃 key；(2) 共享开发机/浏览器配置文件共享 → 同 origin 用户互读；(3) 浏览器同步（Chrome Sync 等）可能把 localStorage 同步至云。最小缓解（OD-SEC-03 CSP）能堵 XSS 路径，但根本缓解须重构存储模型。

**remediation_sketch**: 架构性，非 1-3 文件 route-fix。方案：apiKey 不落浏览器，改由 personal-assistant 后端持有（env/0600 文件/keychain），前端只存 `apiKey_id`（引用句柄），调用经 `/api/llm/*` 后端反代（已有 `/api/health` 反代模式可复用）。CLI `cli llm` 同样改读后端而非 localStorage。跨 index.html + personal_assistant 后端 + cli 代码，需更大单位授权。过渡期 OD-SEC-03 CSP 作 stopgap。入 structural-debt SD-OD-02。

**kind**: structural（跨前端+后端+CLI 存储模型重构）
**priority**: 4（MEDIUM，需架构授权）

## 结论

oneday-design 是个工程素养高的小项目（2401 行回归锁覆盖绝大多数安全属性），但**静态分发的 dotfile 盲区是真实 HIGH 漏洞**——`.git/config` 内嵌凭证可被 `GET /.git/config` 直接 curl 拉取（已实证 200 + 凭证明文）。这不是"看似安全实则死代码"类问题，而是 ⑯ 回归锁覆盖维度外的真缺口。次优先是 CSP 缺失（与 localStorage apiKey 叠加的纵深缺口）。两 route-fix 同触 dev-server.js，下轮可栈堆单 worktree 派发 OD-SEC-01；OD-SEC-03 视 context 预算同栈或下轮。两 structural（凭证轮换、key 存储重构）入 structural-debt 待人工/授权。
