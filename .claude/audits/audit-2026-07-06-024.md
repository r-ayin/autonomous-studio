## Findings

### H-001 [HIGH] save-checkpoint.py:137 `Path` 引用前未导入 → 运行时 NameError

**位置**: `save-checkpoint.py:137` in `find_task_files()`
```python
for root, dirs, files in os.walk(Path(PROJECT_DIR) / ".claude"):
```
`Path` 直到 line 215 (`from pathlib import Path`) 才导入，且只在 `if __name__ == "__main__"` 块内。当 Hook 被 Claude Code 作为模块导入触发时，`find_task_files()` 永远不会执行成功——抛出 `NameError: name 'Path' is not defined`。

**影响**: checkpoint 永远不包含活跃任务文件列表；`recent_activity` 字段可能为空但不会 crash（因为 build_checkpoint 不直接调 find_task_files——但函数存在即死代码）。实际上 `build_checkpoint()` 不调用此函数，所以 bug 处于"潜伏态"：任何未来改动若引用 `find_task_files()` 会立刻暴露。

**严重度**: HIGH — 死代码带未绑定名，违反 Python 基本语义；属于静默失效类 bug。
**Kind**: route-fix
**Remediation**: 把 `from pathlib import Path` 移到文件顶部 import 区（line 13-17 附近），或删除 `find_task_files()` 函数（当前无人调用）。

### H-002 [HIGH] save-checkpoint.py:187-192 checkpoint 写入非原子 + latest.json copy 竞态

**位置**: `save-checkpoint.py:187-192`
```python
with open(filepath, "w", encoding="utf-8") as f:
    json.dump(checkpoint, f, ensure_ascii=False, indent=2)
import shutil
shutil.copy2(filepath, LATEST_LINK)
```
问题双重：
1. `json.dump` 直接写到目标路径，进程中途崩溃留下半截 JSON → 下次 resume-checkpoint.py 读到损坏文件。
2. `shutil.copy2` 不是原子操作；watchdog.sh 每 5 分钟读 `latest.json`，可能在 copy 中间读到半截内容 → `python3 -c "...json.load(f)..."` 抛 JSONDecodeError → watchdog 误判为"checkpoint missing"跳过一轮。

对比同项目 `incremental-save.py:44-58` 已经用了 tempfile+fsync+os.replace 的正确模式——证明团队知道怎么做，只是 save-checkpoint.py 没同步。

**严重度**: HIGH — 数据完整性风险，resume 链路关键节点。
**Kind**: route-fix
**Remediation**: 复制 incremental-save.py 的 atomic write pattern：tempfile.mkstemp 同目录 → json.dump → flush/fsync → os.replace → 同样方式更新 LATEST_LINK。

### M-001 [MEDIUM] protocol-check.py:68-72 subprocess.run env 泄漏父进程全部环境变量

**位置**: `protocol-check.py:68-72`
```python
result = subprocess.run(
    [sys.executable, BOOTSTRAP_SCRIPT, os.path.join(project_dir, "dummy")],
    capture_output=True, text=True, encoding="utf-8", timeout=10,
    env={**os.environ, "CLAUDE_PROJECT_DIR": WORKSPACE_ROOT}
)
```
子进程继承整个 `os.environ`，包括可能的敏感变量（API keys、tokens、credentials）。bootstrap.py 本身是可信脚本，但这是纵深防御缺口：如果 bootstrap.py 或其依赖将来被注入恶意代码，所有环境变量都会泄漏。

**严重度**: MEDIUM — 纵深防御缺陷，当前无直接利用路径。
**Kind**: route-fix
**Remediation**: 构造最小环境 dict：只传 `PATH`, `PYTHONPATH`, `CLAUDE_PROJECT_DIR`, `HOME`, `LANG` 等必要变量。

### M-002 [MEDIUM] decision-observer.py:207-214 session context read-modify-write 无锁

**位置**: `decision-observer.py:207-221` & `write_session_context():172-185`
两个并发会话同时触发 UserPromptSubmit → 各自读同一个 `{session_id}.json` → 各自 merge → 各自写回 → 后者覆盖前者。虽然 session_id 隔离了不同会话，但同一会话内的快速连续事件（如用户连发两条消息）仍可能丢字段。

**严重度**: MEDIUM — 数据丢失窗口小（毫秒级），且只影响统计计数。
**Kind**: route-fix
**Remediation**: 用 `fcntl.flock` 或 atomic write pattern；或者接受最终一致性（计数偏差可容忍），在注释里明确标注 trade-off。

### M-003 [MEDIUM] notify-phone.py:67 TCP payload 无长度前缀 → 粘包/截断

**位置**: `notify-phone.py:67-73`
```python
payload = f"{priority}|{title}|{message}"
sock.sendall(payload.encode("utf-8"))
```
裸 sendall 无长度前缀。Termux listener 端 recv() 可能在任意字节边界切分；如果 title/message 包含 `|` 字符，解析会错位。当前使用场景（短标题短消息）下大概率工作，但协议脆弱。

**严重度**: MEDIUM — 协议设计缺陷，实际故障概率低。
**Kind**: route-fix
**Remediation**: 加 4 字节大端长度前缀；或用 `\n` 终止 + 按行读取；title/message 做转义或 base64。
