## Executive Summary

引擎核心决策管线的五个 skill 整体设计合理、防御性编程到位（fail-safe 吞异常、原子写、路径穿越防护）。但深度审计发现 **3 HIGH / 4 MEDIUM / 3 LOW** 共 10 个问题，集中在：

1. **跨模块数据契约脆弱**：decision-observer 输出格式与 autonomous-loop 消费端无 schema 校验
2. **安全边界不完整**：pipeline-gate git 识别可被间接执行绕过；precheck shell 注入面
3. **评分器退化模式**：route-health-scorer E3 恒返回满分、E4 默认满分掩盖真实风险
4. **零测试覆盖**：5 个 skill 无任何测试文件

## Findings

### HIGH Severity

#### H-001: pipeline-gate.py git commit/push 检测可被间接执行绕过
- **file**: `分诊流水线/scripts/pipeline-gate.py`
- **line**: 76-100
- **severity**: high
- **kind**: route-fix
- **finding**: `_GIT_TOKEN` regex + `_CHAIN_SPLIT` 注释明确承认"不覆盖 eval/exec/sh -c/$()/反引号等间接执行壳"。攻击向量：`eval "git commit -m bypass"` 或 `sh -c "git push"` 可完全绕过 pipeline gate 的阶段约束和规模检查。虽然注释说"studio 流水线阶段约束被绕过的实际风险受 main gate 兜底"，但这意味着 pipeline-gate 作为独立防线是无效的——它的存在给使用者虚假安全感。
- **remediation**: 增加对 `eval|exec|sh\s+-c|\$\(|\`` 模式的阻断（保守策略：含这些 token 的命令一律 exit 2 要求人工确认），或在 SKILL.md 明确标注"本 gate 仅防直接调用，间接执行由 autonomous-commit-gate 兜底"并移除 PreToolUse Bash hook 的虚假拦截。推荐前者（defense-in-depth）。

#### H-002: studio-precheck.sh 内嵌 python3 -c 字符串插值存在 shell 注入面
- **file**: `心跳预检/scripts/studio-precheck.sh`
- **line**: 19-25, 43-49
- **severity**: high
- **kind**: route-fix
- **finding**: `python3 -c "...open('$CAL_FILE')..."` 和 `python3 -c "...open('$STATUS_FILE')..."` 使用 bash 变量直接插入 python 代码字符串。若 `$CAL_FILE` 或 `$STATUS_FILE` 路径含单引号（如项目目录名含 `'`），python 语法错误导致 except 分支 print(0)/print('true') → precheck 永远放行，cooldown/autoAdvance 检查静默失效。更严重：恶意构造的路径可注入任意 python 代码（虽然当前调用方传的是固定路径拼接，但脚本接受 $1 参数且未清洗）。
- **remediation**: 改用环境变量传递路径：`CAL_FILE="$CAL_FILE" python3 -c "import os,json; d=json.load(open(os.environ['CAL_FILE']))"` 或用 heredoc + stdin 传递。消除字符串插值注入面。

#### H-003: decision-observer.py confirm_studio_draft 写 status.json 非原子
- **file**: `决策观察/scripts/decision-observer.py`
- **line**: 211-254
- **severity**: high
- **kind**: route-fix
- **finding**: `confirm_studio_draft()` 读 status → 修改内存对象 → `safe_write_json(STUDIO_STATUS_FILE, status)` 用 open("w")+json.dump 覆写。这与 stop-completion-gate.py `_strike_count()` 注释中明确指出的漏洞模式相同（case-404, audit-2026-07-02-001 H-001）："原 open('w')+json.dump 非原子写在并发 Stop hook 触发时读端可能读到半写 JSON"。decision-observer 在 UserPromptSubmit+Stop 两个事件触发，若用户快速连续操作导致并发写入，status.json 可能被截断 → 后续所有读 status.json 的组件（precheck、completion-gate、autonomous-loop context injection）拿到空 dict → 引擎行为退化。stop-completion-gate 已用 tempfile+os.replace 修复了自身，但 decision-observer 没同步修。
- **remediation**: 将 safe_write_json 改为 tempfile+os.replace 原子写模式（与 stop-completion-gate._strike_count 同一 pattern）。

### MEDIUM Severity

#### M-001: route-health-scorer.py E3 外部稳定性恒返回满分
- **file**: `路由健康评分/scripts/route-health-scorer.py`
- **line**: 172-181
- **severity**: medium
- **kind**: route-fix
- **finding**: `score_e3_external_stability()` 无条件返回 `{"score": 2}`，detail 写明"CodeGraph 无直接贡献"。这意味着 E3 维度对总分贡献恒为 2/2，实际上只有 E1+E2+E4 三个维度在工作（max 变成 8 而非 10）。当决策 agent 依赖此分数做路线修正判断时，E3 维度的信号缺失会导致过度乐观。
- **remediation**: 至少实现基于 package.json/pyproject.toml 的版本变更检测（SKILL.md 建议过但未实现）；无法获取时返回 score=1 + detail="data unavailable" 而非满分。

#### M-002: route-health-scorer.py E4 git diff 失败时默认满分
- **file**: `路由健康评分/scripts/route-health-scorer.py`
- **line**: 184-208
- **severity**: medium
- **kind**: route-fix
- **finding**: `score_e4_cumulative_deviation()` 在 subprocess.run 抛异常时返回 `{"score": 2, "detail": "无法获取 git diff 统计，默认满分"}`。非 git 仓库、git 未安装、timeout 等场景全部给满分。与 H-003 同属"失败=open"反模式——评分器应该在数据不可用时给出中性分或标记不可用，而非假装一切正常。
- **remediation**: 异常时返回 `{"score": 1, "detail": "git diff unavailable", "unavailable": true}`，并在 score_all 聚合时将 unavailable 维度排除出 max_score 计算（避免分母虚高）。

#### M-003: triage.py --done 归档文件名序号竞态
- **file**: `分诊流水线/scripts/triage.py`
- **line**: 88-96
- **severity**: medium
- **kind**: route-fix
- **finding**: `n = len(list(hist.glob("*.json"))) + 1` 后写 `{n:04d}-...json`。并发 triage --done（多 session 同时收尾）时，两个进程可能算出相同 n → 后者覆写前者归档。虽概率低（单用户引擎），但与 _strike_count 同源问题。另外 `current_path(root).unlink(missing_ok=True)` 注释已解释了竞态防御，但归档文件名冲突未防。
- **remediation**: 改用时间戳前缀 `{timestamp}-{kind}.json` 或 tempfile+rename 模式。

#### M-004: audit-cycle-state.json 引用 SD-PD-001/SD-PD-002 但 structural-debt.md 无对应条目
- **file**: `autonomous-studio/.claude/audit-cycle-state.json`
- **line**: 23 (`"structural_debts_new": ["SD-PD-001", "SD-PD-002"]`)
- **severity**: medium
- **kind**: structural
- **finding**: audit-2026-07-03-013 声称新增两条 structural debt（SD-PD-001, SD-PD-002），但 `.claude/structural-debt.md` 中没有这两个 ID 的任何记录。这是审计产出物之间的引用断裂——future context 读到 audit-cycle-state 会去找这两条 debt 但找不到，导致结构性问题丢失追踪。
- **remediation**: 回溯 audit-2026-07-03-013 report 补写 SD-PD-001/SD-PD-002 到 structural-debt.md，或在 audit-cycle-state 中标注 `_lost: true`。本轮审计先记录此 finding，派生 fix 时补写。

### LOW Severity

#### L-001: decision-observer.py count_decisions 正则单次编译但未缓存
- **file**: `决策观察/scripts/decision-observer.py`
- **line**: 448-461
- **severity**: low
- **kind**: route-fix
- **finding**: `_DECISION_RE = re.compile(...)` 在函数体内每次调用都重新编译。虽然 CPython re 有内部缓存使性能影响微小，但违反了代码自身的注释承诺"单次编译正则，一次扫描"。应移到模块级常量。
- **remediation**: 将 `_DECISION_RE` 提升到模块顶层。

#### L-002: pipeline-gate.py DOCS 白名单缺少 .claude/ 子目录通配
- **file**: `分诊流水线/scripts/pipeline-gate.py`
- **line**: 32-33, 139-148
- **severity**: low
- **kind**: route-fix
- **finding**: `_is_doc_or_meta` 对 `.pipeline/` 和 `planning/` 做了前缀匹配放行，但对 `.claude/` 目录只靠 DOCS set 里的具体文件名（CLAUDE.md 等）。`.claude/decisions/case-*.json`、`.claude/memory/*.md` 等引擎自维护文件的写入会被 pipeline-gate 拦截（除非已有 current.json）。这在实际使用中可能导致引擎自我更新被自己的门禁挡住。
- **remediation**: 在 `_is_doc_or_meta` 中添加 `.claude` 前缀匹配（与 .pipeline/planning 对称）。

#### L-003: autonomous-loop.sh PROMPT 变量中的 scout-scan 路径硬编码
- **file**: `自主循环/scripts/autonomous-loop.sh`
- **line**: 82
- **severity**: low
- **kind**: route-fix
- **finding**: PROMPT 字符串中 `scripts/scout-scan.py --workspace '"$WORKSPACE"'` 使用了 shell 变量展开嵌入 heredoc-style 字符串。若 WORKSPACE 路径含单引号，PROMPT 内的引号配对被破坏 → claude -p 收到的 prompt 语法错误。当前 workspace=/home/admin/workspace 安全，但不可移植。
- **remediation**: 对 WORKSPACE 做引号转义或用 printf %q。

## Cross-Module Data Flow Analysis

```
User Input → decision-observer(UserPromptSubmit) → decision-log.jsonl
                                                    ↓ classify/intent
                                              status.json (read/write)
                                                    ↓
Engine Heartbeat → studio-precheck.sh → proceed/skip
                      ↓ reads: calibration.json, autonomous-state.md, status.json
                      ↓
Agent Spawn → triage.py → .pipeline/current.json
                              ↓
Code Change → pipeline-gate(PreToolUse) → block/allow
                              ↓ reads: current.json, git diff
                              ↓
Stop Event → stop-completion-gate → block(exit 2)/allow(exit 0)
           → decision-observer(Stop) → decision-log.jsonl + autonomous-state-runtime.md
                                        ↓ additionalContext injection
                                   Next Claude Session
```

**关键脆弱点**：status.json 是非原子写的共享状态（H-003），被 4/5 个组件读写。它是整个管线的 single point of failure。

## Structural Debt Candidates

### SD-ES-001: 引擎 skill 零测试基础设施
- **description**: engine-skills-extracted 下所有 skill 均无测试文件。这些 skill 是从主引擎提取出来的独立组件，提取过程丢失了原有的测试关联（如果曾有）。需要建立统一的测试框架（pytest + conftest fixtures for mock git repos/status files）、CI gate、以及至少覆盖每个 skill 的核心 happy path + 已知 regression 点。
- **affected_files**: 全部 5 个 skill 目录
- **severity**: high
- **status**: open

### SD-ES-002: status.json 共享状态无并发保护协议

- **affected_files**: decision-observer.py, studio-precheck.sh (via inline python), confirm_studio_draft, safe_write_json
- **severity**: medium
- **status**: open
