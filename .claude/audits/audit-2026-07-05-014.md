## 执行摘要

wanxia-edit 是晚霞预报系统的"内容出口"：从 posts.json 读城市评分 → 生成小红书文案 + AI 封面图 → SMTP 邮件推送。审计发现 **1 个 HIGH 安全漏洞**（GPT-Image API 下载链 SSRF / 本地文件读 → 邮件外泄链）、**2 个 MED 结构性死代码**（server.js / social-scraper.js 引用不存在的 `src/` + `storage.js` + `node_modules`，整体不可启动）、**3 个 LOW**（SMTP_PORT 解析、env.js 路径不一致、Chromium `--no-sandbox`）。

最严重的是 `generate-cover-image.py:download_image()` —— 它用 `urllib.request.urlopen` 拉取 API 返回的 `result_url`，**未校验 scheme / host / IP 字面量 / 大小**。实测 `urllib` 默认 opener 含 `FileHandler`，`file:///etc/passwd` 可直接读。攻击链：被 compromise 的 GPT-Image API（或 MITM，因为 Referer/endpoint 是第三方 `api.lk888.ai`）返回 `result_url=file:///etc/passwd` → `download_image` 把 /etc/passwd 写到 `00-national-cover.png` → `mail_push.py` 把 covers/*.png 作为 JPG 附件邮件外发 → 本地文件读 + 外泄。这与 **audit-2026-07-04-013** 刚给 `generate-image` skill 修的 H02/L02（hostname allowlist + IP 字面量 block + 50MB size cap + Content-Length precheck）是**同一漏洞类**，本文件零防护。

copy-generator.js 本身是纯字符串拼接、无 eval / 无 I/O，无安全面。social-scraper.js 的微博/小红书采集用 `encodeURIComponent` 拼 containerid、Playwright DOM 提取，无明显注入；但 `chromium.launch` 带 `--no-sandbox`。

## Findings

### 🔴 HIGH

#### WANXIA-SSRF-01 — `download_image` 无 scheme/host 校验 → file:// 本地文件读 + SSRF + 无大小上限
- **file**: `generate-cover-image.py`
- **line**: 388-407（`download_image`），调用点 469（`self.download_image(result_url, save_path)`），`result_url` 来自外部 API 响应 `status.get("result_url")`（line 462）
- **kind**: route-fix
- **severity**: high
- **finding**:
  `download_image(url, save_path, max_retries=5)` 用 `urllib.request.urlopen(req, timeout=120)` 拉取 `url`，未做任何校验：
  1. **无 scheme allowlist** — `urllib` 默认 opener 含 `FileHandler`（已实测：`urllib.request.urlopen(Request('file:///etc/hostname'))` 成功读取）。`result_url` 来自第三方 API `api.lk888.ai` 的响应体，若 API 被 compromise 或响应被篡改为 `file:///etc/passwd` / `file:///home/admin/.ssh/id_rsa`，`resp.read()` 返回文件内容，`save_path.write_bytes(...)` 把它写到 `COVER_DIR/00-national-cover.png`。
  2. **无 host allowlist / 无 IP 字面量 block** — `http://169.254.169.254/latest/meta-data/`（云元数据）、`http://127.0.0.1:*/`（内网服务）均可被 urlopen 拉取并落盘。
  3. **无大小上限 / 无 Content-Length 预检** — `resp.read()` 一次性读全 body 到内存再写盘，恶意响应可返回 GB 级 payload → OOM / 磁盘填满（covers 目录写爆）。
  4. **外泄链**：`mail_push.py:142-165` 把 `covers/*.png` 转 JPG 后作为邮件附件外发到 `MAIL_TO`。即 file:// 读到的本地敏感文件会被自动附在下一封晚霞预报邮件里发出。
- **evidence**:
  - 默认 opener handlers 实测含 FileHandler：`['UnknownHandler','HTTPHandler','HTTPDefaultErrorHandler','HTTPRedirectHandler','FTPHandler','FileHandler','DataHandler','HTTPSHandler','HTTPErrorProcessor']`
  - 实测 `urllib.request.urlopen(Request('file:///etc/hostname'), timeout=5).read()` → `b'session0c32610aed00457ea4-qr28p\n'`（成功读宿主 hostname）
  - 与已修的 generate-image skill 同类：commit `617f8a7`（H02 hostname allowlist + IP literal block）、`b528ce4`（L02 50MB cap + Content-Length precheck）—— 本文件无任何对应防护。
- **remediation**: 在 `download_image` 入口加：
  ```python
  from urllib.parse import urlparse
  import ipaddress, socket
  ALLOWED_HOSTS = {"api.lk888.ai"}  # GPT-Image CDN host（按实际 CDN 扩充）
  MAX_BYTES = 50 * 1024 * 1024
  p = urlparse(url)
  if p.scheme != "https": raise ValueError(f"blocked scheme {p.scheme}")
  host = p.hostname or ""
  # 解析 IP 字面量，拒内网/环回/链路本地
  try:
      ip = ipaddress.ip_address(host)
      if ip.is_private or ip.is_loopback or ip.is_link_local or ip.is_reserved:
          raise ValueError(f"blocked ip {ip}")
  except ValueError:
      if host not in ALLOWED_HOSTS: raise ValueError(f"blocked host {host}")
  # Content-Length 预检 + 流式读 + 上限截断
  with urllib.request.urlopen(req, timeout=120) as resp:
      cl = resp.headers.get("Content-Length")
      if cl and int(cl) > MAX_BYTES: raise ValueError("too large")
      save_path.parent.mkdir(parents=True, exist_ok=True)
      buf = bytearray()
      while len(buf) < MAX_BYTES:
          chunk = resp.read(65536)
          if not chunk: break
          buf.extend(chunk)
      if len(buf) >= MAX_BYTES: raise ValueError("exceeded cap")
      save_path.write_bytes(bytes(buf))
  ```
  并把 `result_url` 非 https / 非 allowlist host 时直接 `return None`（不重试，不落盘）。

### 🟠 MEDIUM

#### WANXIA-DEAD-01 — `server.js` 引用 8 个不存在模块，整体不可启动（dead/stale entry point）
- **file**: `server.js`
- **line**: 1, 20-26（`import './src/env.js'`, `import routes from './src/routes.js'`, `initDatabase from './src/storage.js'`, `restoreWeights/runDailyCalibration/fetchActualWeather from './src/calibration.js'`, `fetchAndStoreSocialData... from './src/social-scraper.js'`, `runSocialCalibration from './src/social-calibration.js'`, `startPrefetchScheduler from './src/prefetch-scheduler.js'`, `cities from './src/cities.js'`）+ 16,19（`express`, `node-cron`）
- **kind**: structural
- **severity**: medium
- **finding**: server.js 是项目唯一标注的 HTTP 入口（`app.listen(8080)`），但它 import 的 8 个本地模块全部缺失——磁盘上没有 `src/` 目录，也没有 `node_modules/`（express/node-cron/playwright 都没装）。实测 `ls src node_modules` 均报 No such file or directory。任何 `node server.js` 启动尝试都会在第一条 import 处抛 `ERR_MODULE_NOT_FOUND`。真正能跑的是 4 个根目录脚本（copy-generator.js 作为库、social-scraper.js 作为库但同样 import 了不存在的 storage.js、generate-cover-image.py / mail_push.py 作为 CLI）。cron 编排（22:00 校准 / 12:00 生成 / 23:30 社媒校准）实际无人触发——要么另有部署机制把 src/ 树补齐，要么 server.js 是上游仓库的 stale 残留。
- **evidence**: `find . -maxdepth 3 -name '*.js'` 仅返回 6 个根目录文件，无 `src/`；`ls node_modules` → No such file；`ls src` → No such file。
- **remediation**: 二选一——(a) 若 wanxia-edit 是独立可运行项目：从上游补齐 `src/{env,routes,storage,calibration,social-scraper,social-calibration,prefetch-scheduler,cities}.js` + 装依赖（express/node-cron/playwright），否则 server.js 永远跑不起来；(b) 若本目录只是"4 个脚本的松散集合"：删除 server.js（dead code），把 cron 配置挪到系统 crontab / systemd timer 直接调 `python generate-cover-image.py` + `python mail_push.py` + `node social-scraper.js`。当前状态最坏：运维以为 `node server.js` 在跑，实际从没跑过，cron 静默失效。

#### WANXIA-DEAD-02 — `social-scraper.js` import 不存在的 `./storage.js`，模块加载即崩
- **file**: `social-scraper.js`
- **line**: 11（`import { storeSocialObservation } from './storage.js'`）
- **kind**: structural
- **severity**: medium（降为 low 亦可，因 server.js 已 dead 不会触发链式加载；但若有人单独 `node social-scraper.js` 测试会撞）
- **finding**: 与 WANXIA-DEAD-01 同根因——`storage.js` 在磁盘上不存在。`storeSocialObservation` 在 `fetchAndStoreSocialData`（line 327）被调用。即使只想用 social-scraper 的 fetch 能力，ESM 顶层 import 会在模块求值期就抛 `ERR_MODULE_NOT_FOUND`。
- **evidence**: `find . -name 'storage.js'` 无结果。
- **remediation**: 随 WANXIA-DEAD-01 一并处理——补 storage.js，或把 `storeSocialObservation` 调用改为可选（try/catch 动态 import）并从顶层 import 移除。

### 🟡 LOW

#### WANXIA-SMTP-01 — `SMTP_PORT` 非数字直接 crash，无友好退出
- **file**: `mail_push.py`
- **line**: 94（`port = int(os.environ.get("SMTP_PORT", "465"))`）
- **kind**: route-fix
- **severity**: low
- **finding**: 若 `SMTP_PORT` 被设成非数字（如 `465/tcp` 或空格串），`int()` 抛 `ValueError`，未被 try/except 包裹，main() 直接 traceback 退出（退出码 1 来自未捕获异常，而非干净的 1）。cron 里表现为"脚本崩 + 堆栈 stderr"，不利于排障。
- **evidence**: 静态读 line 92-102，`run()` 只对 `not (host and user and pwd and to)` 做了空值检查，未对 port 做类型校验。
- **remediation**:
  ```python
  try:
      port = int(os.environ.get("SMTP_PORT", "465"))
  except ValueError:
      print("⚠️ SMTP_PORT 非数字，跳过邮件推送", file=sys.stderr)
      return 1
  ```

#### WANXIA-ENV-01 — env.js 与 server.js 的 .env 路径不一致
- **file**: `env.js`
- **line**: 22（`const envPath = join(__dirname, '..', '.env')`）vs `server.js:6`（`new URL('.env', import.meta.url)` → 当前目录）
- **kind**: route-fix
- **severity**: low
- **finding**: env.js 去**父目录**找 .env，server.js 去**当前目录**找 .env。两个加载器各读各的，若 .env 只放在一处，必有一个加载器静默 fallback 到真实环境变量（env.js 的 catch 块吞错）。鉴于 WANXIA-DEAD-01（server.js 不可跑）此 finding 当前是 moot，但一旦修复 DEAD-01 让 server 跑起来，路径不一致会让 cookie/token 在两个模块里看到不同值（典型就是 social-scraper 捕获到空 cookie 的 bug，env.js 的注释自己都写了这个坑）。
- **evidence**: env.js 注释明确说"ESM import 提升，social-scraper 会捕获空 cookie"，但自己又去父目录读 .env，与 server.js 当前目录的读取不统一。
- **remediation**: 统一为一个 `resolveEnvPath()`——优先 `process.env.WANXIA_ENV_PATH`，否则从项目根（可由 `find-up` 或固定相对路径）加载；两个加载器共用同一函数。或干脆删一个（env.js 已存在就让它独占，server.js:6-14 的二次加载删掉）。

#### WANXIA-XHS-NO-SANDBOX-01 — Chromium `--no-sandbox` 以 root 跑时削弱隔离
- **file**: `social-scraper.js`
- **line**: 26-33（`chromium.launch({ args: ['--no-sandbox', ...] })`）
- **kind**: route-fix
- **severity**: low (info-adjacent)
- **finding**: Playwright headless Chromium 带 `--no-sandbox`。容器内常见且 Playwright 默认就加，但若进程以 root 身份跑（cron 常以 root），seccomp/sandbox 全失，浏览器渲染引擎漏洞直接提权到 root。小红书页面含第三方 JS，攻击面非零。
- **remediation**: 若必须 root 跑，至少加 `--disable-setuid-sandbox` 之外的真实沙箱（bwrap/firejail）；理想是降权到非 root 用户跑 chromium。优先级低，但记一笔。

## 结构性债务

### SD-WANXIA-01 — 项目布局碎片化：根目录 4 脚本 vs server.js/social-scraper.js 引用的 src/ 树不一致
- **kind**: structural
- **severity**: medium
- **description**: wanxia-edit 磁盘上是"6 个根目录文件 + 2 张图 + __pycache__"，但 server.js 和 social-scraper.js 的 import 声明了一个 `src/` 模块化布局（env/routes/storage/calibration/social-scraper/social-calibration/prefetch-scheduler/cities）+ npm 依赖（express/node-cron/playwright）。两者无法同时为真：要么 src/ 树在上游被删/未同步、server.js 是 stale 残留，要么根目录 4 脚本是后来抽出来的快照、import 路径没改。结果是项目目前**没有一个可引导的入口**——`node server.js` 崩、`node social-scraper.js` 崩、只有两个 .py CLI 能独立跑。
- **impact**: 运维若按 server.js 的 cron schedule（22:00/12:00/23:30）期望自动校准/生成/推送，实际全静默失效；社媒数据不采集 → 校准权重无样本 → 预测精度退化无人察觉。
- **fix direction**: 由 owner 澄清 wanxia-edit 的"正式布局"——是 `src/` monolith（则补齐模块+依赖）还是"4 脚本松散集合"（则删 server.js + 改 social-scraper.js 去 storage.js 依赖 + cron 走系统 crontab）。在澄清前，派生 fix 只能治标（删 dead server.js / stub storage 调用）。
