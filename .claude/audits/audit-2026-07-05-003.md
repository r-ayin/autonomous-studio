## Executive summary

sunset-prediction is a single-host Node.js/Express + SQLite sunset-quality predictor that auto-posts to Xiaohongshu (小红书) via Playwright + an LLM browser-use agent, using real XHS creator-platform session cookies and QQ SMTP credentials. The architecture is internally consistent and several controls are genuinely good (all sqlite queries are parameterized; `timedFetch` has a 9s timeout; `/social/fetch` date is regex-validated; `/grid/:name` is whitelist-guarded; engagement counts are capped at 5000 to suppress viral outliers). However the **credential and exposure surface is unsafe** for anything beyond a strictly-localhost personal box:

- **A live XHS session-cookie store is committed to git and pushed to origin** (`scripts/.xhs_storage_state.json.bak`, 15 cookies incl. `access-token-creator.xiaohongshu.com`, `galaxy_creator_session_id`, `x-user-id-creator.xiaohongshu.com`). `.gitignore` ignores the live `…storage_state.json` but not the `.bak` variant — classic hygiene gap. The remote is `code.alibaba-inc.com/xtool/wanxia.git`, so the tokens are on the corporate git server and must be treated as compromised.
- **The git remote URL itself embeds credentials** (`syp02536326%40taobao.com:HStZLJue84222jXR1oWW@…`) in `.git/config`.
- **The Express server has zero auth and binds 0.0.0.0:8080** — anyone network-reachable can `POST /api/xhs/generate` (consumes real XHS posting quota + cookies), `POST /api/social/fetch` (drives Playwright + scrapes), or `POST /api/social/calibrate`.
- Cookie material is routinely printed to stdout (cron logs) and `.env` is rewritten by a keepalive script with default 0644 perms.

8 route-fix findings derived (next round dispatches them one per round). 2 structural debts queued (auth/RBAC layer; cookie/secrets lifecycle). No code changed this round — audit-only, per cycle protocol.

## Findings

### SP-SEC-01 — HIGH — XHS session-cookie store committed to git + pushed to origin

- **file**: `scripts/.xhs_storage_state.json.bak`, `.gitignore`
- **line**: `.gitignore:1-3` (ignores `.env`, `*.pem`, but the `scripts/.xhs_storage_state.json` rule at line ~21 only matches the exact filename, not the `.bak`)
- **kind**: route-fix
- **evidence**:
  - `git ls-files` → `scripts/.xhs_storage_state.json.bak` is tracked.
  - `git ls-tree -r --name-only origin/main` → `scripts/.xhs_storage_state.json.bak` is on origin/main (remote `code.alibaba-inc.com/xtool/wanxia.git`).
  - File content (12733 bytes): 15 cookies, `has_values: True`, names include `access-token-creator.xiaohongshu.com`, `galaxy_creator_session_id`, `galaxy.creator.beaker.session.id`, `x-user-id-creator.xiaohongshu.com`, `customer-sso-sid`, `acw_tc`, `a1`, `webId`, `xsecappid` — these are XHS creator-platform authentication/session tokens.
  - Introduced by commit `32ccdb3` "同步 GitHub r-ayin/wanxia 内容" — the leak is in git history on the remote, not just the working tree.
  - `.gitignore` line `scripts/.xhs_storage_state.json` does NOT match `…json.bak`.
- **impact**: Anyone with read access to the origin repo (corporate git server, plus any mirror/fork from the prior GitHub sync) holds a full XHS creator session — can post as the account, read DMs, scrape analytics, trigger rate-limit/风控 against the operator's account. The `.bak` was captured at `Jun 28 17:09`; sessions rotate ~24h so the access-token is likely stale by now, but `galaxy_creator_session_id` / `a1` / `webId` are longer-lived identity cookies and the file is a permanent fixture in history.
- **remediation**: (a) `git rm --cached scripts/.xhs_storage_state.json.bak`; (b) broaden `.gitignore` rule to `scripts/.xhs_storage_state.json*` (glob, covers `.bak`/`.tmp`/`.old`); (c) **rotate the XHS session** at creator.xiaohongshu.com (sign out all sessions / re-login) since the leaked tokens are in remote history; (d) purge history with `git filter-repo --path scripts/.xhs_storage_state.json.bak --invert-paths` and force-push (coordinate — rewrites shared remote). (a)+(b) is the route-fix unit; (c)+(d) are operator actions documented in the case but not automated by the engine.

### SP-SEC-02 — MEDIUM — Git remote URL embeds credentials

- **file**: `.git/config` (remote `origin`)
- **line**: `origin url = https://syp02536326%40taobao.com:HStZLJue84222jXR1oWW@code.alibaba-inc.com/xtool/wanxia.git`
- **kind**: route-fix
- **evidence**: `git remote -v` shows the URL-encoded Taobao email `syp02536326%40taobao.com` and password `HStZLJue84222jXR1oWW` embedded in the remote URL. Any tool that reads `.git/config` or runs `git remote -v` (CI, hooks, `scout-scan`, error logs that dump `git` invocations) leaks the credential pair.
- **impact**: Plaintext credential for the corporate git account sitting in every clone's `.git/config`.
- **remediation**: `git remote set-url origin https://code.alibaba-inc.com/xtool/wanxia.git` (strip embedded creds); authenticate via credential helper / `GIT_ASKPASS` / SSH. This is a runtime git-config operation, not a source commit — the derived fix registers it as a manual op (no `_planned_files`), to be executed by the operator or an engine round that runs the set-url command. **Also rotate the `HStZLJue84222jXR1oWW` password** — it is now in `.git/config` of every clone and may already be in logs.

### SP-HTTP-01 — HIGH — Express server has no auth and binds 0.0.0.0; unauthenticated state-changing + credential-consuming routes

- **file**: `server.js`, `src/routes.js`
- **line**: `server.js:140` (`app.listen(PORT, …)` — no host arg → binds `0.0.0.0`); `routes.js:396` (`POST /social/fetch`), `routes.js:426` (`POST /social/calibrate`), `routes.js:461` (`POST /xhs/generate`)
- **kind**: route-fix (bind-localhost is a one-line fix; full auth middleware is structural — see SD-SP-01)
- **evidence**: No auth middleware anywhere in `server.js`/`routes.js` — no token check, no `app.use(basicAuth)`, no rate limit. `app.listen(PORT)` defaults to all interfaces. `POST /api/xhs/generate` (`routes.js:461-484`) calls `execAsync('node scripts/publish-xhs.js')` (uses the real XHS cookie + Playwright) and is guarded only by `hasXiaohongshuCookie()` (a 503 if absent) — not by caller identity. `POST /api/social/fetch` opens an SSE stream that drives `fetchAndStoreSocialData` (Playwright + weibo/XHS scraping) for anyone who asks.
- **impact**: Any network-reachable client can: burn the operator's XHS posting quota via `/xhs/generate`; trigger Playwright browser launches (DoS-by-browser — each `/xhs/generate` runs a 120s headless Chromium); run social scraping that risks XHS/Weibo 风控 against the operator's cookies; run calibration that mutates DB weights. On a personal laptop behind NAT this is low; on a cloud VM (CLAUDE.md says "跑本地电脑或云服务器") it is high.
- **remediation**: `app.listen(PORT, '127.0.0.1')` (bind loopback only) — minimal route-fix. If remote access is needed, add a token gate (`Authorization: Bearer $SP_API_TOKEN` from env, unset → 127.0.0.1 only) — that is structural SD-SP-01.

### SP-XHS-01 — MEDIUM — Full XHS cookie string printed to stdout (cron/log leak vector)

- **file**: `scripts/export-xhs-cookie.js`, `scripts/post_xhs_browseruse.py`
- **line**: `export-xhs-cookie.js:67` (`console.log(\`   XIAOHONGSHU_COOKIE=${cookieStr}\`)`); `post_xhs_browseruse.py:144-145` (`print(f"\n📋 请将以下 cookie 更新到 .env 的 XIAOHONGSHU_COOKIE："); print(f"   XIAOHONGSHU_COOKIE={cookie_str}")`)
- **kind**: route-fix
- **evidence**: Both scripts dump the full cookie string (all name=value pairs) to stdout. The 22:30 cron (`server.js:72`) and 4h keepalive cron (`server.js:124`) run via `execAsync`/node-cron and capture stdout/stderr into cron logs. If `post_xhs_browseruse.py --setup` is ever run under cron or with redirected stdout, the live cookie lands in `logs/` or journald.
- **remediation**: Redact to `${name}=…${value.slice(-6)}` for the console message; write the full value only to `.env` (which is gitignored) via the existing `.env`-update code path. One-line change per file.

### SP-XHS-02 — MEDIUM — `xhs_keepalive.py` rewrites `.env` every 4h; cookie/secrets file has no 0600 enforcement

- **file**: `scripts/xhs_keepalive.py`
- **line**: `xhs_keepalive.py:45-57` (`update_env_cookie` reads `.env`, regex-replaces the XIAOHONGSHU_COOKIE line, writes back), `:104-109` (exports `storage_state` + rewrites `.env` cookie)
- **kind**: route-fix
- **evidence**: The 4-hourly keepalive cron calls `update_env_cookie(cookie_str)` which does `ENV_FILE.write_text(new_content, encoding="utf-8")` — no `os.chmod(0o600)`, no temp+atomic-rename. `.env` holds `SMTP_PASS`, `ANTHROPIC_API_KEY`, `XIAOHONGSHU_COOKIE` (long-lived identity tokens). On a multi-user host `ls -la .env` would show 0644 (world-readable). The `storage_state.json` write in `save_storage_state` (`post_xhs_browseruse.py:163`) and `export_storage_state` have the same gap.
- **remediation**: After writing `.env` / `storage_state.json`, `os.chmod(path, 0o600)`. Better: write to a temp file with 0600 and `os.replace` for atomicity. If `.env` doesn't exist, create with 0600. (The working tree currently has no `.env` — but the code path must be correct for when one is provisioned.)

### SP-CLI-01 — MEDIUM (defense-in-depth) — `innerHTML` with server data in detail-panel/app.js; no escaping helper

- **file**: `public/js/detail-panel.js`, `public/js/app.js`
- **line**: `detail-panel.js:65-148` (`renderDetailPanel` builds a template string interpolating `city.name`, `city.nameEn`, `city.province`, `city.region`, `city.dominantColor.hex`, `city.dominantColor.name` into `html`, then `openPanel` does `content.innerHTML = renderDetailPanel(city)` at `:194`); `app.js:35-41,53-59,65-71` (`updateSummary`/`updateGridSummary`/`updateContourSummary` interpolate `best.name`, `s.averageScore` into `bar.innerHTML`)
- **kind**: route-fix
- **evidence**: The DOM sink is `innerHTML` with no escaping. **Today** the data is not attacker-controlled: `city.name`/`province`/`region` come from the static `src/cities.js` array; `dominantColor.name` from `copy-generator.js`; `r.date`/`r.score` from the DB (numeric/date). So no live XSS chain exists. The finding is defense-in-depth: the moment any field becomes user/scrape-influenced (e.g. a future route renders XHS post titles — `social-scraper.js` does capture `rawPosts[].title`-ish fields), the unescaped `innerHTML` becomes exploitable.
- **remediation**: Either (a) switch city-name/color-name interpolations to `textContent` where the value is plain text, or (b) add a `esc()` helper (`str.replace(/[&<>"']/g, …)`) and use it in the template. Minimal 1-file change to `detail-panel.js`.

### SP-HTTP-03 — LOW — Error responses leak internal `err.message` to client

- **file**: `src/routes.js`
- **line**: `:130` (`res.status(502).json({ error: '气象数据暂时不可用', detail: err.message, retry: 30 })`), `:340`, `:350`, `:360` (`res.status(500).json({ error: '…失败', detail: err.message })`), `:178`, `:239` (grid/contour `detail: err.message`)
- **kind**: route-fix
- **evidence**: Internal error strings (sqlite errors, file paths, fetch stack) are mirrored into the JSON response `detail` field. Information disclosure of backend internals to any caller.
- **remediation**: Log full `err` server-side (`console.error`); return only the generic `error` string + a correlation id. Drop the `detail: err.message` fields.

### SP-FETCH-01 — LOW — `weather-fetcher.js` / `grid-fetcher.js` `fetch()` with no per-request timeout and no response size cap

- **file**: `src/weather-fetcher.js`, `src/grid-fetcher.js`
- **line**: `weather-fetcher.js:7` (`const res = await fetch(url)` — no AbortController, no size cap); `grid-fetcher.js:61` (`const res = await fetch(url)` inside `fetchJSON` — has 429-retry but no timeout; `res.json()` unbounded)
- **kind**: route-fix
- **evidence**: open-meteo is a trusted upstream, but there is no timeout (a stalled connection blocks the request indefinitely — `weather-fetcher` is on the `/predictions` hot path) and no response-size cap (a compromised/hijacked upstream could return a multi-GB body → OOM). Contrast with `social-scraper.js:103-113` `timedFetch` which correctly uses `AbortController` + 9s timeout — that pattern should be reused here.
- **remediation**: Port the `timedFetch` pattern (AbortController timeout 9-15s) into `weather-fetcher.js` and `grid-fetcher.js`; optionally cap `res.json()` size by checking `Content-Length` and streaming-truncating.

### SD-SP-02 — Cookie / secrets lifecycle sprawl

- **audit_id**: audit-2026-07-05-003
- **project**: sunset-prediction
- **description**: XHS credential material flows through 5 surfaces with no single source of truth: `.env` (`XIAOHONGSHU_COOKIE`), `scripts/.xhs_storage_state.json` (Playwright storage_state), `scripts/.xhs_storage_state.json.bak` (committed leak — SP-SEC-01), stdout prints (SP-XHS-01), and the keepalive rewrite loop (`xhs_keepalive.py` mutates `.env` every 4h — SP-XHS-02). Rotating or revoking the XHS session requires touching all five; the keepalive loop actively re-extracts and re-writes cookies, so a leaked-then-rotated token can silently re-leak via the next keepalive stdout/`.env` write. Needs a single secrets store (0600 file or OS keychain) with read-only access from the scrapers and no stdout echo, plus a `--rotate` subcommand that invalidates the old session atomically. Cross-file, not a minimal unit.
- **affected_files**: scripts/xhs_keepalive.py, scripts/export-xhs-cookie.js, scripts/post_xhs_browseruse.py, src/social-scraper.js, .env, .gitignore
- **severity**: high
- **status**: open
- **source_finding**: SP-SEC-01, SP-XHS-01, SP-XHS-02
