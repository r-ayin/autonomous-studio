### HV-001 · verifier.py:445 merge_retry_report 传入单元素列表丢失历史
- **File**: `script-verifier/verifier.py` L445
- **Severity**: HIGH
- **Kind**: route-fix
- **Finding**: `retry` 子命令在达上限时调用 `merge_retry_report([report])`，仅传入当前单次报告。函数内部 `final = attempts[-1].copy()` 后遍历 `attempts` 构建 `retry_history`，但 `[report]` 只有 1 项 → 合并产出的 `retry_history` 永远只有 1 条，前 N-1 次尝试的 falsified/unverifiable 计数全部丢失。L433 已从报告读出 `_retry_history` 并 append 了本次，但这个累积的 history 没传给 merge，只用作 `merged["_retry_history"] = history` 旁路写回——与 merge 自建的 `retry_history` 字段不一致，下游消费者读到哪个取决于字段名拼写。
- **Impact**: 修复闭环的可观测性断裂；Claude/人工复盘时看不到真实重试轨迹；PASS_AFTER_N_RETRIES 状态判定基于残缺数据。
- **Remediation**: 把 `history` 完整传给 merge：要么改签名让 merge 接收外部 history，要么在调用处构造 `attempts` 列表包含历次报告（从 `_retry_history` 反查或落盘时保留）。同时统一字段名（`retry_history` vs `_retry_history`）。
- **Evidence**: verifier.py L433-L453 控制流 + merge_retry_report L252-L285。

### HV-002 · fact_checker.py:379 data_year 时效检测用 date.today() 非 UTC
- **File**: `script-verifier/fact_checker.py` L376-L382
- **Severity**: HIGH
- **Kind**: route-fix
- **Finding**: `_judge_claim` DATA 分支用 `from datetime import date; age = date.today().year - int(claim["data_year"])` 计算数据年龄。但本文件其他所有时间戳均使用 `datetime.now(timezone.utc).isoformat()` (L192, L622, L647)。混用本地时区与 UTC 导致：① 跨年边界 (12-31/01-01) 不同机器给出不同过期判定；② 单元测试无法固定时钟注入；③ 与报告中 `verified_at` UTC 时间戳不一致，审计追溯困难。

- **Evidence**: fact_checker.py L376-L382 vs L192/L622/L647。

### MV-002 · fact_checker.py _check_corroboration 关键实体正则过宽
- **File**: `script-verifier/fact_checker.py` L465
- **Severity**: MEDIUM
- **Kind**: route-fix
- **Finding**: `key_entities = re.findall(r'[\d,]+\.?\d*|[《〈][^》〉]+[》〉]|[A-Z一-鿿]{2,6}', claim_text)` 中 `[\d,]+\.?\d*` 匹配任意数字串包括 "0"、"1"、年份 "2026" 等噪声；`[A-Z一-鿿]{2,6}` 会把 "我们"、"他们"、"但是" 等高频双字词当作关键实体。当 claim_text 缺少强特征实体时，这些噪声 entity 在 search_result 里高概率命中 → 半数阈值轻易达成 → 虚假 VERIFIED。
- **Impact**: 误放行 hallucinated claims；与 HV-001 叠加会让错误验证结果流入修复闭环。
- **Remediation**: 过滤长度≤2 的纯 CJK entity；数字 entity 要求 ≥3 位或带千分位/小数；增加停用词表。
- **Evidence**: fact_checker.py L462-L475。

## LOW Findings

### LV-001 · claim_extractor.py cn2num 不支持「零」作占位符
- **File**: `script-verifier/claim_extractor.py` L134-L159
- **Severity**: LOW
- **Kind**: route-fix
- **Finding**: `cn2num("三百零五")` 解析过程：遇到 '零' 时 ch in _CN_DIGIT → number=0；接着 '五' → number=5；结束 total+section+number=300+0+5=305 ✓。但 `"三百零"` → number=0 → result=300+0+0=300，语义应是 300 但用户可能输入不完整。更严重：`"一千零二十三"` 正确得 1023，但 `"一千零零三"` 得 1003（第二个零覆盖 number=0 后再遇 '三' → number=3），不符合中文数字规范。
- **Impact**: 边缘 case 数值转换偏差；对验证置信度影响小（±50% 容差通常能吸收）。
- **Remediation**: 加零占位处理：遇到 '零' 时不立即置 number=0，而是标记下一位应跳过的数量级。
- **Evidence**: claim_extractor.py L134-L159。

### LV-002 · verifier.py _count_words 正则把 emoji 计为 0 词但 CJK 标点区间含歧义
- **File**: `script-verifier/verifier.py` L55
- **Severity**: LOW
- **Kind**: route-fix
- **Finding**: `re.findall(r'[一-鿿]|[A-Za-z]+|\d+', text)` 中 `[一-鿿]` 是 CJK Unified Ideographs 主块 (U+4E00-U+9FFF)，不含扩展 A/B/C/D 及兼容汉字。罕见字（如 "𠀀" U+20000）不被计入词数。对常规脚本影响极小，但对古文/方言内容 word count 偏低。
- **Impact**: total_words 统计轻微偏差；不影响验证逻辑。
- **Remediation**: 如需精确，扩至 `[一-鿿㐀-䶿...]` 或用 unicodedata.category 判断 Lo。低优。
- **Evidence**: verifier.py L49-L55。

### LV-003 · hot-scanner.py _dedup_queries Jaccard 阈值 0.6 对短查询过于激进
- **File**: `topic-pool/hot-scanner.py` L399-L421
- **Severity**: LOW
- **Kind**: route-fix
- **Finding**: 短查询如 "房价 调控" (2 token) 与 "房价 调控 新政" (3 token) 的 Jaccard = 2/3 ≈ 0.67 > 0.6 → 被去重保留更长者。这本身是期望行为，但当两个查询意图不同（前者看历史政策，后者看最新动态）时合并会丢失细粒度扫描机会。
- **Impact**: 轻微召回损失；可通过调整阈值或加 min-tokens 守卫缓解。
- **Remediation**: 短查询 (<3 token) 提高阈值到 0.8；或对 focus/category 不同的查询豁免去重。
- **Evidence**: hot-scanner.py L399-L421。
