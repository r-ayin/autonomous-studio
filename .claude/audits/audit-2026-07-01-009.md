## Findings

### AE-H-001 [HIGH] resume-checkpoint.py 固件硬编码 `E:/x-tool` Windows 路径 — Linux 部署下自主引擎完全失效

- **文件**: `.claude/hooks/resume-checkpoint.py`
- **行号**: 121, 124, 135
- **类型**: route-fix
- **finding**: `build_engine_firmware()` 返回的字符串中三处硬编码 `E:/x-tool/.claude/...` 路径：
  - L121: "读 `E:/x-tool/.claude/decision-log.jsonl` 最后 5 行"
  - L124: "读 `E:/x-tool/.claude/memory/autonomous-state.md`"
  - L135: "尝试 `wsl -d Ubuntu -- sudo service cron start`"（WSL-only 命令）
  
  此函数每次 SessionStart 强制注入到 Claude 系统提示。当引擎部署在 Linux（非 WSL）时：
  1. 注入的路径不存在 → Claude 读文件失败 → 引擎无法感知当前状态
  2. WSL 命令不存在 → 自检报错 → 引擎误判 L6 失活
  3. 整个 v2.2 冷启动协议依赖这些路径正确 → 链路断在第一步
  
  这与 x-tool audit-2026-07-01-008 XT-H-001 完全同构（x-tool 已修 commit ba9d4f3）。
- **remediation**: 用 `{PROJECT_DIR}` f-string 替换所有硬编码路径；WSL 命令改为条件分支（检测 sys.platform == 'win32' 才注入 WSL 指令，Linux 下注入 systemd/cron 等效指令）。
- **证据**: Read 文件 L121/L124/L135 直接可见硬编码字符串。

### AE-H-002 [HIGH] save-checkpoint.py 引用未导入 `Path` + audit log 路径错误 — checkpoint 静默残缺

- **文件**: `.claude/hooks/save-checkpoint.py`
- **行号**: 137, 80-82
- **类型**: route-fix
- **finding**: 
  1. **L137 `find_task_files()` 使用 `Path(PROJECT_DIR)` 但 `from pathlib import Path` 在 L215（main 块内）**。当 `build_checkpoint()` 在主流程调用 `collect_*` 系列函数时若间接调用 find_task_files（实际目前 build_checkpoint 没调它，但它是公开函数，任何扩展调用都会 NameError）。更关键的是：**import 在 main 块内意味着被 import 为模块时 Path 永远不可用**。
  2. **L80-82 `collect_recent_activity()` 读 `~/.claude/audit/audit.jsonl`**——这是全局 Claude audit 路径，不是引擎自己的 decision-log.jsonl。当引擎以 hook 形式运行时，Claude Code 的全局 audit.jsonl 格式与引擎期望的 `_event`/`session_id` 字段不匹配 → activities 永远返回空列表 → checkpoint 的 recent_activity 字段恒为 [] → resume-checkpoint 恢复时丢失最近对话上下文。
- **remediation**: 
  1. 把 `from pathlib import Path` 移到文件顶部 imports 区。
  2. `collect_recent_activity()` 改为读 `PROJECT_DIR/.claude/decision-log.jsonl`（引擎自己的日志），字段名对齐 decision-observer.py 的输出（type/session_id/prompt_preview）。
- **证据**: Read 文件 L137 可见 `Path(PROJECT_DIR)`；L215 才 import Path。L80-82 可见 `os.path.expanduser("~"), ".claude", "audit", "audit.jsonl"`。

### AE-M-001 [MEDIUM] watchdog.sh PROJECT_DIR 硬编码 `/mnt/e/x-tool` — Linux 原生部署下 L6 监控死代码

- **文件**: `.claude/watchdog.sh`
- **行号**: 15
- **类型**: route-fix
- **finding**: `PROJECT_DIR="/mnt/e/x-tool"` 硬编码 WSL 挂载路径。当引擎部署在 Linux 原生环境（如本 DevBox `/home/admin/workspace/autonomous-engine`）时：
  - 所有 `$PROJECT_DIR/.claude/...` 路径都不存在
  - L31 `[ ! -f "$LATEST" ]` 立即 exit 0
  - L6 看门狗变成 no-op，失去外部监控能力
  
  与 x-tool audit-2026-07-01-008 XT-M-004 完全同构（x-tool 已修 commit 892e4a0）。
- **remediation**: `PROJECT_DIR="$(cd "$(dirname "$0")/../.." && pwd)"` 相对脚本位置推断；或读环境变量 `CLAUDE_PROJECT_DIR` fallback。
- **证据**: Read 文件 L15 直接可见硬编码。

### AE-M-002 [MEDIUM] notify-phone.py send() 返回值未被消费 — 通知失败静默丢失

- **文件**: `.claude/hooks/notify-phone.py`
- **行号**: 206
- **类型**: route-fix
- **finding**: `main()` L206 调用 `send(title, message, reason, priority)` 但不捕获返回值。`send()` 返回 bool 表示是否成功送达，但调用方丢弃了。后果：
  - confirm 级别通知（需要用户确认）失败时无任何反馈
  - 无法在日志/状态文件中记录通知失败
  - 用户以为通知已发但实际未收到 → 错过关键确认请求
  
  与 x-tool audit-2026-07-01-008 XT-M-003 同构（x-tool 已修）。
- **remediation**: `success = send(...)`; if not success and reason == "confirm": 写 stderr 警告 + 在 .claude/.last_phone_notify_failed 标记。
- **证据**: Read 文件 L206 `send(title, message, reason, priority)` 无赋值。

### AE-M-003 [MEDIUM] decision-observer.py handle_stop 反向读日志 chunk 拼接 off-by-one

- **文件**: `.claude/hooks/decision-observer.py`
- **行号**: 263-288
- **类型**: route-fix
- **finding**: 反向读取 LOG_FILE 最后 20 行的算法有 chunk 边界拼接 bug：
  ```python
  while pos > 0 and lines_found < 20:
      read_size = min(chunk_size, pos)
      pos -= read_size
      f.seek(pos)
      buf = f.read(read_size) + buf   # ← 新 chunk 拼到前面
      lines = buf.split(b"\n")
      needed = min(20, len(lines))
      buf = b"\n".join(lines[-needed:])  # ← 只保留后 needed 段
      lines_found = len([l for l in lines[-needed:] if l.strip()])
  ```
  问题：当某行跨越两个 chunk 边界时，第一次读到该行的后半段（不含换行符），第二次读到前半段。`buf = f.read(read_size) + buf` 正确拼接了字节序，但 `lines[-needed:]` 截断时可能把跨边界行切成两段残片，导致 JSON parse 失败。此外 `lines_found` 计数基于截断后的子集，可能提前退出循环。
  
  实际影响：active_project 提取偶尔拿到 "unknown" → 注入上下文的项目提示缺失 → 引擎跟进建议偏离当前项目。不是致命 bug 但是间歇性失灵。
  
  与 x-tool audit-2026-07-01-008 XT-H-002 同构（x-tool 已修 commit 68cf537）。
- **remediation**: 简化为一次性 tail-read：seek 到 max(0, filesize - 8192)，read 剩余全部，splitlines，取最后 20 行。8KB 足够覆盖 20 行 JSONL（每行通常 < 300B）。
- **证据**: Read 文件 L263-288 算法逻辑可见。

### AE-L-001 [LOW] protocol-check.py bootstrap 调用传 dummy 路径 — 隐式约定脆弱

- **文件**: `.claude/hooks/protocol-check.py`
- **行号**: 70
- **类型**: route-fix（低优先级）
- **finding**: `subprocess.run([sys.executable, BOOTSTRAP_SCRIPT, os.path.join(project_dir, "dummy")], ...)` 传了一个虚构的 `dummy` 文件路径给 bootstrap.py。这依赖 bootstrap.py 内部对参数做 dirname 提取的隐式约定。如果 bootstrap.py 改签名或校验参数存在性，自举立刻失败。
- **remediation**: 传真实 project_dir 作为参数，或在 bootstrap.py 入口显式支持 `--project-dir` flag。
- **证据**: Read 文件 L70 可见 `"dummy"` 字面量。

## 结构性债务（不派生 fix，写入 structural-debt.md）

### AE-S-001 [STRUCTURAL] 固件注入与 Skill/Skill-prompt 三重冗余 — 版本漂移风险

- **涉及文件**: resume-checkpoint.py (firmware)、SKILL.md、decision-agent-prompt.md
- **描述**: 引擎的核心行为规则（信心阈值、硬限制、冷却机制、检查点保护流程）在三处独立维护：
  1. `resume-checkpoint.py` 的 `build_engine_firmware()` 字符串（SessionStart 注入）
  2. `SKILL.md` 的执行流程描述（调度器激活时读）
  3. `decision-agent-prompt.md` 的子代理操作手册（spawn 后读）
  
  三者内容高度重叠但不完全一致（例如 SKILL.md 说 L2 每 7min，ARCHITECTURE.md 说 L2 每 30min；firmware 说 L2 每 7min）。修改一处极易遗忘同步其他两处。v2.2 升级已经出现过这种漂移（见 autonomous-state.md 中 L2 间隔从 30min→7min 的历史）。
  
  这不是一个可以 1-3 文件修复的问题，需要架构层面重新设计「单一事实源」机制（例如 firmware 从 YAML/JSON 配置生成，Skill 和 prompt 都引用同一份配置）。
- **建议**: 列入 structural-debt.md，等用户授权后做方向更新。
