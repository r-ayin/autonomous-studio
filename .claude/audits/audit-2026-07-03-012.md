## Findings

### SCRIPTS-M01: parallel-dispatch.sh — TASK_ID unsanitized in branch names and paths
- **File**: `/home/admin/workspace/scripts/parallel-dispatch.sh`
- **Line**: 55-56
- **Severity**: medium
- **Finding**: `TASK_ID` is extracted from `parallel-plan.json` via python and used directly in git branch names (`parallel/${TASK_ID}`) and filesystem paths (`${WORKTREE_BASE}/${TASK_ID}`). If a task ID contains shell metacharacters, path separators, or git-ref-invalid characters (e.g., `../evil`, `foo bar`, `~`), this creates path traversal or command injection vectors. Unlike opt-worktree.sh which has `_validate_commit_file_path()`, parallel-dispatch.sh has no input validation on task IDs.
- **Remediation**: Add a `validate_task_id()` function that rejects IDs containing `/`, `..`, spaces, or non-alphanumeric characters (except `-` and `_`). Apply before line 55.
- **Kind**: route-fix

### SCRIPTS-M02: parallel-merge.sh — `git reset --hard HEAD@{1}` fallback can destroy uncommitted work
- **File**: `/home/admin/workspace/scripts/parallel-merge.sh`
- **Line**: 78
- **Severity**: medium
- **Finding**: When integration tests fail after merge, the rollback logic is `git merge --abort 2>/dev/null || git reset --hard HEAD@{1} 2>/dev/null || true`. If `merge --abort` fails (e.g., merge was already committed or abort itself errors), `reset --hard HEAD@{1}` will **destroy any uncommitted changes** in the working directory. The `|| true` at the end means this destructive operation's failure is also silently swallowed. In a parallel-build context where multiple branches may be tested sequentially, this could wipe legitimate WIP.
- **Remediation**: Replace `git reset --hard HEAD@{1}` with a safer two-step: first check if there are uncommitted changes (`git status --porcelain`), and if so, stash them before reset. Or use `git revert HEAD` instead of hard reset. At minimum, log a warning before the hard reset.
- **Kind**: route-fix

### SCRIPTS-L01: verify-pattern.sh — GATE-0 uses string interpolation in python -c (injection vector)
- **File**: `/home/admin/workspace/scripts/verify-pattern.sh`
- **Line**: 21
- **Severity**: low
- **Finding**: `python3 -c "import json,sys; json.load(open('$CAL'))"` interpolates `$CAL` directly into a Python string literal. If the project path contains single quotes or backslashes, this breaks the Python syntax or potentially allows code injection. The same pattern appears in studio-precheck.sh lines 19-24 and 43-48. While these paths are typically controlled by the engine (not user input), defense-in-depth argues for passing via argv.
- **Remediation**: Use `python3 -c 'import json,sys; json.load(open(sys.argv[1]))' "$CAL"` to pass the path as an argument instead of interpolating.
- **Kind**: route-fix

### SCRIPTS-L02: studio-precheck.sh — bare except clauses swallow all errors silently
- **File**: `/home/admin/workspace/scripts/studio-precheck.sh`
- **Lines**: 19-24, 43-48
- **Severity**: low
- **Finding**: Multiple inline Python snippets use bare `except: print(0)` which catches `KeyboardInterrupt`, `SystemExit`, and `MemoryError` in addition to expected JSON parse errors. More importantly, if the Python invocation itself fails (e.g., python3 not found), the bash variable captures empty string and `${CONSECUTIVE:-0}` defaults to 0, making the precheck always pass when it should arguably fail-safe.
- **Remediation**: Use `except (json.JSONDecodeError, FileNotFoundError, KeyError):` for specificity. Consider failing closed (returning "skip:error") when calibration/status files exist but can't be parsed.
- **Kind**: route-fix

### SCRIPTS-L03: autonomous-loop.sh — PROMPT variable contains unescaped single quotes risk
- **File**: `/home/admin/workspace/scripts/autonomous-loop.sh`
- **Lines**: 35-57
- **Severity**: low
- **Finding**: The `PROMPT` variable is defined with single-quote delimiters (`PROMPT='...'`) and passed to `claude -p "$PROMPT"`. If the prompt text ever needs to contain a literal single quote (e.g., in English contractions or code examples), the current heredoc-style definition would break. Currently safe because the prompt content doesn't contain single quotes, but fragile against future edits. The `$WORKSPACE` interpolation on line 45 inside the single-quoted string actually works because bash concatenates adjacent strings, but this is subtle and error-prone.
- **Remediation**: Convert to a heredoc (`PROMPT=$(cat <<'EOF' ... EOF)`) for clarity and safety, or use double quotes with proper escaping.
- **Kind**: route-fix

### SCRIPTS-L04: distill-patterns.py — load_json_robust swallows all exceptions including IOError
- **File**: `/home/admin/workspace/scripts/distill-patterns.py`
- **Lines**: 37-43
- **Severity**: low
- **Finding**: `load_json_robust` catches all `Exception` types, meaning permission errors, disk full, and encoding errors are silently treated as "missing file". This can mask real infrastructure problems (e.g., NFS mount issues, permission changes) that should surface as alerts rather than being silently recovered.
- **Remediation**: Narrow to `except (json.JSONDecodeError, FileNotFoundError, UnicodeDecodeError):` and let IOErrors propagate.
- **Kind**: route-fix

### SCRIPTS-I01: opt-worktree.sh — redundant `mkdir -p "$target"` before worktree add
- **File**: `/home/admin/workspace/scripts/opt-worktree.sh`
- **Lines**: 370, 394
- **Severity**: info
- **Finding**: `mkdir -p "$target"` is called before `git worktree add`, but `git worktree add` itself creates the directory. The pre-mkdir is needed only as a fallback for the error-path `rmdir` cleanup, but creates a TOCTOU window where another process could populate the directory between mkdir and worktree add. Not exploitable in practice (single-user engine), but worth noting.
- **Remediation**: Document why mkdir precedes worktree add (cleanup semantics), or restructure to let worktree add create the dir and handle the no-dir-to-rmdir case in the error path.
- **Kind**: route-fix

### SCRIPTS-I02: parallel-dispatch.sh — node_modules symlink assumes host has node_modules
- **File**: `/home/admin/workspace/scripts/parallel-dispatch.sh`
- **Lines**: 70-72
- **Severity**: info
- **Finding**: The node_modules symlink optimization (`ln -s ... node_modules`) silently fails with `|| true` if the source doesn't exist or symlinks aren't supported. This is correct behavior (graceful degradation), but the silent failure means agents spawned in these worktrees may waste time running `npm install` without understanding why the optimization didn't apply. A debug-level log would help troubleshooting.
- **Remediation**: Add an else-branch log message when the symlink is skipped.
- **Kind**: route-fix
