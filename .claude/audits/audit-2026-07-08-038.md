## Finding 详情

### DW-CORE-H-001: BFF call log writes full response.data without redaction

**位置**: `/home/admin/.claude/skills/dataworks-dev-assistant/core/bff_client.py:386-407`

```python
def _log(self, path, method, params, data, json_body, response, cost_ms):
    log_entry = {
        ...
        "response": {
            "code": response.get("code"),
            "data": response.get("data"),     # ← 完整响应体，可能含用户邮箱/工号/表内容
            "message": response.get("message"),
            "requestId": response.get("requestId")
        }
    }
    with open(self.log_file, "a", encoding="utf-8") as f:
        f.write(json.dumps(log_entry, ensure_ascii=False) + "\n")
```

**证据**: DataWorks API 如 `currentUser`/`searchTables`/`getExecutorJobResult4Ida` 的 response.data 字段包含用户 baseId/displayName/account/email、表元数据、甚至查询结果行。日志文件 `~/.dataworks/logs/dw_bff_calls.log` 权限默认 644，多租户机器上其他用户可读。

**影响**: PII/凭证泄露到本地日志；合规风险（GDPR/个人信息保护法）。

**修复 sketch**: 在 `_log()` 入口对 response.data 做脱敏：dict/list 递归替换已知敏感 key（email/account/displayName/baseId/token）为 `"***"`；或对 data 字段只记录 type+length 不记值。

### DW-SQL-M-001: WITH-prefix CTE can bypass write-operation two-phase confirm

**位置**: `modules/sql-execution/scripts/execute_sql.py:38-46`

```python
_READONLY_PREFIXES = re.compile(
    r"^\s*(SELECT|SHOW|DESCRIBE|DESC|EXPLAIN|WITH)\b",
    re.IGNORECASE,
)

def _is_readonly_sql(sql):
    return bool(_READONLY_PREFIXES.match(sql))
```

**证据**: SQL `WITH cte AS (SELECT ...) INSERT INTO target SELECT * FROM cte` 以 WITH 开头被判定为 readonly，跳过 `prepare_write_sql` 两阶段确认直接执行。autonomous-studio 主引擎在 commit e37253f (2026-07-07) 修复了完全相同的 bug（opt-sql-execution-1783292031），说明这是跨项目的共性模式缺陷。

**影响**: 写操作未经用户确认即执行；违反 SKILL.md 声明的"所有写操作必须先请求用户确认"契约。

**修复 sketch**: 复用主引擎 e37253f 的方案——剥字符串字面量和注释后扫 INSERT/UPDATE/DELETE/MERGE/DROP/CREATE/ALTER/TRUNCATE 关键词；命中则判为写。或更简：WITH 开头时 fallback 检查后续是否含写关键词。
