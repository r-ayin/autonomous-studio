## Findings

### M-ANDROID-001 · android liveBus WebSocket has no max message size cap
- **file**: android/index.html
- **line**: 600 (`ws.onmessage = (ev) => { let m; try { m = JSON.parse(ev.data); } catch { return; } emit(m); }`)
- **severity**: medium
- **kind**: route-fix
- **finding**: Unlike relay_bridge.py which enforces MAX_FRAME_SIZE=64KB server-side, the android client's WebSocket consumer accepts arbitrarily large messages before JSON.parse. A malicious or buggy server (or MITM on ws:// fallback path at line 594) can send a multi-MB frame that freezes the UI thread during JSON.parse or exhausts mobile memory. The web/ dashboard doesn't have this issue because it uses api.js get/post with AbortController timeouts and doesn't consume /ws/live directly.
- **remediation**: Before JSON.parse, check `ev.data.length` (string) or `ev.data.byteLength` (Blob/ArrayBuffer) against a 256KB threshold; drop + console.warn if exceeded. Alternatively use TextDecoder stream with early-abort. Also consider adding an AbortController-equivalent idle timeout to liveBus (currently only exponential-backoff reconnect at line 602, no inactivity kick).

### L-RELAY-001 · _process_request swallows Response construction failure silently
- **file**: scripts/ecs_relay/relay_bridge.py
- **line**: 122-128
- **severity**: low
- **kind**: route-fix
- **finding**: The `except Exception:` fallback returns `(403, [], b"invalid token\n")` tuple for older websockets versions, but if BOTH paths fail (e.g., websockets API change in future version), the exception propagates to websockets.serve which may log stack trace without failing closed. Current code is correct for websockets 12.x/13.x but brittle across versions.
- **remediation**: Add explicit version check or wrap outer try/except in _process_request itself that always returns a valid rejection. Log the inner exception in the fallback branch so version drift is detectable via fail2ban/journalctl.

### L-ANDROID-002 · liveBus retry counter never resets on successful send-after-reconnect
- **file**: android/index.html
- **line**: 597-602
- **severity**: low
- **kind**: route-fix
- **finding**: `retry` is reset to 0 in `ws.onopen` (line 597), which is correct. However, if the connection opens then immediately closes (server accepts handshake then drops due to policy), each cycle increments retry without ever delivering data. After 9 such cycles (~8 min total due to exponential backoff capping at 30s), liveBus permanently stops trying (`if (retry > 8) return`). User sees "重连中" forever with no recovery path except page reload.
- **remediation**: Distinguish "connected-but-dropped" from "never-connected". Reset retry only after first successful message exchange (not just open). Or add user-visible "reconnect now" button that calls `liveBus.start()` with retry=0.

### I-WEB-001 · web/api.js SLOW_PATHS only covers /recommend; other LLM endpoints share 8s timeout
- **file**: web/api.js
- **line**: 44-48
- **severity**: info
- **kind**: informational
- **finding**: `/chat-log` POST (which triggers LLM response generation when user sends chat via WS fallback), `/wiki/build` POST (LLM compilation), and `/profile` GET (may invoke embedding) all use default 8s timeout. Only `/recommend` gets 60s. In practice these endpoints are fast (<2s) because they read cached state, but if backend adds synchronous LLM calls to any of them, 8s will be too tight.
- **remediation**: Document the timeout contract in api.js header comment. When backend adds sync LLM to new endpoint, update SLOW_PATHS proactively. No code change needed now.
