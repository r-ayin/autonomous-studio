## Executive Summary

wechat-main 是微信公众号深度长文的生产管线：六阶段流水线（竞品→简报→分析→persona 重写→QA 事实核查→输出），由 `pipeline.py` 确定性编排 + `pipeline-gate.sh` 硬门禁 + `script-verifier/` 做事实核查（claim 提取→WebSearch→LLM-free 判定→FALSIFIED 闭环）。整体设计扎实（无 shell=True 的 subprocess、白名单工具调度、QQ/SMTP 默认 TLS、API key 走 env、错误体脱敏）。

但深度审计发现 **1 HIGH / 7 MEDIUM / 5 LOW / 3 INFO** 共 16 个问题，集中在三处：

1. **LLM 输出 → shell 边界**：Phase 3 子 agent 命名的文章文件名（含中文标题）经 `__ARTICLE__` 占位符无引号注入 `bash STEP.cmd`——标题含空格即断参，含 `;`/`$()` 即命令注入。
2. **QA 门禁被静默绕过**：`pipeline-gate.sh:213` 对已绝对路径的 find 输出再前缀 `$WECHAT_ROOT`，致 3 个 advisory checker 收到不存在的路径、python 报错、门禁不打印任何告警即放行；叠加 `fact_checker._check_value_range` 在搜索结果无数值时默认通过，DATA 声明可无数值佐证即 VERIFIED。
3. **持久化原子性 + 日志脱敏**：state/报告/搜索结果全为非原子 `write_text`，崩溃半写 → gate 的 grep 兜底可能读到残缺字段；qq_push 把外部 API 响应体（含 token 邻近字段）当"本地 JSON"误打到 stderr。

修复路线：HIGH/MEDIUM 多为 1-3 文件 route-fix（quote 路径、改 `full="$f"`、原子写、收紧 `_check_value_range`、脱敏异常体）；INFO 的跨项目 .env 读取为 direction-shift。

## Cross-Module Data Flow Notes

1. **LLM→shell 边界**（WM-PIP-01）：Phase 3 子 agent 产出文章 → 文件名含 LLM 标题 → `_resolve_article` glob 命中 → `cmd_next` 裸 replace `__ARTICLE__` → agent-loop `bash STEP.cmd`。无任何 quoting/sanitize。这是本审计最高优先级：一个良性含空格标题就能让 QA extract 静默失败（FALSIFIED=0），一个恶意标题可 RCE。
2. **QA 门禁静默放行链**（WM-GATE-01 + WM-FC-01 + WM-VER-02）：advisory checker 路径双前缀失效 → 不告警；DATA 无数值默认 VERIFIED；报告非原子写 + grep 兜底读残缺。三层叠加可使"数据声明无任何数值佐证"的文章通过 Phase 3.5 QA 门禁。CLAUDE.md 规则 6（FALSIFIED=0 才算通过）的刚性被悄然削弱。
3. **持久化原子性**（WM-PIP-02/WM-BOCHA-01/WM-VER-02）：state/搜索结果/QA 报告三类 JSON 全非原子写，崩溃窗口期内 gate 可能读到残缺字段。autonomous-studio 自身审计（audit-2026-07-03-017 M-009/M-010）已确立 tmp+os.replace+fsync 为标准模式，wechat-main 未跟进。
