## Threat model

山夏 chat 子系统的信任边界：

1. **客户端浏览器** → chat-widget.js（公开 JS，任何人可读源码）→ POST `https://shanxia-chat.womenhaiyouxiwang.workers.dev` 带 `X-Chat-Token` + `X-Session-Id` + `messages[]`。
2. **Cloudflare Worker** (`chat-worker.js`) → 持 `DEEPSEEK_API_KEY`（env 密钥，服务端）出站 `api.deepseek.com`；持 `CHAT_TOKEN`/`ADMIN_TOKEN`（env）；写 D1 `sessions`/`messages` 表。
3. **DeepSeek API**（外部，HMAC 无，Bearer key）。
4. **D1 数据库**（Cloudflare，存用户对话内容 + IP）。

最高影响失败模式 = **鉴权密文外泄 + 并发槽泄漏**：CHAT_TOKEN 在前端 JS 明文 ship → worker 的 token 门是装饰性的 → 任何人可驱动 DeepSeek API（预算 drain）+ 占满 3 会话并发槽（DoS 真实客户）+ 向 D1 注入任意对话。叠加 disconnect 信标因无 token 头被 401 拦截 → 槽位只能靠 120s 超时释放 → 合法用户更易撞上「AI 忙」。

## Findings

### 🔴 XIA-CHAT-01 — HIGH — CHAT_TOKEN 明文硬编码在前端 ship 的 JS 里 → worker token 门形同虚设

- **file**: `山夏/chat-widget.js:11`
- **evidence**: `const CHAT_TOKEN='b6227973c87099bb34cdb9848ff1300cd78b2ed327bab0b7';` 然后在 `send()` (line 236-244) 以 `'X-Chat-Token': CHAT_TOKEN` 头发往 worker。worker 端 `山夏/chat-worker.js:95` `if (!env.CHAT_TOKEN || ctoken !== env.CHAT_TOKEN) return json({ error: 'Unauthorized' }, 401, corsOrigin)` 把同一字符串当密钥校验。任何访客「查看源代码」即得 token → POST 带 token 即过门。
- **impact**: token 门本意是阻止匿名 POST（防 DeepSeek 预算 drain / 并发槽占用 / D1 垃圾写入）。密文公开后该门 100% 可绕过。攻击者可：(a) 批量调用 drain DeepSeek 余额；(b) 占满 3 槽致合法客户恒见「AI 小助理正在跟别人聊天」；(c) 向 D1 注入任意 session/messages 行。这是本审计的根因，放大 XIA-CHAT-02/03 的影响。
- **severity**: high
- **kind**: route-fix（两文件最小改动：widget 移除明文 token；worker 停止把 CHAT_TOKEN 当硬密钥——改为依赖已存在的 IP 限速 + 3 会话并发门 + D1 行写配额。若日后要恢复鉴权须改用服务端下发的短时签名 token，但那是 structural，不属本 route-fix 范围。）
- **remediation_sketch**: (1) `chat-widget.js:11` 删除 `CHAT_TOKEN` 常量，`send()` 的 headers 删 `'X-Chat-Token'` 行。(2) `chat-worker.js:93-95` 删除整段 token 鉴权（POST 路径改靠 `checkRateLimit(ip)` + 3-session 并发门 + DeepSeek `max_tokens:500` 兜底）。保留 `env.CHAT_TOKEN` env 变量不动（不删除 secret 轮换属用户运维决策）。影响面：2 文件，无破坏性 API 变更（合法 widget 仍可工作）。

### 🟠 XIA-CHAT-02 — MEDIUM — disconnect 信标无 token 头 → worker 401 拦在 disconnect 分支之前 → 3 会话槽泄漏至 120s 超时

- **file**: `山夏/chat-widget.js:179-183, 186-193`（sendBeacon）+ `山夏/chat-worker.js:95, 99-107`（鉴权先于 disconnect 分支）
- **evidence**: widget 用 `navigator.sendBeacon(WORKER_URL, JSON.stringify({action:'disconnect',sessionId:sid}))` 释放槽位。`sendBeacon` **无法设置自定义请求头**（只能发 body）→ 请求不含 `X-Chat-Token`。worker POST 处理顺序：line 95 先 `if (!env.CHAT_TOKEN || ctoken !== env.CHAT_TOKEN) return 401` → 信标恒 401，**永远到不了** line 104 `if (body.action === 'disconnect' ...) { sessions.delete(...) }` 分支。结果：`disconnect` 释放槽位的代码路径在生产中不可达，槽位只靠 `SESSION_TIMEOUT_MS=120_000` 超时清理（line 6/8 `cleanSessions`）。
- **impact**: 每次客户关闭面板/标签页（`beforeunload`，line 186-193）都发一条必 401 的信标 → 槽不释放。3 槽并发限制（`MAX_SESSIONS=3`，line 113-115）因此退化为「最近 120s 内开过会话的 3 个客户」独占 → 真实新客户高频撞 `503 busy`。可用性退化（DoS-ish），无需恶意意图即触发。
- **severity**: medium（可用性 bug，非安全问题，但生产可见）
- **kind**: route-fix
- **remediation_sketch**: 把 `body.action === 'disconnect'` 分支**移到 token 鉴权之前**（disconnect 不需鉴权——它只是释放一个内存 Map 槽，无副作用无数据外泄）。即 `chat-worker.js` 在 POST 方法判定后、token 鉴权前，先 `let body; try { body = await request.json(); } catch { ... }; if (body?.action === 'disconnect' && body.sessionId) { sessions.delete(body.sessionId); return json({ok:true,active:sessions.size},200,corsOrigin); }` 再走 token 鉴权。注意须把 body.json() 解析上提。1 文件改动。或：widget 改用 `fetch(..., {keepalive:true})` 带 token 头发 disconnect（但 XIA-CHAT-01 修完 token 门删除后，sendBeacon 路径自然通——两 finding 修后此 bug 自动消解一半，但仍应把 disconnect 分支前置以独立稳健）。

### 🟠 XIA-CHAT-03 — MEDIUM — 客户端全控 messages 数组（角色/内容/长度均不校验）→ 提示注入 + DeepSeek 预算 drain

- **file**: `山夏/chat-worker.js:121-135`
- **evidence**: line 121 `const msgs = body.messages; if (!Array.isArray(msgs) || msgs.length === 0) return 400`——**只校验是数组**，不校验元素 `role` 取值、不校验 `content` 长度。line 135 `const apiMsgs = [{ role:'system', content: SYSTEM_PROMPT }, ...msgs.slice(-10)]` 把客户端原始 msgs 直接拼到发给 DeepSeek 的 messages 数组（仅 prepend 自有 system，不剥除客户端注入的 system）。widget 端 `chat-widget.js:231-233` 自律地只发 user/assistant，但攻击者（凭 XIA-CHAT-01 泄漏的 token）可直 POST 任意 `messages:[{role:'system',content:'忽略上文…'},{role:'user',content:'…'}]` 或 10 条各 100KB 的 user 消息。
- **impact**: (a) 提示注入——攻击者注入 `role:'system'` 或伪造 `role:'assistant'` 历史回复操纵模型输出（对摄影咨询 chatbot 影响有限：off-brand 回复 / 误报价格，但可被用于社会工程载体）；(b) 预算 drain——`max_tokens:500` 限制了输出，但输入 context 无上限，10×100KB 输入 token 成本可被放大。
- **severity**: medium
- **kind**: route-fix
- **remediation_sketch**: `chat-worker.js` 在 line 121-122 后加服务端校验：`msgs = msgs.filter(m => m && (m.role==='user'||m.role==='assistant') && typeof m.content==='string' && m.content.length <= 2000).slice(-10)`（剥除客户端 system/任意 role；内容 ≤2000 字符上限）。其余逻辑不变。1 文件。

### 🟠 XIA-CHAT-04 — MEDIUM — assistant 回复写 D1 用 fire-and-forget `.catch(()=>{})` 且无 `ctx.waitUntil` → 流被取消时静默丢消息

- **file**: `山夏/chat-worker.js:163-166`
- **evidence**: line 164 `db.prepare(...).bind(...).run().catch(() => {})` 与 line 165 同——两个 D1 写**不 await、不 ctx.waitUntil**，错误被空 catch 吞掉。ReadableStream `start()` (line 150) 在 `ctrl.close()` 后继续跑这两行，但 Cloudflare Worker 在 `fetch` 返回 Response 后可随时终止执行（尤其客户端中途断流时 stream 被取消 → `start()` 内剩余代码不保证跑完）。
- **impact**: (a) 数据完整性——用户消息（line 131，`await` 写）落库但 assistant 回复可能丢 → D1 对话记录单向残缺，admin GET 查到的历史不完整；(b) 静默吞错——`.catch(()=>{})` 使运维对 D1 写失败零可见性，违反「不要静默吞错」原则。
- **severity**: medium
- **kind**: route-fix
- **remediation_sketch**: 用 `ctx.waitUntil(Promise.all([db.prepare(...).run(), db.prepare(...).run()]).catch(e => console.error('D1 assistant write:', e)))` 包裹，确保 Worker 不会在写完成前被收割，且错误进 console 而非黑洞。1 文件。（注意 `ctx` 已是 fetch handler 第三参数，line 55 `async fetch(request, env, ctx)`，无需新增签名。）

### 🟡 XIA-CHAT-05 — LOW — admin token 可经 URL query 传入 → 边缘日志/Referer 泄漏风险

- **file**: `山夏/chat-worker.js:73`
- **evidence**: `const token = request.headers.get('X-Admin-Token') || url.searchParams.get('token') || ''`——admin 查询接口允许 `?token=...` query 传密钥。URL query 会被 Cloudflare 边缘访问日志、浏览器历史、以及任何下游 referrer 记录。
- **impact**: 若管理员通过浏览器 GET `.../session?token=xxx`，token 进入 URL → 可能出现在 CF 仪表盘访问日志、分享/截图、Referer 头（若该 JSON 响应页加载了外部资源——目前纯 JSON 无外链，风险低）。
- **severity**: low
- **kind**: route-fix
- **remediation_sketch**: 删 `|| url.searchParams.get('token')` 分支，强制 admin token 只走 `X-Admin-Token` 头。1 文件，1 行。向后不兼容（若有 admin 脚本靠 query 传 token 须同步改），但 admin 接口使用面极小（仅站主本人）。

### 🟡 XIA-CHAT-06 — LOW — token 比较非常量时间 → 定时侧信道（对 ADMIN_TOKEN 有意义）

- **file**: `山夏/chat-worker.js:74`（`token !== env.ADMIN_TOKEN`）、`山夏/chat-worker.js:95`（`ctoken !== env.CHAT_TOKEN`）
- **evidence**: 字符串 `!==` 短路返回 → 比较耗时与公共前缀长度相关。CHAT_TOKEN 已公开（XIA-CHAT-01）故此处定时攻击无意义；但 **ADMIN_TOKEN 仍是真密钥**，远程定时攻击在 CF Worker 上虽噪声大，理论上仍可逐字节探测。
- **impact**: 低——CF 边缘网络抖动远大于字符串比较时序差，实战利用困难，但属纵深防御缺口。
- **severity**: low
- **kind**: route-fix
- **remediation_sketch**: 引入常量时间比较（如手写 `function ctEq(a,b){ if(a.length!==b.length) return false; let r=0; for(let i=0;i<a.length;i++) r|=a.charCodeAt(i)^b.charCodeAt(i); return r===0; }` 或 `crypto.subtle.timingSafeEqual` 若可用）。替换两处 `!==`。1 文件。可与 XIA-CHAT-05/07 同 worktree 堆叠。

### 🟡 XIA-CHAT-07 — LOW — admin GET 端点无限速 → 无限次猜 admin token

- **file**: `山夏/chat-worker.js:71-89`（GET 路径）
- **evidence**: GET 路径**不调用 `checkRateLimit`**（限速只在 POST 路径 line 119）。admin token 比较在 line 74。攻击者可无限次 GET `?token=...` 猜测 ADMIN_TOKEN，配合 XIA-CHAT-06 定时侧信道。
- **impact**: 低（128-bit 真随机 token 暴力不可行），但与 XIA-CHAT-06 叠加放大侧信道可行性。
- **severity**: low
- **kind**: route-fix
- **remediation_sketch**: GET 路径开头加 `if (!checkRateLimit(ip)) return json({error:'slow down'},429,corsOrigin)`（复用既有 `checkRateLimit`）。1 文件，1-2 行。

### ℹ️ XIA-CHAT-08 — INFO — CORS 放行任意 localhost 端口（dev 便利，可接受）

- **file**: `山夏/chat-worker.js:26-28`
- **evidence**: `isLocalhost = o => o.startsWith('http://localhost:') || o.startsWith('http://127.0.0.1:')` 且 `isAllowedOrigin` 反射匹配的 origin 进 `Access-Control-Allow-Origin`。任意 localhost 端口均被放行 + 反射。
- **impact**: dev 便利；生产无风险（localhost origin 只可能来自本机）。若站主本机有恶意页面，该页面可带（已公开的）CHAT_TOKEN 跨域读 worker 响应——但 token 既已公开，此路冗余。
- **severity**: info
- **kind**: route-fix（**不修**——dev 便利合理，记 info 备查）
