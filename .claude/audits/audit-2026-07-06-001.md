## Threat model

prod-deploy orchestrates production deployment: 变更单 → 流水线 → 构建 → 发布准入 → 分批部署 → 完成. It holds deploy gating authority (resume-next-batch decides whether to advance batches). Two trust boundaries:

1. **DEVOUT aone-agent-server** (`DEVOUT_SERVER_URL`) — receives auth token (`CODE_PRIVATE_TOKEN`/`DEVOUT_TOKEN`) and all task/event state. Operator-configured endpoint.
2. **Sunfire OpenAPI** (`api.x.alibaba-inc.com`) — HMAC-SHA1 signed with `SUNFIRE_SECRET_KEY`. Returns anomaly data that feeds the observation gate.
3. **`a1` CLI** — invoked via `execFile` (no shell) for pipeline/stage/job/task status. Argv-safe; no injection surface.

Highest-impact failure mode = **deploy-gate logic corruption**: if the observation time-gate or observation-result gate can be bypassed or corrupted, batches advance without real health verification → bad production push.

## Findings

### PD-TIME-01 — HIGH — observation time-gate uses `updated_at` as "batch success time" (corrupted by payload writes)

**Files**: `scripts/resume-next-batch.js:51`, `scripts/report-observation.js:57-59`, `scripts/report-event.js:179-232` (handleDeployBatch)

**Bug (logic, data-flow)**:
`resume-next-batch.js` computes the elapsed observation window as:
```js
const successTime = new Date(batchEvent.updated_at).getTime();   // :51
const elapsedMs = Date.now() - successTime;
const requiredMs = observeMinutes * 60 * 1000;
if (elapsedMs < requiredMs) { /* reject: "观察期未满" */ }
```
`report-observation.js:57-59` uses the same `updated_at` as the Sunfire query start time, with the comment: *"uses batch event's updated_at timestamp (when the batch became SUCCESS) as the observation start time."*

That assumption is **wrong** once observation starts. `report-observation.js:91` writes `observation_result` into the **same** batch event on every tick (`updateEvent(batchEvent, null, { observation_result })`). Any PUT to the event — including payload-only updates — re-sends `status: existingEvent.status` (see `event-client.js:206`) and writes a new payload. On a server that bumps `updated_at` on write (the standard REST behavior), `updated_at` is no longer "when the batch became SUCCESS"; it is "when the last observation tick wrote."

**Manifestation**:
- After the batch hits SUCCESS at T, observation ticks every 60 s, each bumping `updated_at` → T+60, T+120, … T+300.
- At T+300 (5 min, gate should pass), the agent calls `resume-next-batch`. It reads `updated_at ≈ T+300` (or T+240 if the last tick was 60 s ago). `elapsedMs = now - (T+240) ≈ 60 s < 300 s` → **gate fails: "剩余 4 分钟"**.
- The agent keeps polling. The gate only passes once observation **stops** for a full `observe_minutes`. Effective observation becomes ~2×`observe_minutes`, with misleading "remaining" messages; or, if the agent interleaves `report-observation` then `resume-next-batch` every tick (a natural loop), `elapsedMs ≈ 0` and the gate **never passes**.

This is a silent deploy-gate corruption: the time gate measures the wrong thing. It cannot under-block on its own (it over-blocks), but it erodes trust in the gate and trains operators to work around it, which is how under-blocking happens in practice.

**Even if** the server today does *not* bump `updated_at` on payload-only writes, the code's correctness depends on an undocumented server invariant. Any future server change to bump-on-write (or any other field that triggers a bump) silently breaks the gate. Defense-in-depth demands an explicit timestamp.

**Remediation sketch** (kind: route-fix, 3 files):
1. `report-event.js` `handleDeployBatch`: when `values.status === 'SUCCESS'` and the event is not already terminal, set `payload.success_at = Date.now()` in the update fields (and also stamp it on create-with-status). One-line addition in the SUCCESS branch.
2. `resume-next-batch.js:51`: `const successTime = batchEvent.payload?.success_at ?? new Date(batchEvent.updated_at).getTime();` (fallback preserves backward compat with in-flight batches).
3. `report-observation.js:57-59`: same fallback to `batchEvent.payload?.success_at` for `startTime` when `--start-time` omitted.

### PD-HANG-01 — MEDIUM — no timeout on outbound HTTP (Sunfire + DEVOUT fetch)

**Files**: `scripts/lib/sunfire-client.js:32-54` (`httpPost`), `scripts/lib/task-client.js:32-72`, `scripts/lib/event-client.js:32-58`

**Bug (availability)**:
- `sunfire-client.httpPost` uses `https.request` with **no `req.setTimeout()` / no socket timeout**. If Sunfire's dispatcher hangs (TCP stall, slowloris-style), `report-observation.js` hangs forever — the observation tick never returns, blocking batch progression with no error.
- `task-client`/`event-client` use global `fetch()` with **no `AbortController`/timeout**. A hung `DEVOUT_SERVER_URL` hangs every script (`report-event`, `resume-next-batch`, `complete-task`, `poll-build`, `poll-pre-check`).

Compare `poll-build.js:42-46` / `poll-pre-check.js:58-62`, which correctly set `timeout: 30000` on the `a1` `execFile` call. The HTTP clients lack the equivalent.

**Remediation sketch** (kind: route-fix, 3 lib files): add a 30 s `AbortController`-based timeout to each `fetch` in task-client/event-client (raise a typed error on abort); add `req.setTimeout(30000, () => req.destroy(new Error('Sunfire request timed out')))` in `sunfire-client.httpPost`. Surface the timeout as `skipped:true, reason:'…timed out'` in `querySunfireInsights` so observation degrades gracefully instead of hanging.

### PD-TOKEN-01 — MEDIUM — auth token sent to arbitrary host in `DEVOUT_SERVER_URL` (no allowlist)

**Files**: `scripts/lib/task-client.js:18-26`, `scripts/lib/event-client.js:22-30`

**Bug (credential disclosure / SSRF-adjacent)**:
`getHeaders()` attaches `X-Agent-Authorization: Code <CODE_PRIVATE_TOKEN>` (or `token: <DEVOUT_TOKEN>`) to **every** request, to whatever host `DEVOUT_SERVER_URL` resolves to. There is no hostname validation, no IP-literal block, no allowlist. If `DEVOUT_SERVER_URL` is misconfigured (e.g. a typo'd external host, a DNS-rebinding target, or a compromised env) the deploy token leaks to an attacker in an Authorization-style header, and the leak is silent (no log, no refuse).

This is directly analogous to the `generate-image` H02 finding (download_image hostname allowlist + IP literal block, audit-2026-07-04-013), which was merged. The same defense applies here for the token-bearing requests.

**Remediation sketch** (kind: route-fix, 2 files — shared `getBaseUrl`/`getHeaders`): validate the parsed `DEVOUT_SERVER_URL` hostname against an allowlist (`localhost`, `127.0.0.1`, `*.alibaba-inc.com`, and any explicit internal ranges the operator configures via `DEVOUT_ALLOWED_HOSTS`). Reject IP literals not in the allowlist. Only attach the token header when the host passes; otherwise throw a clear error at `getBaseUrl()` time so no request is ever sent with the token to an unknown host.

### PD-BATCH-01 — LOW — `--batch-index` not validated in `handleDeployBatch` → `batch_index: null` poisons event

**File**: `scripts/report-event.js:195`

**Bug (robustness)**:
```js
const batchIndex = parseInt(values['batch-index'], 10);   // NaN if missing/blank
```
`findBatchEvent` filters `payload.batch_index === batchIndex`. `NaN !== NaN`, so a missing `--batch-index` always misses → a new event is created with `batch_index: NaN`, which `JSON.stringify` serializes as `null`. Every later legitimate `findBatchEvent(taskId, <real N>)` then fails to match the poisoned event → duplicate events / lost updates for that batch. `resume-next-batch` and `report-observation` also call `findBatchEvent` and would not find the batch.

**Remediation sketch** (kind: route-fix, 1 file): after `parseInt`, guard `if (!Number.isInteger(batchIndex) || batchIndex < 0) { console.error('ERROR: --batch-index must be a non-negative integer'); process.exit(1); }`. Same for `batch-total`/`instances` if present.

### PD-CONCL-01 — LOW — `deriveConclusion` dedup assumes chronological array order (unspecified by Sunfire)

**File**: `scripts/lib/sunfire-client.js:165-185`

**Bug (correctness, safe-side-leaning)**:
`deriveConclusion` dedups by `insightRule` keeping the **last array occurrence** as "newest status":
```js
for (const c of checks) { if (c.insightRule) byRule.set(c.insightRule, c); }
```
The comment claims "last (newest) status per rule," but Sunfire's response order is unspecified. If Sunfire returns newest-first, "last" is the **oldest**. For a rule that was WARNING then RECOVERed, reversed order keeps the WARNING → false alarm (safe side). For a rule that RECOVERed then went WARNING again (still active), reversed order keeps the RECOVER → conclusion "passed" → **observation gate bypassed** (unsafe side, rare path). The same fragility applies to `resume-next-batch.js:71-77` which re-implements the dedup inline.

**Remediation sketch** (kind: route-fix, 1-2 files): before dedup, sort `checks` by `timestamp` descending (numeric) so "last in array" is genuinely newest; or track per-rule `anyWarning` + `lastStatusByTime`. Extract the dedup into a shared helper in `sunfire-client.js` and reuse from `resume-next-batch.js` (currently duplicated).

### PD-OBSERVATION-OVERWRITE-01 — LOW (observation) — terminal event payload is mutable, `observation_result` can be rewritten

**File**: `scripts/lib/event-client.js:193-200`

`updateEvent` explicitly allows payload-only writes on terminal events (comment: *"Allow payload-only updates on terminal events (e.g. observation_result after SUCCESS)"*). This is intentional (observation arrives after a batch is SUCCESS). But it means the `observation_result` that `resume-next-batch` reads is mutable after the fact — an agent (or anything with task-event PUT access) can overwrite `observation_result.checks = []` → `deriveConclusion` returns `'passed'` → bypasses the observation gate. Threat model is operator self-harm (the agent is the operator's own code), so low. Noting for completeness; no fix mandated unless multi-tenant trust is added later.

## What was checked and found clean

- **Shell injection**: `poll-build.js`/`poll-pre-check.js` use `execFile('a1', args)` (no shell). `pipeline-id`/`app-name`/`--task-id` flow as separate argv. Safe. ✓
- **`extractUrlFromTips`**: regex validates `^https?://` before returning; `&amp;` decoded. No `javascript:` / data-URI surface. ✓
- **Sunfire HMAC**: `secretKey` is used only as the HMAC key, never placed in the request body or URL. Signature covers all params. `timestamp` present (no nonce, but that's Sunfire's API design). ✓
- **`JSON.parse` of CLI args** (`check-items`, `resolved-strategy`, `payload`): throws → `process.exit(1)` on malformed input. No silent corruption; crash-only. Acceptable for operator-driven CLI. ✓
- **Idempotency**: `findEvent`/`findBatchEvent` before create; terminal-status guard prevents status regressions. ✓
- **Audit-log instrumentation (constraint B)**: deploy/release/CR/pipeline are sensitive paths. The skill's entire purpose is recording deploy progress events (status SUCCESS/FAILED, error_message, observation_result) to the DEVOUT task-events API — that **is** the audit channel, with `result` reflecting success/failure faithfully. No separate local JSONL audit logger is warranted here; the remote event stream satisfies the instrumentation requirement. ✓
