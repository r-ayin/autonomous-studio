## Detailed Findings

### PA-AUD-H-001: IngestionPipeline.scan_once/run_loop orphaned (asr.py:186-202)

**Evidence**: Python AST parses successfully but `scan_once` and `run_loop` are defined at module level (same indentation as `query_habits()`), yet take `self` parameter. Runtime check confirms `'scan_once' not in dir(IngestionPipeline)` and `'run_loop' not in dir(IngestionPipeline)`.

**Impact**: The entire ingestion polling loop (`IngestionPipeline.run_loop()`) is unreachable. Any code calling `pipeline.scan_once()` or `pipeline.run_loop()` raises `AttributeError`. This means the documented "轮询 inbox → 转写 → 入库" pipeline cannot run programmatically — only manual `process_file()` calls work.

**Remediation**: Re-indent lines 186-202 to be inside `IngestionPipeline` class body (4-space indent under class). Remove the duplicate `self`-less structure if any.

**Kind**: route-fix (mechanical indentation fix, 1 file)

### PA-AUD-H-002: Unbounded wav write in xiaozhi _asr() (xiaozhi_server.py:171-192)

**Evidence**: `_asr(self, wav_bytes: bytes)` writes `wav_bytes` directly to temp file via `open(path, "wb")` at line 180-181. No length check before write. The upstream `_on_audio()` accumulates PCM into `self._pcm_buf` (bytearray) with no cap — a device that sends continuous listen:start without stop can grow buffer indefinitely, then `_finalize_utterance()` writes entire buffer to disk.

**Attack scenario**: Malicious xiaozhi client sends hello + listen:start + 10GB of PCM data → server writes 10GB temp file → disk full → service crash.

**Remediation**: Add `MAX_UTTERANCE_BYTES = 50 * 1024 * 1024` (50MB, ~15min at 16kHz stereo). Check `len(self._pcm_buf)` in `_on_audio()` and abort if exceeded. Also check `len(wav_bytes)` before write in `_asr()`.

**Kind**: route-fix

### PA-AUD-M-001: Unsanitized audio_params in xiaozhi hello (xiaozhi_server.py:86-90)

**Evidence**: 
```python
self.sample_rate = int(ap.get("sample_rate", SAMPLE_RATE))  # line 88
self.channels = int(ap.get("channels", CHANNELS))           # line 89
self.frame_duration = int(ap.get("frame_duration", 60))     # line 90
```
No range validation. At line 128: `frame_size=int(self.sample_rate * self.frame_duration / 1000)`. With `sample_rate=999999999, frame_duration=60` → `frame_size=59999999`. Opus decoder would attempt to allocate ~120MB buffer per decode call.

**Remediation**: Clamp values: `sample_rate` ∈ {8000, 16000, 24000, 48000}, `channels` ∈ {1, 2}, `frame_duration` ∈ {10, 20, 40, 60}. Reject hello with invalid params.

**Kind**: route-fix

---

### PA-AUD-M-002: Discard threshold ignores negotiated rate (xiaozhi_server.py:141)

**Evidence**: `if len(pcm) < SAMPLE_RATE * SAMPLE_WIDTH // 2:` uses module-level `SAMPLE_RATE=16000`, but session may have negotiated 48000 via hello. At 48kHz, 0.5s = 48000 samples = 96000 bytes, but threshold is 16000 bytes → short utterances at high rate incorrectly kept; long utterances at low rate incorrectly discarded.

**Remediation**: Use `self.sample_rate` instead of `SAMPLE_RATE`: `if len(pcm) < self.sample_rate * SAMPLE_WIDTH // 2:`

**Kind**: route-fix

---

### PA-AUD-M-003: Log injection in CLINotifier (proactive.py:49-53)

**Evidence**: 
```python
line = f"[{storage.now_iso()}] ({trigger_kind}) {message}  ← {evidence}"
```
`message` comes from LLM output (line 84: `msg = self.llm.chat(...)`). If LLM returns text containing `\n` or `←`, log line structure breaks. Downstream log parsers would misparse entries.

**Remediation**: Replace newlines in message/evidence: `message.replace('\n', ' ')`, or use structured JSON logging.

**Kind**: route-fix

---

### PA-AUD-M-004: Unbounded read in StubTranscriber (asr.py:52)

**Evidence**: `p.read_text(encoding="utf-8")` with no size check. Inbox `.txt` files are user-supplied (via file drop or relay). A 1GB .txt file → OOM.

**Remediation**: Add `MAX_TXT_BYTES = 10 * 1024 * 1024` guard before read. Truncate or skip oversized files.

**Kind**: route-fix

---

### PA-AUD-L-001: Silent JSON parse failure (xiaozhi_server.py:232-235)

**Evidence**: `except json.JSONDecodeError: continue` — no log, no counter. Malformed client messages vanish silently. Makes protocol debugging impossible.

**Remediation**: Add `log.debug("xiaozhi malformed JSON: %s", msg["text"][:200])` before continue.

**Kind**: route-fix

---

### PA-AUD-L-002: Unbounded read in transcript.parse() (transcript.py:34)

**Evidence**: `p.read_text(encoding="utf-8")` no size guard. Lower risk than asr.py since transcript files are typically small, but defense-in-depth warrants a cap.

**Remediation**: Same pattern as M-004: size check before read.

**Kind**: route-fix
