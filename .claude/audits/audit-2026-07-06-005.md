## 执行摘要

dataworks-dev-assistant 是一个代码量颇重的 Python skill（core/ ~6.9K 行 + 60+ 模块脚本），封装阿里 DataWorks 平台 API，触及凭证、SQL 执行、DI 任务构建、部署、删除等敏感操作。整体安全 posture **中等**：核心 `bff_client.py` 设计了良好的安全机制——两阶段写确认门禁（write→confirm_write）、HTTP chokepoint 统一审计埋点（`_do_request`→`audit_log.record`）、10 秒人类思考时间锁。**但有两类系统性问题**：

1. **结构性绕过**（HIGH）：`deploy_udf.py` 与 `manage_node_deps.py` 完全绕开 BFFClient 的 HTTP chokepoint，用 `requests.post/put/delete` 直发，手工拼 Authorization 头。后果：**无两阶段确认、无审计记录**——一次调用即创建资源 + 上传 OSS + 部署生产，且 `.dataworks/.audit/` 里不留痕。这是本审计最严重的一类：安全控制被静默旁路。
2. **防御纵深缺失**（MEDIUM）：DuckDB 两处连接（`duckdb_query.py` 只读连接、`duckdb_loader.py` 读写连接）均未设 `enable_external_access=false`，用户 SQL 可经 `read_csv('/etc/passwd')` / `read_parquet` 任意读文件系统；telemetry 管线把完整 SQL（可能含 PII / 业务数据 / WHERE 值）未脱敏写入本地 JSONL 并 `telemetry_upload` 上传远端 BFF suggest 端点。

另一条 HIGH 是 `execute_sql.py` 的只读判定正则把 `WITH` 前缀一律当只读，`WITH x AS (SELECT 1) INSERT INTO t SELECT * FROM x` 可绕过写操作两阶段确认门禁（后端是否再拦一层是另一道防线，但本 skill 的安全门禁确被绕过）。

**派生 fix 派发顺序**（route-fix 优先，HIGH→MEDIUM→LOW）：
1. SQLEXE-01（HIGH，1 文件 1 行正则收紧）
2. DATAAN-01 + DATAAN-06（MEDIUM，DuckDB 两连接加 `enable_external_access=false`，可合并为一个 route-fix worktree）
3. AUDITT-01（MEDIUM，telemetry_start 对 `sql` 等敏感 kwarg 脱敏）
4. WRITEO-02（MEDIUM，manage_node_deps 改走 `_do_request` 触发审计）
5. 其余 LOW 批量（HTTP 硬编码 / 文件权限 / 非原子写 / 日志脱敏）

WRITEO-01（deploy_udf 结构性绕过）为 `kind=structural`，入 `structural-debt.md`（SD-DW-01），不直接派 fix——需更大重构授权。
