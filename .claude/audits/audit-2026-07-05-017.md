## Findings

### RE-CMDI-01 [HIGH, route-fix] — PowerShell 单引号命令注入：路径未转义

**文件/行**:
- `write-file.js:27-28` — `const parent = windowsPath.replace(...)`; `Set-Content -Path '${windowsPath}' -Value "${escapePS(content)}"`
- `batch-write.js:27-29` — `Set-Content -Path '${f.path}' -Value "${escapePS(f.content)}"`
- `push.js:40,57-60` — `winPath = winDir + '\\' + rel...`; `New-Item -ItemType Directory -Force -Path '${parent}'`; `Set-Content -Path '${f.winPath}' -Value "${escapePS(f.content)}"`

**证据**: `escapePS()` 只转义**双引号字符串上下文**的元字符（`` ` `` `$ `"` 换行），用于 `-Value "..."`。但 **path 插值在单引号字符串** `-Path '...'`，单引号在 PowerShell 中唯一的特殊字符是单引号本身（转义方式是 doubling `'` → `''`）。`escapePS` 不转义单引号 → 任何含 `'` 的路径破出单引号上下文，注入任意 PS 命令。

**PoC（write-file.js）**: `node write-file.js 'C:\x'"'"';Remove-Item C:\Users -Recurse -Force;Set-Content -Path '"'"'' 'payload'`
- `windowsPath` = `C:\x';Remove-Item C:\Users -Recurse -Force;Set-Content -Path '`
- 拼接后命令: `Set-Content -Path 'C:\x';Remove-Item C:\Users -Recurse -Force;Set-Content -Path '' -Value "..." -Encoding UTF8`
- 闭合原单引号 → `;Remove-Item...` 自由执行。

**输入源**:
- write-file.js: `process.argv[2]`（CLI，用户控制）
- batch-write.js: `f.path` from manifest JSON（`JSON.parse(fs.readFileSync(manifestPath))`，无校验）
- push.js: `winDir`=`process.argv[3]` + `rel`=本地文件名（**恶意本地文件名含 `'` 即可注入远程 Runner**）

**影响**: 远程凭证-bearing Windows Runner 上任意命令执行（删除、凭证窃取、横向移动）。Constrained Language Mode 不拦核心 cmdlet。

**remediation_sketch**: 新增 `escapePSPath(s)` 返回 `s.replace(/'/g, "''")`（PS 单引号标准转义），应用到 write-file.js / batch-write.js / push.js 所有 `-Path '...'` 与 `New-Item -Path '...'` 插值。更优：路径用 `-Path` 参数化（PS 支持 `-Path $path` 变量），但需改 input 协议；当前最小修法是单引号 doubling。

### RE-EXEC-02 [MEDIUM, route-fix] — 静默超时：exec.js / session.js 超时 resolve 部分输出，调用方无法区分成功与截断

**文件/行**:
- `exec.js:24` — `timer = setTimeout(() => { ws.close(); resolve(output); }, timeoutMs);` （resolve，非 reject；无 ok/error 标记）
- `session.js:41-46` — `const timer = setTimeout(() => { const result = this.buf; this.buf = ''; ... resolve(result); }, timeoutMs);`

**对比**: write-file.js:30 `resolve({ ok: false, error: 'timeout' })` ✓；batch-write.js:34 `resolve({ ok: false, error: 'timeout', output })` ✓。exec.js / session.js 不一致 —— 超时静默返回部分输出，CLI 调用方 `console.log(out)` 误以为命令成功完成。

**影响**: 长命令超时 → 调用方拿到截断输出却不报错，下游决策基于错误数据（如基于不完整 stdout 判断命令成功）。在自动化管线里这是隐蔽正确性 bug。

**remediation_sketch**: exec.js execOnRunner 超时改为 `resolve({ ok: false, error: 'timeout', output })` 或 reject；CLI 分支检查 `r.ok`。session.js exec() 同步加 `{ok,error}` 返回或 throw。保持与 write-file/batch-write 一致的 `{ok,error,output}` 契约。

### RE-EXEC-01 [MEDIUM, route-fix] — 输出无界累积 → OOM

**文件/行**:
- `exec.js:27` — `output += msg.data;`（无 cap）
- `session.js:26` — `this.buf += msg.data;`（无 cap）
- `write-file.js:33` — `output += msg.data;`
- `batch-write.js:37` — `output += msg.data;`

**证据**: Runner 上命令可能 emit GB 级输出（`Get-Content huge.log`、`tree /f`、误执行 `Get-ChildItem -Recurse C:\`）。`output += msg.data` 字符串拼接无上限，node 进程 OOM 崩溃。write-file/batch-write 实际只需检测 MARKER，不需要完整 output；exec.js/session.js 需要返回 output 但仍可 cap。

**影响**: DoS（本地 node 进程崩溃）+ 超时窗口内 OOM 前可能已耗尽内存。Runner 是共享资源。

**remediation_sketch**: 加 `MAX_OUTPUT = 16 * 1024 * 1024`（16MB）；累积超过 cap 时 `resolve({ ok:false, error:'output_overflow', output: output.slice(0,MAX) })`（或 reject）；write-file/batch-write 只需在 cap 触顶时停止累积、仍扫 MARKER。

### RE-MANIFEST-01 [MEDIUM, route-fix] — batch-write.js manifest 无 schema 校验

**文件/行**: `batch-write.js:55` — `const files = JSON.parse(fs.readFileSync(manifestPath, 'utf-8'));`

**证据**: `files` 直接 `for (const f of files)` 取 `f.path` / `f.content`，无类型/内容校验。恶意 manifest（`[{"path":null}]` 或 `{"path":123}` 或非数组）→ `f.path.replace` 抛 TypeError 崩溃；`f.path` 含 `'` 直达 RE-CMDI-01；`f.content` 非 string 致 `escapePS(f.content)` 抛 `str.replace is not a function`。

**影响**: 输入校验缺失放大 RE-CMDI-01 与 DoS 面；恶意/损坏 manifest 崩溃进程或注入。

**remediation_sketch**: `if (!Array.isArray(files)) throw new Error('manifest must be array'); for (const f of files) { if (typeof f.path !== 'string' || typeof f.content !== 'string') throw new Error('manifest entry {path,content} must be strings'); }`。

### RE-SECRET-01 [MEDIUM, route-fix] — creds.js 空 catch 吞错 + sqlite3 经 shell 调用

**文件/行**:
- `creds.js:14` — `catch {}`（getInternalSecret 文件读失败静默 fallback）
- `creds.js:30` — `catch {}`（getSessionToken sqlite3 失败，真实错误丢失，throw "无法获取 session token"）
- `creds.js:29` — `` execSync(`sqlite3 "${dbPath}" "SELECT token FROM active_tokens ORDER BY created_at DESC LIMIT 1;"`, { encoding: 'utf-8' }) ``

**证据**:
1. L30 空 catch 把 sqlite3 的真实失败（文件不存在 / 表不存在 / 权限）吞成无上下文的 "无法获取 session token"，排障困难。
2. L29 `execSync` 经 shell，`dbPath` 在双引号中。`dbPath = path.join(process.env.HOME || '/home/admin', '.aone-cloud-cli', 'auth.db')`。HOME 受信任但若被设为含 `"` / `$()` / `` ` `` 的值，双引号上下文注入。虽需已能设 HOME 的攻击者，但 `execFileSync('sqlite3', [dbPath, query])` 完全规避 shell，零成本。

**影响**: 可观测性（排障）+ 防御纵深（shell-quoting）。

**remediation_sketch**: L29 改 `execFileSync('sqlite3', [dbPath, 'SELECT token FROM active_tokens ORDER BY created_at DESC LIMIT 1;'], {encoding:'utf-8'})`；L14/L30 catch 改 `catch (e) { /* keep fallback */ }` 与 `catch (e) { throw new Error('无法获取 session token: ' + e.message); }`。

**文件/行**: `creds.js:7-8` — `const SERVER = 'http://127.0.0.1:58596'; const WS_HOST = 'ws://127.0.0.1:58596';`

### RE-PATH-01 [LOW, route-fix] — push.js walkDir 跟随符号链接读 localDir 外内容

**文件/行**: `push.js:11-20` walkDir + `push.js:35` `fs.readFileSync(localPath, 'utf-8')`

**证据**: walkDir 用 `entry.isDirectory()` 判定目录递归，`fs.readFileSync` 默认跟随符号链接。若 localDir 内含指向 `/etc/...` 或 `~/.ssh/...` 的符号链接，readFileSync 读取其内容并推送至远程 Runner。本地攻击者（或恶意 mirror 目录）可借机外泄本机敏感文件到 Runner。

**影响**: 本地文件外泄至远程 Runner（需本地可放符号链接，低概率但零成本修）。

**remediation_sketch**: walkDir 用 `entry.isSymbolicLink()` 跳过，或 `fs.realpathSync(localPath)` 后校验仍在 `fs.realpathSync(localDir)` 之下。

### RE-TOKEN-URL [INFO, structural] — session token 在 WebSocket URL query string

**文件/行**: `creds.js:18`（导出 WS_HOST）；`exec.js:11` / `session.js:18` / `write-file.js:15` / `batch-write.js:13` — `new WebSocket(`${WS_HOST}/runner-shell/${runnerId}?token=${token}`)`

**证据**: session token 放 URL query。SKILL.md:65 明确 "认证: WebSocket URL `?token=` 参数，不是 Authorization header" —— 这是**服务端协议要求**，客户端无法单方面改用 header（服务端不解析）。token 经 HTTP Upgrade 请求行 → server/proxy access log、代理历史可能记录。

**影响**: token 泄漏面（日志/历史）。但 by-design，需服务端协议升级支持 `Authorization` header 才能根治。

**kind=structural**：写入 .claude/structural-debt.md（SD-RE-01），不派 fix（需 cloud-cli server 侧改造，超出 autonomous-studio 仓库范围）。
