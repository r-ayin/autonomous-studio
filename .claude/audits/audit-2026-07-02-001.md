## Findings

### H-001: stop-completion-gate._strike_count 非原子写致防死循环失效

- **file**: .claude/hooks/stop-completion-gate.py
- **line**: 46-59
- **severity**: high
- **kind**: route-fix
- **finding**: `_strike_count()` 用 `open("w") + json.dump` 非原子写 STRIKE_FILE。并发 Stop hook 触发时（如快速连续 stop→resume→stop），读端可能读到半写 JSON → `_read_json` 返回 None → strike count 静默归零 → MAX_STRIKES=3 防死循环机制永远达不到阈值。与 discovery-gate.py case-404 修复前完全相同的漏洞模式。其他所有 checkpoint/gate 文件（save-checkpoint.py `_atomic_write_json`、discovery-gate.py `_atomic_write_lock`、incremental-save.py tempfile+os.replace、codegraph-sync.py `_atomic_write_json`）都已修过此问题，唯独 stop-completion-gate.py 漏掉。
- **remediation**: 改用 tempfile.mkstemp + os.fdopen + os.replace 原子写模式（与同目录 save-checkpoint.py `_atomic_write_json` 一致）。保留现有 inc/reset 语义。

### M-001: post-edit-lint py_compile 缺 `--` 终止符致假阳性

- **file**: .claude/hooks/post-edit-lint.py
- **line**: 57
- **severity**: medium
- **kind**: route-fix
- **finding**: `subprocess.run(["python", "-m", "py_compile", file_path], ...)` 直接传 file_path 未加 `--` 终止选项解析。若被编辑的文件名以 `-` 开头（如根目录 `-x.py` 或恶意构造路径），py_compile argparse 把它当 CLI 选项而非文件名 → returncode!=0 → check_syntax 误判为语法错误 → 触发 Stop gate 阻断（最多 3 次后 MAX_STRIKES 放行自愈，但仍属假阳性浪费 token）。case-472 security-review 已在 stop-completion-gate.py:109 修了同样的 bug（加了 `--`），但 post-edit-lint.py 同位置未同步修。
- **remediation**: 改为 `["python", "-m", "py_compile", "--", file_path]`，与 stop-completion-gate.py 对齐。

### M-002: pipeline-gate git 识别比主 gate 简化过度致绕过向量

- **file**: .claude/hooks/pipeline-gate.py
- **line**: 73, 155
- **severity**: medium
- **kind**: route-fix
- **finding**: `_GIT_INVOKED` regex `(?:^|[\s/])git(?=\s|$)` 与 `_parse_git_cwd` 仅 `-C\s+(\S+)` 简单匹配，相比 autonomous-commit-gate.py 缺失：(a) 不识别 `/usr/bin/git` 全路径调用（basename=git 的路径形）；(b) 不跨 `-c`/`--git-dir`/`--work-tree` 等全局选项解析真实子命令；(c) 不支持命令链 `A && git commit` 拆段评估（_git_segments）；(d) 不解 eval/exec/sh -c 间接执行壳（_unwrap_indirect）；(e) 不解 $()/反引号命令替换（_cmd_substitutions）。studio 项目下可构造 `eval "git commit"` / `/usr/bin/git commit` / `out=$(git push)` 等形态绕过 pipeline gate 的 commit/push 拦截，使"complex 任务须先 verify"约束失效。注：autonomous-commit-gate 仍会拦 main 直写，故不构成 main 安全风险，但 studio 流水线阶段约束被绕过会导致未验证代码进入 worktree。
- **remediation**: 将 autonomous-commit-gate.py 的 `_tokenize`/`_is_git_token`/`_git_parse`/`_git_segments`/`_unwrap_indirect`/`_cmd_substitutions` 抽成共享模块 `.claude/hooks/_git_cmd_parser.py`，pipeline-gate 导入复用。短期最小修复：至少在 `_is_git_commit_or_push` 中加 basename=git 全路径识别 + 命令链按 `&&/||/;/|` 拆段逐段检查。

### L-001: save-checkpoint.py Path import 在 main guard 内致潜在 NameError

- **file**: .claude/hooks/save-checkpoint.py
- **line**: 140 (find_task_files 引用 Path), 235 (from pathlib import Path 在 `if __name__ == "__main__"` 内)
- **severity**: low
- **kind**: route-fix
- **finding**: `find_task_files()` 函数体使用 `Path(PROJECT_DIR)` 但 `from pathlib import Path` 仅在 `if __name__ == "__main__":` 块内导入。当作为 hook 被 PreCompact/Stop/SessionEnd 触发时，`__name__` 是 `"__main__"` 所以实际会执行导入——当前不会出错。但若未来重构把 find_task_files 提取到其他模块或被 import 调用，Path 未定义 → NameError。属代码卫生问题，不影响当前运行。
- **remediation**: 将 `from pathlib import Path` 移到模块顶层 imports 区（第 17 行附近），与其他 stdlib import 并列。

### I-001: 审计埋点模板代码 6 文件重复实现

- **file**: autonomous-commit-gate.py, discovery-gate.py, pipeline-gate.py, auto-commit.py, notify-phone.py, codegraph-sync.py (及 incremental-save.py 间接)
- **line**: 多处 `_audit_log_*` 函数
- **severity**: info
- **kind**: structural
- **finding**: 6 个 hook 文件各自实现了几乎相同的审计埋点模板：构造 id (audit-YYYYMMDD-HHmmss-rand6)、timestamp (ISO8601 UTC)、userId/userRole="engine"、resource dict、result、ip="local"、sensitive/sensitiveLevel、details dict → makedirs .audit/ → open append JSONL。差异仅在 action/resource.identifier/details.reason 等业务字段。违反 DRY，增加维护负担：schema 变更需改 6 处，易漂移（如某文件忘了加 sensitiveLevel 字段）。
- **remediation**: 抽成 `.claude/hooks/_audit_log_helper.py`，暴露 `emit_audit(action, resource_type, resource_id, result, detail="", sensitive_level="medium")` 单函数。各 hook 一行调用。属 structural debt（拆不成 1-3 文件最小 fix 且触及公共接口），写入 structural-debt.md 队列。
