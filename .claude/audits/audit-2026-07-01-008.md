### XT-H-001 [HIGH] resume-checkpoint.py 固件注入硬编码 Windows 路径 `E:/x-tool/...`

- **文件**: `.claude/hooks/resume-checkpoint.py:121,124`
- **现象**: `build_engine_firmware()` 字符串字面量中写死 `E:/x-tool/.claude/decision-log.jsonl` 和 `E:/x-tool/.claude/memory/autonomous-state.md`。每次 SessionStart 该固件作为 additionalContext 注入系统提示，指导 Agent 读取这两个文件。
- **影响**: 在 Linux 部署（`/home/admin/workspace/x-tool`）下，Agent 按指令尝试读 `E:/x-tool/...` → 文件不存在 → L1 内联检查永远读不到日志 → 自主决策引擎实质失效。这是 firmware 级污染，比一般代码 bug 更严重——它让"自我修复"机制本身出错。
- **根因**: 工作区从 Windows (`E:\x-tool`) 迁到 Linux 后，字符串模板未参数化。`PROJECT_DIR` 变量已经在脚本顶部计算好（line 25），但 firmware 字符串没用 f-string 插值。
- **修复建议**: 把 firmware 中的硬编码路径替换为 `{PROJECT_DIR}` 插值（参考 line 151-154 已有的做法）。同步清理 `.claude/memory/project-index.md`、`.claude/skills/wechat-push/SKILL.md:58,95` 中的同类硬编码。
- **kind**: route-fix（仅改 resume-checkpoint.py 一个文件即可恢复核心功能；memory/skill 文档可后续单独修）

### XT-H-002 [HIGH] decision-observer.py handle_stop 反向读日志逻辑有 off-by-one + chunk 拼接错误

- **文件**: `.claude/hooks/decision-observer.py:262-288`
- **现象**: 反向读最后 20 行时，循环用 `buf = f.read(read_size) + buf` 拼接，但当 chunk 边界切在某行的中间时，该行会被拆成两段分别进入前后两次 `lines[-needed:]` 切片，导致 JSON 解析失败并 continue 跳过。同时 `lines_found` 计数基于 `if l.strip()` 的非空行数，但 `buf.split(b"\n")` 产生的首段可能是前次 chunk 的残余，被错误计入。
- **影响**: 当 decision-log.jsonl > 4KB 且某条记录跨 chunk 边界时，active_project 推断可能拿到错误的 project_hint（或始终 "unknown"）。Stop hook 输出的 additionalContext 会带错项目名 → Agent 跟进错误项目的 PROGRESS.md/GATES.md。低频但后果严重。
- **根因**: 手写反向 readline 没有维护"上一轮残留片段"的状态机。标准做法是用 deque + 累积 buffer 直到凑够 N 个完整换行。
- **修复建议**: 改用 `collections.deque(maxlen=20)` + 正向 tail 读法（先 seek 到末尾附近，readline 循环），或直接用 `subprocess.check_output(["tail", "-n", "20", LOG_FILE])` 简化（Linux-only 场景足够）。
- **kind**: route-fix

### XT-M-001 [MEDIUM] save-checkpoint.py collect_recent_activity 假设 ~/.claude/audit/audit.jsonl 存在

- **文件**: `.claude/hooks/save-checkpoint.py:81-132`
- **现象**: 硬编码读取 `~/.claude/audit/audit.jsonl`。该路径是 Claude Code 内部审计日志位置，在不同版本/配置下可能不存在或格式变化。当前 try/except 吞掉所有异常返回空列表，checkpoint 仍能保存但 recent_activity 永远为空。
- **影响**: 恢复会话时"最近活动"字段总是空 → 用户/Agent 无法从 checkpoint 中看到上次做了什么 → 恢复体验降级。不是致命问题但削弱了 checkpoint 的核心价值。
- **修复建议**: 增加环境变量 `CLAUDE_AUDIT_LOG` 允许覆盖路径；若默认路径不存在则 fallback 到 `.claude/decision-log.jsonl`（本工作区自己写的日志，保证存在）。
- **kind**: route-fix

### XT-M-002 [MEDIUM] protocol-check.py bootstrap 调用传 dummy 路径导致生成文件名异常

- **文件**: `.claude/hooks/protocol-check.py:70`
- **现象**: `subprocess.run([sys.executable, BOOTSTRAP_SCRIPT, os.path.join(project_dir, "dummy")], ...)` 传入 `project_dir/dummy` 作为 file_path 参数给 bootstrap.py。bootstrap.py 用 `os.path.dirname(file_path)` 推断项目目录——这恰好能工作，但依赖了一个未在接口契约中声明的隐式约定。如果 bootstrap.py 重构为从 file_path 提取项目名（而非 dirname），就会生成名为 "dummy" 的项目模板。
- **影响**: 脆弱耦合，未来维护风险高。当前不触发 bug 但属于技术债。
- **修复建议**: protocol-check.py 直接传 `project_dir`（不带 dummy），或在 bootstrap.py 增加显式 `--project-dir` 参数。
- **kind**: route-fix

### XT-M-003 [MEDIUM] notify-phone.py send() 返回值未被任何调用方消费

- **文件**: `.claude/hooks/notify-phone.py:163-187, 206`
- **现象**: `send()` 返回 bool 表示是否发送成功，但 `main()` line 206 调用 `send(...)` 后丢弃返回值。Hook 环境下 stdout/stderr 是唯一反馈通道，失败完全静默。
- **影响**: TCP 隧道断开 + ntfy.sh 不可达时，用户收不到通知也看不到任何错误。对于"需要确认"的高优先级通知尤其危险——Agent 以为通知到了，用户实际没收到，双方僵持。
- **修复建议**: 失败时向 stderr 输出一行 `[NOTIFY] ❌ all backends failed for {reason}`；或在 confirm 场景下将失败信息写入 `.claude/.notify_failed` 标记文件供下次 SessionStart 读取。
- **kind**: route-fix

### XT-M-004 [MEDIUM] watchdog.sh PROJECT_DIR 硬编码 `/mnt/e/x-tool`

- **文件**: `.claude/watchdog/watchdog.sh:15`
- **现象**: `PROJECT_DIR="/mnt/e/x-tool"` 写死 WSL 挂载路径。当前 Linux 部署在 `/home/admin/workspace/x-tool`，watchdog 若被启用会找不到 checkpoint 目录 → line 31-34 直接 exit 0 → 外部监控层完全失效。
- **影响**: L6 外部监控在 Linux 上是死代码。ARCHITECTURE.md 宣传的"五层防御"实际只有四层。
- **修复建议**: 改为 `PROJECT_DIR="${XTOOL_DIR:-/home/admin/workspace/x-tool}"` 并支持环境变量覆盖；或从脚本所在目录反推 (`$(cd "$(dirname "$0")/../.." && pwd)`)。
- **kind**: route-fix

### XT-L-001 [LOW] incremental-save.py 每 120s 重写整个 session-progress.md

- **文件**: `.claude/hooks/incremental-save.py:32-41`
- **现象**: 每次更新 auto-saved 标记时 read → regex replace → write 整个文件。session-progress.md 增长后 IO 放大。
- **影响**: 120s 周期 × 大文件 = 不必要的磁盘写入。当前文件小所以无感，但不 scale。
- **修复建议**: 用 append-only 策略：只在文件末尾追加新标记，启动时一次性清理旧标记。或用 mmap/truncate 原地更新固定长度的时间戳区域。
- **kind**: route-fix
