a1 是集团内部统一研发命令行客户端，将代码平台（Code/GitLab）、发布平台、CI 平台（AoneCI）、项目空间、二方库发布等多个研发平台的操作聚合到一个 CLI 中。

## ⚠️ 使用约定

1. **修复 CR 评论后默认不 resolve**：处理完 MR/CR 上的评审评论后，**不要**自动执行 `a1 repo mr comment resolve`，除非用户明确要求。评论由评审人确认后自行关闭。
2. **创建 MR 时自动关联工作项**：如果工作流是「获取工作项 → 修复代码 → 创建 MR」，创建 MR 时**必须**带 `--work-items <工作项ID>`，无需用户额外指定。
3. **工作项 `--body` 支持贴图**：`workitem create --body` / `update --body` 支持 Markdown 图片语法 `![desc](/path/to/image.png)`，CLI 会自动上传到图床。当用户提供截图或本地图片文件、且意图是创建/更新工作项时，把图片以 Markdown 写入 `--body`，**不要**先去分析图片内容。
4. **禁止传空字符串给枚举类型参数**：如 `--trigger-mr-type`、`--status` 等枚举类型参数，**禁止传空字符串 `""`**。这类参数要么传有效枚举值，要么完全不传（使用默认值）。空字符串会导致服务端 JSON 解析失败。

## 前置条件

### 安装

```bash
curl -fsSL https://git.cn-hangzhou.oss-cdn.aliyun-inc.com/aone-cli/install.sh | sh
```

官方文档：<https://a1.io.alibaba-inc.com/>

### 禁用后台升级检查（CI / 脚本）

a1 每次执行会在后台检查新版本。在 CI、离线/弱网或批量脚本中可一次性关闭：设置环境变量 `A1_NO_UPDATE_CHECK=1`，或给命令加全局 flag `--no-update-check`。两者等价、任一为真即禁用；环境变量适合写入 CI / dotfiles / 基础镜像统一生效，无需每次显式传参。

### 认证

**a1-server 命令**（工作项、CR、流水线等）与 **kit/MCP/Skill 命令**（`a1 mcp call-tool`、`a1 skill find/install/publish`）使用不同的认证机制。

#### a1-server（默认）

```bash
a1 auth login --buc    # BUC SSO（推荐）
a1 auth whoami         # 查看当前登录用户
```

命令返回认证错误时，提示用户执行 `a1 auth login --buc`。

#### kit / MCP / Skill（A1_AUTH_PROVIDER）

kit/MCP/Skill 命令支持 **CIAP 与 Bearer 双向 fallback**（`AONE_MCP_TOKEN` 或 `a1 auth login` Code PAT）：

| 场景 | 顺序 |
|------|------|
| 默认 | CIAP AuthX → Bearer |
| `export A1_AUTH_PROVIDER=private_token` | Bearer → CIAP |
| `export A1_AUTH_PROVIDER=buc` | CIAP BUC → Bearer |

MCP 单次调用可用 `--auth-provider`（优先于环境变量）。`--provider zetta` 仅支持 CIAP。

涉及 Skill 搜索/安装/发布、MCP call-tool 时，先 Read `references/skill-commands.md` 中的「认证」一节。

### 资源绑定（Link Context）

很多命令依赖"当前绑定的资源"。绑定后，后续命令自动使用绑定的资源 ID：

```bash
a1 link              # 交互式绑定（自动检测 git remote）
a1 link status       # 查看当前绑定状态
a1 repo link         # 绑定代码仓库
a1 app link <name>   # 绑定应用
a1 project link      # 绑定项目空间
```

执行命令前先 `a1 link status -f json` 确认绑定。未绑定且命令需要资源 ID 时，提示用户先绑定或用 `--repo`/`--app`/`--project-id` 等标志手动指定。

### 构建任务与日志（build）

`a1 build job` 面向 Aone Build 的构建 Job，不是 AoneCI 的 `ci job`。当前支持按 pipeline instance id 查询构建 Job，再下钻到 Step 日志。

```bash
a1 build job list --pipeline-instance-id <pipeline-instance-id>  # 按 pipeline instance id 查构建 Job
a1 build job get <job-id>                                        # 查构建 Job 详情
a1 build job steps <job-id>                                      # 查可用日志 Step
a1 build job log <job-id> --step build-package                   # 一次性拉取 Step 日志
a1 build job log <job-id> --step build-package --offset 0 --limit 500 -f json
a1 build job log <job-id> --step build-package --follow          # 仅用于开始查询时仍在运行的 Step
a1 build job log <job-id> --step build-package --download build.log
```

常用判断：
- `--pipeline-instance-id` 是当前支持的构建 Job 查询维度之一；CLI 通过 a1-server 转给构建服务查询，不要让用户自己提供底层 `mixFlowInstId`。
- `--limit` 表示本次最多拉取的日志行数，默认 `500`；`--offset` 使用上一次 JSON 响应里的 `nextOffset` 做增量查询。
- 需要脚本化轮询时用 `-f json`，重点看 `nextOffset`、`finished`、`stepRunning`、`stepStatus`；不要用是否返回空日志来判断结束。
- `--follow` 第一次请求要求目标 Step 正在运行；开始 follow 后会持续轮询，直到服务端返回 `finished=true` 且 `stepRunning=false`。
- 已结束的任务优先用普通 `log` 或 `--download`，不要强行走 `--follow`。

完整 flag 参见 `references/build-commands.md`。

```bash

# CR 描述会被 a1 按 2000 display-width 自动截断；Agent 生成时仍应先摘要成 3-5 行，完整说明放 PRD/tech-design 链接。
a1 app cr create "变更描述" --branch new-feat                 # 新建分支 feature/new-feat
a1 app cr create "变更描述" --existing-branch hotfix/fix      # 用已有远程分支
a1 app cr create "变更描述" --trunk feature/main --format json # trunk_based 主干研发变更
a1 app cr create "变更描述" --app my-service --code-module-id 2820201 --existing-branch hotfix/fix
a1 app cr create "变更描述" --app my-service --item-name my-code-module --branch new-feat
a1 app cr create "变更描述" --branch new-feat --plan-release-date "2026-06-10 14:00"
a1 app cr create "变更描述" --branch new-feat --plan-release-date +2d
# branch 模式不传分支参数会报错；trunk_based 应用不传时会使用应用默认主干

a1 app cr submit <cr-id>                              # 提交到待发布
a1 app cr submit <cr-id> --pipeline-id <id>           # 提交到发布流水线并触发执行
a1 app pipeline status --pipeline-id <id> --wait-until-settled  # 等待流水线结束
a1 app cr submit-integration <cr-id> --project-env-id <id>      # 提交到集成阶段/项目环境
```

用户说“提交流水线发布”“提交到发布流水线”“发到流水线”时，使用 `a1 app cr submit <cr-id> --pipeline-id <id>`；不要改用 `a1 app cr submit-integration`。`submit-integration` 是项目环境/集成阶段命令，只在用户明确说“集成阶段”“项目环境”“联调环境”并给出 projectEnvId 时使用。

**CR 配置项：**
```bash
a1 app cr item list <cr-id>                                      # 列出 CR 下的变更项
a1 app cr config-item schema list <cr-id> --item <cr-item-id>    # 查询可操作配置分组
```

**⚠️ 发布状态解读（非显式行为，必须主动检查）：**

`a1 app pipeline status` 返回的 `status` 字段不能完全反映"卡住"的真实原因——

1. `status=running` 但实际被某节点 `extendedStatus` 阻塞——整体未结束所以仍显示 running。
2. `status=waiting` 既可能是正常等待条件，也可能需要人工介入。

遇到「发布卡住不动」时，按这个顺序排查：
```bash
a1 app pipeline status --pipeline-id <id>          # 1. 看整体状态（但 status 字段可能误导）
a1 app pipeline stage list --pipeline-id <id>      # 2. 找出 extendedStatus 失败/异常的阶段（阻塞的真实位置）
a1 app pipeline stage status --stage-id <id>       # 3. 看该阶段的具体原因
a1 app pipeline branch --instance-id <id>          # 4. 看 release 分支和分支合并 CR
```

**发布单操作：** 流水线任务支持动态动作（deploy-order-list / view-diagnosis / resume / finish / skip / log），也支持组件名快捷访问（如 `CR_PUBLISH_RELEASE`、`BUILD`）。完整工作流和参数说明参见 `references/app-commands.md`。

### 制品与二方包（artifact / pkg）

- **`a1 artifact mvn deploy <dir>`**：本地工程扫描模块并上传到 **Artlab**，适合 SNAPSHOT/本机验证。
- **`a1 pkg deploy-cr`**：**Aone 官方 Maven 二方库发布流水线**（Mix OpenAPI）。用于 CR 详情页/单变更发布类流程（start-flow → preview-modules → submit），与本地 `deploy` 直传不是同一路径。
- **`a1 pkg cr submit` + `a1 pkg deploy-intg preview/submit`**：**发布页流水线集成区二方包部署**。用于包 CR 已提交到发布页流水线集成区、需要填写「提交二方库发布」组件的场景，支持多变更集成。不要把该场景放到 `artifact` 命令域，也不要传发布页 URL；`deploy-intg` 只需要包上下文和 `--flow-inst-id`。

官方二方包发布的完整流程与 flag 参见 `references/pkg-commands.md` 和 `references/app-commands.md`；本地 Artlab 上传参见 `references/artifact-commands.md`。构造 `pkg deploy-cr submit`、`pkg deploy-intg submit` 或 `app pipeline lib-deploy submit` 前先执行 preview：模块取 `modules[].name`；默认 JDK 取 `defaultJdkVersion`，切换 JDK 取 `jdkOptions[].resource`；默认 Maven 取 `mvnVersion`，切换 Maven 取 `mvnOptions[].resource`；SNAPSHOT 包发布时，`pkg deploy-intg` 传 `--is-fl`。

### 常见工作流提示

- **排查 CI 失败**：`run get` → `job list --run <run-id>` → `job log <job-id> --run <run-id>`；日志里出现 `TEST_CASE` 或 `issue summary` 关键字时，从日志找 report uid（uuid 格式），再 `quality testcase <uid>` 或 `quality issue --severity critical,blocker` 拉详情。
- **排查构建失败**：用户给的是 pipeline instance id 时，走 `build job list --pipeline-instance-id <id>` → `build job steps <job-id>` → `build job log <job-id> --step <step>`；运行中 Step 才用 `--follow`，结束任务优先普通 `log` 或 `--download`。
- **排查部署/发布单问题**：`pipeline status` → `stage list` → `stage job list` → `job task list` → `task status` 看 `supportedActions`，再按需执行动作（view-diagnosis / resume / finish 等）。完整流程参见 `references/app-commands.md`。
- **管理 Aone Pages 站点配置 vs 部署内容**：两件事不要混：
  - **管理站点配置**（建/查/改/删 siteName 和 domains）→ `a1 pages view\|create\|update\|delete`，详见 `references/pages-commands.md`。触发词：`查 pages 站点` / `创建 pages 站点` / `改 pages 域名` / `删 pages` 等。
  - **部署内容到 Pages**（构建 + 推 OSS）→ `a1 ci` 创建流水线，模板 `10014197`，四步：查重 → 本地代码检查 → 推导 `build-website-command` / `deploy-dir` → 创建。完整骨架与参数推导规则去 Read `references/ci-commands.md` 示例 5，不要凭记忆写命令也不要回退到自定义 YAML。触发词：`部署 pages` / `deploy pages` / `刷新 pages` 等。
  - 仅出现「pages」名词无明确动词（如 `/a1 pages`）时先反问"是想管理站点配置还是部署内容？"再决定走哪条路径。

### 查找并安装 Skill


发布 Contextlab Skill 的逐步流程见 `references/skill-commands.md` 文末。
